## 1. Frontend — SSR / гидратация

### F-1 (High) — рассинхрон локали в шапке → hydration mismatch ✅

**Симптом (репорт владельца):**
```
Hydration failed because the server rendered HTML didn't match the client.
... at MobileNav ... at AppHeader ...
server: <a data-testid="nav-sign-in-link" href="/ru/login">
client: <button data-testid="mobile-nav-trigger" ...>
```

**Root cause (подтверждён экспериментально).**
`apps/web/src/app/[locale]/layout.tsx` рендерил
`<NextIntlClientProvider messages={messages}>` **без явного пропа `locale`**,
полагаясь на авто-наследование локали next-intl. Шапка (`AppHeader`) живёт
внутри `<Suspense fallback={null}>` (нужен из-за `useSearchParams()` в
`LocaleSwitcher`/`CommandPalette`). При стриминге этой Suspense-границы
клиентский провайдер мог инициализироваться **до** того, как унаследованная
локаль станет доступна, и откатывался к `defaultLocale` (`en`). В результате
все `<Link>` next-intl в шапке строили `/en/...`-адреса на странице `/ru`,
расходясь с серверным HTML (`/ru/...`).

**Доказательство (live).** На `/ru` в один момент:
сервер-HTML `nav-sign-in-link → /ru/login`, клиентский DOM `→ /en/login`
(зафиксировано через `fetch('/ru')` vs `document.querySelector`).

**Почему именно `MobileNav` в стектрейсе.** `<Link asChild>` внутри
Radix/vaul `Slot`-обёрток делает границы примирения «размытыми»: React
рапортует ближайший расходящийся узел, которым оказывается соседний
триггер `MobileNav`.

**Фикс.** Явно передан `locale` в провайдер (единственный надёжный способ,
не зависящий от тайминга авто-наследования):
```tsx
<NextIntlClientProvider locale={locale} messages={messages}>
```
Файл: `apps/web/src/app/[locale]/layout.tsx`.

**Верификация.** На чистой загрузке `/ru`: `url=/ru`, `lang=ru`,
`locale-switcher=RU`, `nav-sign-in-link=/ru/login`, консоль без ошибок.

> Это не только устраняло hydration-warning, но и функциональный баг:
> клик «Войти» на русской странице вёл на английский логин.

### Прочее по SSR/гидратации — проверено, чисто

- Прогон `Date.now()` / `Math.random()` / `new Date()` / `typeof window` во
  фронтенде: все вхождения — в обработчиках событий и не-render-хелперах
  (аналитика, анонимные ID, локаль в http-клиенте), **не** в рендере. Риска
  гидратации нет.
- `AuthProvider` инициализируется детерминированно (`user=null`,
  `isLoading=false`) одинаково на сервере и клиенте — auth-состояние в
  принципе не может дать hydration mismatch (аутентификация целиком
  клиентская, после гидратации).

### F-3 (High) — hydration #418 на glossary / sources / legal-signals ✅ (СЛОЙ 2)

> Метод: написан авто-прогон (Playwright) по **всем 36 достижимым
> маршрутам** (static + `russia` + все `/internal/*`) на desktop (1280) и
> mobile (375), с перехватом console-error / pageerror / hydration / crash /
> горизонтального overflow. Прогон выполнен и против **dev**, и против
> **prod-сборки** — это ключ к отделению реальных багов от dev-артефактов.

**Находки прогона.**
- **Реальные (воспроизводятся в prod-сборке, React #418):**
  `/ru/glossary`, `/ru/sources`, `/ru/legal-signals`,
  `/ru/legal-signals/timeline` (последний — редирект на registry-view).
- **Только dev (в prod чисто):** `/ru/countries/russia` — транзиентный
  артефакт стриминга dev-компиляции при первом (холодном) заходе; в
  prod-сборке досье чистое. Не баг приложения.

**Root cause реальных (#418).** Эти view грузят все данные на клиенте
(TanStack Query **без** SSR-dehydration) и читают URL через
`nuqs`/`useSearchParams` внутри `<Suspense>`. На SSR эта граница
suspend-ится и отдаёт loading-фолбэк; при гидратации сверка приостановленной
границы «сервер vs клиент» расходится → #418. Поскольку сервер для этих view
всё равно может отрендерить **только** loading, SSR приостановленной границы
— чистый минус.

**Фикс.** Мелкий враппер `shared/ui/ClientOnly.tsx`: рендерит один и тот же
фолбэк на сервере и на **первой** клиентской отрисовке (обычный `div`, а не
приостановленная граница) → гидратируется чисто; реальный `Suspense+Inner`
монтируется уже на клиенте. SSR-контент не теряется (его и не было). Применён
к `GlossaryView`, `SourcesView`, `LegalSignalsRegistryView`.

**Верификация.** Все 4 маршрута: из #418 → чисто, по 2+ повторных prod-загрузки
каждый. Полный прогон 36 маршрутов в prod — **без единой проблемы** (нет
hydration, console-error, overflow, крашей). `typecheck`/`lint`/86 web-тестов
— зелёные.

**Открытый момент (честно).** `/search` и `/countries` используют **тот же**
паттерн (client-fetch + nuqs + Suspense), но #418 не дают — ни в dev, ни в
prod, ни при повторных прогонах. Точный триггер, из-за которого расходятся
именно эти 4 view, изолировать не удалось (dev-ошибка не детерминирована,
prod минифицирована). Фикс `ClientOnly` устраняет весь класс независимо от
триггера, поэтому оставлять «рабочие» страницы без него безопасно; при желании
паттерн можно применить и к ним для единообразия.

### Раскладка / overflow по всем маршрутам — verified

Авто-прогон проверял `scrollWidth > clientWidth` на каждом маршруте на обоих
брейкпоинтах — **горизонтального overflow нет нигде** (кроме исправленного
F-2 клиппинга таблиц, теперь скролл внутри `overflow-x-auto`).

### Frontend-безопасность — verified

- `dangerouslySetInnerHTML` — **нигде** (нет XSS-поверхности).
- Все внешние ссылки (`source.url`, target=_blank) имеют `rel="noreferrer"`.
- `useSearchParams`/`nuqs` не ломают статику: все маршруты `ƒ` (dynamic),
  сборка без warning.

## 2. Frontend — раскладка / коллизии

### F-2 (Medium) — обрезание широких таблиц на мобильном ✅

**Симптом.** На досье страны (`/ru/countries/russia`) при 375px таблица
разбора баллов (462px, колонки «Критерий / Оценка / Вес / Взвешенная /
Уверенность») обрезалась родительской карточкой `overflow-hidden`. Правая
колонка «Взвешенная» была недостижима — нет обёртки с горизонтальным
скроллом. Страница при этом не скроллилась (`docW == vw`), т.е. контент
именно **терялся**, а не переполнял.

**Root cause.** Общий примитив `packages/ui/src/primitives/DataTable.tsx`
рендерил голый `<table>` без скролл-обёртки. Дополнительно hover-подчёркивание
строки использовало `w-[100vw]` — наивная обёртка в `overflow-x-auto` дала бы
**постоянный** горизонтальный скроллбар.

**Фикс.**
1. Таблица обёрнута в `<div className="overflow-x-auto">`.
2. Sweep-подчёркивание перепривязано к строке (`<tr>` — relative-контейнер),
   `left-0 right-0` вместо `w-[100vw]` — точная ширина строки, без ложного
   скроллбара.

Файл: `packages/ui/src/primitives/DataTable.tsx`. Фикс — в общем примитиве,
поэтому чинит сразу всех потребителей (`ScoreBreakdown`,
`DecisionBreakdown`, `CountryScenarioMatrix`, `HomeMatrixPreview`).

**Верификация (live, 375px).** Таблицы (462px) теперь в `overflow-x-auto`,
`wrapScrollable=true`, страница без горизонтального overflow, ложного
скроллбара нет.

### Прочее по раскладке — проверено, чисто

- **Горизонтальный overflow:** desktop (1265px) и mobile (375px) на
  `/`, `/countries`, досье — переполнения документа нет. «Вылезающие»
  элементы на главной — это off-screen слайды `HorizontalPager` (by design).
- **Другие таблицы** (`CountryScenarioMatrix`, `HomeMatrixPreview`) уже
  обёрнуты в `overflow-x-auto` — правкам не подлежат.
- **z-index-система оверлеев** — упорядочена и без коллизий:
  фон `z-0` → шапка `z-40` → ChartFrame-fullscreen `z-70` →
  Drawer/Analysis `z-80/81` → Dropdown/Popover/Select/Tooltip `z-85` →
  Dialog `z-90/91` → CommandPalette `z-95`.

## 3. Backend — безопасность и корректность (СЛОЙ 1, сплошной аудит)

> Метод: систематические сканы по **всем** 42 роутерам / 100 репозиториям /
> 136 сервисам (auth-матрица, SQL-паттерны, обработка ошибок, транзакции,
> N+1, инварианты) + глубокое чтение фундамента авторизации + адресное
> расследование **каждой** аномалии. Багов не найдено — слой здоров.

### 3.1 Авторизация / RBAC — verified

- Построена **матрица auth по всем 42 роутерам**. Первичный скан показал
  «дыры» (`admin_country_contribution` 9 роутов/1 auth,
  `country_contribution` 14/1, `community` 8/0) — при проверке оказалось,
  что auth навешан через **type-alias паттерн**
  (`_Editor = Annotated[CurrentUser, Depends(require_editor)]`,
  `_Contributor = ...require_capability(CONTRIBUTOR_COUNTRIES)`), который
  построчный grep не считал.
  - `admin_country_contribution`: 9/9 эндпоинтов под `_Editor`. ✓
  - `country_contribution`: 14/14 под `_Contributor`. ✓
  - `community`: write-эндпоинты (вопросы/ответы/голоса/репорты)
    **анонимны намеренно** (совпадает с фронтовым `anonymousIdentity` /
    `getOrCreateAnonymousSessionId`) — это дизайн-решение (анонимное
    участие), не дыра авторизации.
  - **Итог: ни одного эндпоинта без нужной защиты.**
- **Фундамент авторизации прочитан построчно** (`core/rbac.py`,
  `core/auth.py`, `services/auth.py::validate_session_token`):
  корректные ролевые кортежи (`require_admin = {ADMIN, OWNER}`),
  capability-гейты через БД, 403 при отказе. Токен: bearer-заголовок **или**
  httponly-cookie (`httponly`, `secure` в prod, `samesite=lax`), хранится
  **хешем** (не плейнтекстом), с ротацией + grace-периодом, проверкой
  revoked/expired.
- **Статус пользователя enforced**: `validate_session_token` бросает 403 для
  `suspended` и `deleted` **до** любого `get_current_active_user` — сессия
  забаненного пользователя не проходит.

### 3.2 IDOR — verified

Пользовательские ресурсы (`watchlists`, `weight_profiles`, `trip_planner`,
`author_metrics` и т.д.) скоупятся по `user_id=current_user.id` из сессии,
**не** из клиентского параметра — подмена чужого `user_id` невозможна.

### 3.3 SQL-инъекции — verified

Прогон по всем репозиториям: каждый `f"""`-SQL интерполирует **только**
статические allowlist'ы колонок (`USER_FIELDS`, `SOURCE_RETURNING`,
`_JOB_COLUMNS`, …) и заранее собранные `WHERE`-фрагменты со `%s`; значения
**всегда** параметризованы. Инъекций нет.

### 3.4 Обработка ошибок — verified

Скан по services + repositories: **ноль** «глухих» `except: pass` /
проглоченных ошибок. Ошибки поднимаются через `api_error(...)`.

### 3.5 Транзакции — verified (расследована аномалия)

Скан «mutations vs commit» выявил роутеры с мутациями без явного
`connection.commit()` (`create_missing`/`create_stale`/`process_batch` в
`admin_translation_jobs`, а также `ai`, `analytics`, `decision_passports`).
Расследование: `get_connection` использует пул psycopg,
**авто-коммит при чистом выходе** из контекст-менеджера и rollback при
исключении (документировано в `core/database.py`, решение P2-9,
Аудит-эпизод 4). Явные commit'ы в роутерах избыточны, но безвредны.
`process_batch` дополнительно коммитит внутри сервиса. **Потери данных нет.**

### 3.6 N+1 — verified

Единственный per-entity цикл с записью (`platform_metrics.recompute_all` →
по странам) — это **офлайн batch-джоб** (bootstrap/recompute), не hot-path
запроса. Остальные `for row in repo.list_...` — обработка результата
одного запроса в памяти. N+1 в запросном пути не найдено.

### 3.7 Инварианты и секреты — verified

- Скан на нарушения реестра инвариантов (деньги→права/ранжирование,
  репутация→авто-выдача прав) — чисто.
- `config.py::_forbid_default_secrets_in_production` — прод не стартует с
  дефолтными плейсхолдер-секретами. В репозитории только `.env.example`.

**Охват слоя:** auth/SQL/ошибки/транзакции/N+1/инварианты просканированы по
100% файлов; фундамент авторизации и все аномалии прочитаны построчно. Не
делалось построчного чтения всех 236 файлов services/repositories — для
аудита безопасности/корректности достаточно сплошного сканирования рисковых
измерений + глубокого чтения критичного. Багов на этом слое не найдено.

## 3bis. Notifier (Go gRPC-сервис) — СЛОЙ 3, verified

> Прочитаны ключевые файлы (gRPC-сервер, consume-loop, handler, telegram-
> клиент, dedup/DLQ); `go vet ./...` чисто, `go build ./...` чисто, `go test
> ./...` — все пакеты зелёные. Багов не найдено; **оба прежних замечания
> аудита закрыты**.

- **gRPC-авторизация — solid.** `tokenAuthInterceptor` навешан на **все**
  unary-методы: fail-closed при пустом токене, сравнение
  `subtle.ConstantTimeCompare` (без тайминг-утечки), метрики auth-fail.
  Это service-to-service граница (FastAPI backend → notifier по общему
  `notifier_internal_auth_token`, у которого есть prod-guard секретов).
- **IDOR — контролируемая модель доверия.** Методы принимают
  `telegram_user_id` из запроса; per-user авторизацию делает вызывающий
  backend (только он держит токен), а связка web↔telegram (`SetWebUserID`,
  link-коды с TTL и хешированием) фиксирует, каким `telegram_user_id`
  владеет web-пользователь. Внутренний сервис не доверяет никому, кроме
  токен-владельца.
- **Доставка — effectively-once (прежнее «at-most-once» закрыто).**
  Consume-loop коммитит offset **после** успешного `Handle` (commit-after-
  process = at-least-once): при ошибке хендлера — `continue` без коммита →
  переобработка. Плюс dedup (`Exists` в начале, `TryInsert` после доставки)
  → повторов пользователю нет. Poison-сообщения уходят в DLQ и коммитятся,
  не блокируя партицию.
- **RealClient — реализован (прежний «stub» закрыт).**
  `telegram/client.go::RealClient.SendMessage` делает реальный POST на
  `api.telegram.org/bot<token>/sendMessage` с таймаутом, контекстом и
  обработкой non-2xx. `FakeClient` — fake-by-default seam (режим через
  конфиг), что соответствует правилу автономной разработки.
- **Обработка ошибок — чисто.** Нет проглоченных ошибок на критичных
  вызовах (Send/Insert/Upsert/Commit/Handle/Write), нет `panic()`.
- **Наблюдение (не баг).** `FormatMessage` — тексты Telegram-уведомлений
  захардкожены по-русски; notifier не входит в i18n-контур интерфейса, для
  текущего этапа приемлемо (зафиксировано как наблюдение).

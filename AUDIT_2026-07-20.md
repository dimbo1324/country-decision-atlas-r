## 1. Frontend — SSR / гидратация

### F-1 (High) — рассинхрон локали в шапке → hydration mismatch ✅

**Симптом (репорт владельца):**
```
Hydration failed because the server rendered HTML didn't match the client.
... at MobileNav ... at AppHeader ...
server: <a data-testid="nav-sign-in-link" href="/ru/login">
client: <button data-testid="mobile-nav-trigger" ...>
```

**Root cause (подтверждён экспериментально).**
`apps/web/src/app/[locale]/layout.tsx` рендерил
`<NextIntlClientProvider messages={messages}>` **без явного пропа `locale`**,
полагаясь на авто-наследование локали next-intl. Шапка (`AppHeader`) живёт
внутри `<Suspense fallback={null}>` (нужен из-за `useSearchParams()` в
`LocaleSwitcher`/`CommandPalette`). При стриминге этой Suspense-границы
клиентский провайдер мог инициализироваться **до** того, как унаследованная
локаль станет доступна, и откатывался к `defaultLocale` (`en`). В результате
все `<Link>` next-intl в шапке строили `/en/...`-адреса на странице `/ru`,
расходясь с серверным HTML (`/ru/...`).

**Доказательство (live).** На `/ru` в один момент:
сервер-HTML `nav-sign-in-link → /ru/login`, клиентский DOM `→ /en/login`
(зафиксировано через `fetch('/ru')` vs `document.querySelector`).

**Почему именно `MobileNav` в стектрейсе.** `<Link asChild>` внутри
Radix/vaul `Slot`-обёрток делает границы примирения «размытыми»: React
рапортует ближайший расходящийся узел, которым оказывается соседний
триггер `MobileNav`.

**Фикс.** Явно передан `locale` в провайдер (единственный надёжный способ,
не зависящий от тайминга авто-наследования):
```tsx
<NextIntlClientProvider locale={locale} messages={messages}>
```
Файл: `apps/web/src/app/[locale]/layout.tsx`.

**Верификация.** На чистой загрузке `/ru`: `url=/ru`, `lang=ru`,
`locale-switcher=RU`, `nav-sign-in-link=/ru/login`, консоль без ошибок.

> Это не только устраняло hydration-warning, но и функциональный баг:
> клик «Войти» на русской странице вёл на английский логин.

### Прочее по SSR/гидратации — проверено, чисто

- Прогон `Date.now()` / `Math.random()` / `new Date()` / `typeof window` во
  фронтенде: все вхождения — в обработчиках событий и не-render-хелперах
  (аналитика, анонимные ID, локаль в http-клиенте), **не** в рендере. Риска
  гидратации нет.
- `AuthProvider` инициализируется детерминированно (`user=null`,
  `isLoading=false`) одинаково на сервере и клиенте — auth-состояние в
  принципе не может дать hydration mismatch (аутентификация целиком
  клиентская, после гидратации).

## 2. Frontend — раскладка / коллизии

### F-2 (Medium) — обрезание широких таблиц на мобильном ✅

**Симптом.** На досье страны (`/ru/countries/russia`) при 375px таблица
разбора баллов (462px, колонки «Критерий / Оценка / Вес / Взвешенная /
Уверенность») обрезалась родительской карточкой `overflow-hidden`. Правая
колонка «Взвешенная» была недостижима — нет обёртки с горизонтальным
скроллом. Страница при этом не скроллилась (`docW == vw`), т.е. контент
именно **терялся**, а не переполнял.

**Root cause.** Общий примитив `packages/ui/src/primitives/DataTable.tsx`
рендерил голый `<table>` без скролл-обёртки. Дополнительно hover-подчёркивание
строки использовало `w-[100vw]` — наивная обёртка в `overflow-x-auto` дала бы
**постоянный** горизонтальный скроллбар.

**Фикс.**
1. Таблица обёрнута в `<div className="overflow-x-auto">`.
2. Sweep-подчёркивание перепривязано к строке (`<tr>` — relative-контейнер),
   `left-0 right-0` вместо `w-[100vw]` — точная ширина строки, без ложного
   скроллбара.

Файл: `packages/ui/src/primitives/DataTable.tsx`. Фикс — в общем примитиве,
поэтому чинит сразу всех потребителей (`ScoreBreakdown`,
`DecisionBreakdown`, `CountryScenarioMatrix`, `HomeMatrixPreview`).

**Верификация (live, 375px).** Таблицы (462px) теперь в `overflow-x-auto`,
`wrapScrollable=true`, страница без горизонтального overflow, ложного
скроллбара нет.

### Прочее по раскладке — проверено, чисто

- **Горизонтальный overflow:** desktop (1265px) и mobile (375px) на
  `/`, `/countries`, досье — переполнения документа нет. «Вылезающие»
  элементы на главной — это off-screen слайды `HorizontalPager` (by design).
- **Другие таблицы** (`CountryScenarioMatrix`, `HomeMatrixPreview`) уже
  обёрнуты в `overflow-x-auto` — правкам не подлежат.
- **z-index-система оверлеев** — упорядочена и без коллизий:
  фон `z-0` → шапка `z-40` → ChartFrame-fullscreen `z-70` →
  Drawer/Analysis `z-80/81` → Dropdown/Popover/Select/Tooltip `z-85` →
  Dialog `z-90/91` → CommandPalette `z-95`.

## 3. Backend — безопасность

Проверено и подтверждено здоровым:

- **Авторизация / RBAC.** Админ-роутеры требуют `Depends(require_admin)`
  (`app/core/rbac.py`, фабрика `require_roles(*roles)`); пользовательские —
  `Depends(get_current_active_user)`.
- **IDOR.** Пользовательские ресурсы (напр. `watchlists.py`) скоупятся по
  `user_id=current_user.id` из сессии, **не** из клиентского параметра —
  подмена чужого `user_id` невозможна.
- **SQL-инъекции.** Все f-string в SQL интерполируют только статические
  allowlist'ы колонок (`SOURCE_RETURNING`, `_JOB_COLUMNS`) и заранее
  собранные фрагменты `WHERE` со `%s`-плейсхолдерами; значения всегда
  параметризованы (`translation_jobs.py`, `admin_content.py` проверены).
- **Секреты.** `config.py` содержит явную защиту
  `_forbid_default_secrets_in_production` — прод не стартует с
  дефолтными плейсхолдер-секретами (`database_url`, `analytics_salt`,
  `notifier_internal_auth_token`). В репозитории только `.env.example`.

  картинку с наслоением — **она не пришла в чат**; если укажете конкретный
  экран, проверю адресно.

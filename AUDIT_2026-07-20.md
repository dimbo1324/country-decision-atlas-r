## 1. Находки

<a id="a-1"></a>
### A-1 🟡 14 полей формы без доступного имени

**Проблема.** У элементов нет ни `aria-label`, ни `id` со связанным
`<label htmlFor>`, ни обёртки `<label>`. Скринридер объявляет их как
безымянные («edit text», «combo box»), пользователь не понимает, что
вводить. Часть — чекбоксы в чек-листах поездок, то есть основной сценарий.

**Где.**
```
apps/web/src/features/admin-users/UsersAdminView.tsx:90            <select>
apps/web/src/features/author-metrics/AuthorMetricsStudioView.tsx:186 <input>
apps/web/src/features/community/CommunityCountryBlock.tsx:426      <input>
apps/web/src/features/community/CommunityCountryBlock.tsx:437      <textarea>
apps/web/src/features/community/CommunityCountryBlock.tsx:467      <select>
apps/web/src/features/community/CommunityCountryBlock.tsx:491      <textarea>
apps/web/src/features/community/CommunityCountryBlock.tsx:534      <input>
apps/web/src/features/community/CommunityCountryBlock.tsx:555      <textarea>
apps/web/src/features/migration-board/MigrationBoardDetailView.tsx:153 <textarea>
apps/web/src/features/migration-board/MigrationBoardDetailView.tsx:186 <textarea>
apps/web/src/features/trips/TripChecklist.tsx:37                   <input type=checkbox>
apps/web/src/features/trips/TripChecklist.tsx:105                  <input>
apps/web/src/features/trips/TripChecklist.tsx:162                  <input>
apps/web/src/features/trips/TripReminders.tsx:91                   <input>
```

**Как чинить.** Любой из трёх способов, по ситуации:
1. `aria-label={t("...")}` — быстрее всего там, где подписи visually нет
   (чекбоксы в `TripChecklist`);
2. `id` на поле + `htmlFor` на подписи — если подпись уже нарисована рядом
   (`UsersAdminView.tsx:87` — там подпись лежит в `<span>`, её достаточно
   заменить на `<FieldLabel htmlFor="...">`);
3. обернуть контрол в `<label>` — как уже сделано в
   `GlossaryFilters.tsx` (хороший образец в проекте).

**Важно знать.** В проекте **уже есть правильный паттерн** — смотрите
`apps/web/src/features/glossary/GlossaryFilters.tsx:31-41`
(обёртка `<label>`) и `DecisionWeightSlider.tsx:35` (`aria-label={label}`).
Стоит выровнять остальные по ним, а не изобретать новый.

<a id="a-2"></a>
### A-2 🟡 8 `<FieldLabel>` без `htmlFor` — подпись не связана с полем

**Проблема.** `FieldLabel` рендерит настоящий `<label htmlFor={htmlFor}>`,
но проп **опциональный**. Когда его не передают, а у соседнего поля нет
`id`, получается `<label>` без связи: подпись видна глазами, но
программно с полем не сцеплена. Скринридер поле не назовёт, и клик по
подписи не сфокусирует поле.

**Где.**
- `packages/ui/src/primitives/Field.tsx:20-40` — источник (`htmlFor?`).
- `apps/web/src/features/community/CommunityCountryBlock.tsx` — например
  строки 425/426 (`<FieldLabel>` + `<input>` — соседи, без `id`).
- `apps/web/src/features/decision-run/DecisionRunForm.tsx`.

Масштаб: **8 использований из 80** — остальные 72 `htmlFor` передают
корректно, то есть это точечная недоделка, а не системная.

**Как чинить.** Проставить пару `htmlFor`/`id`:
```tsx
<FieldLabel htmlFor="community-question-title">{t("questionTitleLabel")}</FieldLabel>
<input id="community-question-title" ... />
```

**Важно знать.** Радикальнее и надёжнее — сделать `htmlFor` **обязательным**
в типе `FieldLabel`. Тогда TypeScript сам найдёт все 8 мест и не даст
завести новые. Это одна правка в `Field.tsx` + 8 подстановок, и проблема
закрывается навсегда.

<a id="ux-1"></a>
### UX-1 🟡 Оптимистичные мутации откатываются молча

**Проблема.** Мутации применяют изменение оптимистично, а при ошибке
**просто возвращают кэш назад** — без какого-либо уведомления. С точки
зрения пользователя: нажал «в избранное» — звёздочка загорелась и через
мгновение сама погасла, без объяснений. Выглядит как глюк интерфейса, а
не как «сервер отверг запрос».

**Где.**
- `apps/web/src/entities/watchlist/api.ts:99-103` — `onError` только
  `setQueryData(previous)`, тоста нет.
- `apps/web/src/entities/watchlist/api.ts:151` — то же для
  notification-preference.
- `apps/web/src/entities/trips/api.ts:175`, `:251` — тот же паттерн.
- `apps/web/src/app/providers.tsx` — у `QueryClient` **не задан**
  `MutationCache` с общим `onError`, то есть глобального «подхвата» тоже
  нет.
- Вызывающая сторона, например
  `apps/web/src/features/country-catalog/WatchlistStar.tsx:48` — просто
  `toggle.mutate(...)`, ошибку не смотрит.

**Как чинить.** Лучше централизованно, одним местом на всё приложение —
в `providers.tsx`:
```tsx
import { MutationCache } from "@tanstack/react-query";
import { toast } from "@country-decision-atlas/ui";

new QueryClient({
  mutationCache: new MutationCache({
    onError: (error) => toast.error(extractMessage(error)),
  }),
  defaultOptions: { /* как сейчас */ },
});
```
Тогда откат останется в локальных `onError`, а сообщение пользователю
появится само во всех мутациях сразу.

**Важно знать.** Инфраструктура для этого **уже есть**: `Toaster` смонтирован
в `apps/web/src/app/layout.tsx`, а `toast` экспортируется из
`@country-decision-atlas/ui`. Добавлять зависимости не нужно.

<a id="i18n-1"></a>
### I18N-1 🟡 Нет `[locale]/not-found.tsx` → EN/ES видят русскую 404

**Проблема.** Единственная страница 404 — корневая, с **захардкоженным
русским** текстом. Внутри `[locale]`-дерева своей `not-found.tsx` нет,
поэтому англо- и испаноязычный пользователь на несуществующем адресе
получает «Страница не найдена / На главную» по-русски.

**Где.**
- `apps/web/src/app/not-found.tsx:11-14` — хардкод RU.
- `apps/web/src/app/[locale]/layout.tsx:35` — вызывает `notFound()` при
  неизвестной локали; ближайшей границы внутри `[locale]` нет, поэтому
  срабатывает корневая.
- Для сравнения: `apps/web/src/app/[locale]/error.tsx` **уже** локализован —
  то есть для ошибок паттерн соблюдён, а для 404 нет.

**Как чинить.** Добавить `apps/web/src/app/[locale]/not-found.tsx`, который
берёт тексты через `useTranslations`/`getTranslations` (он рендерится
внутри `NextIntlClientProvider`, переводы доступны). Корневую
`not-found.tsx` **оставить как есть** — она нужна как запасной вариант.

**Важно знать.** Комментарий в корневой `not-found.tsx` объясняет, почему
там нет next-intl: она может отрисоваться для `/internal/**` и для путей
вообще без локали. Это **верно** — не убирайте её и не тащите туда
переводы; нужна именно **дополнительная** страница внутри `[locale]`.

<a id="sec-1"></a>
### SEC-1 🟡 Порты dev-стека открыты на `0.0.0.0`, redis и mongo без auth

**Проблема.** Инфраструктурные сервисы публикуются в форме `"6379:6379"`,
что в Docker означает привязку ко **всем интерфейсам**, а не к localhost.
При этом:
- **redis** — вообще без `requirepass`;
- **mongo** — без `MONGO_INITDB_ROOT_*` и без `--auth`;
- **postgres** — с паролем по умолчанию `change-me`.

В любой недоверенной сети (коворкинг, кафе, общий офисный Wi-Fi) любой
сосед по подсети может подключиться к вашим базам и прочитать/изменить
данные.

**Где.** `docker-compose.yml`:
```
строка 24  postgres  "${POSTGRES_PORT:-5433}:5432"
строка 36  redis     "6379:6379"        # без requirepass
строка 119 redpanda  "19092:19092"
строка 132 mongo     "27017:27017"      # без auth
строка 160 notifier  "8081:8081", "9090:9090"
```

**Как чинить.** Привязать к loopback:
```yaml
ports:
  - "127.0.0.1:6379:6379"
  - "127.0.0.1:27017:27017"
  - "127.0.0.1:${POSTGRES_PORT:-5433}:5432"
```

**Важно знать.**
- На работу стека это **не влияет**: контейнеры общаются между собой по
  внутренней сети compose (по именам сервисов), а не через опубликованные
  порты. Инструменты на хосте (psql, `apply_migrations.py`, dev-сервер
  Next.js) продолжат ходить на `localhost`.
- **Сломает** единственный сценарий: доступ к этим базам с **другой
  машины/устройства**. Если вы так делаете — оставьте как есть либо
  привяжите к конкретному интерфейсу.
- Прод это **не затрагивает**: `apps/api/app/core/config.py:81-95`
  (`_forbid_default_secrets_in_production`) не даёт приложению стартовать в
  production с дефолтными секретами. Проблема ровно про dev-машину.

<a id="i18n-2"></a>
### I18N-2 🟢 Захардкожена локаль `"ru"` в таймстампах админки

**Проблема.** Время в журнале событий панели пересчёта форматируется
жёстко по-русски, независимо от языка интерфейса.

**Где.** `apps/web/src/features/admin-recompute/RecomputePanelView.tsx:76`
и `:85` — `new Date().toLocaleTimeString("ru")`.

**Как чинить.** Взять текущую локаль и прокинуть её в форматирование:
```tsx
const locale = useAppLocale();
// ...
timestamp: new Date().toLocaleTimeString(locale),
```

**Важно знать.** В проекте уже есть готовые хелперы —
`apps/web/src/shared/lib/format.ts` (`formatDateTime(value, locale)`) и
`apps/web/src/shared/lib/useAppLocale.ts`. Лучше использовать их, а не
звать `toLocaleTimeString` напрямую. Это не баг гидратации: код внутри
обработчика события, не в рендере.

<a id="arch-1"></a>
### ARCH-1 🟢 `sys.path`-хак в worker тянется во внутренности `apps/api`

**Проблема.** Worker вычисляет корень репозитория **четырьмя вложенными**
`os.path.dirname` и вручную подставляет `apps/api` в `sys.path`, чтобы
импортировать `app.services.translation_jobs`. Работает, но: (1) молча
сломается, если файл переедет на другой уровень вложенности; (2) worker
жёстко завязан на внутренний сервисный слой API.

**Где.** `apps/worker/country_decision_atlas_worker/main.py:6-11`, затем
импорты внутри функции — строки 21-22.

**Как чинить.** Варианты по возрастанию усилий:
1. Запускать worker как модуль установленного пакета
   (`pip install -e .` уже настроен: `pyproject.toml` объявляет
   `apps/worker` в `[tool.setuptools.packages.find]`) и убрать хак;
2. вынести используемую логику в `packages/shared`, чтобы и API, и worker
   импортировали её как общий пакет, а не лазили друг к другу.

**Важно знать.** Сервис **рабочий** — я его запускал: `--help` отрабатывает,
`translation-jobs --dry-run --limit 2` даёт `processed=0` (корректно при
пустой очереди), тесты `tests/test_translation_jobs_worker.py` — 30 штук,
все зелёные. Это долг, а не поломка; чинить можно спокойно.

<a id="notif-1"></a>
### NOTIF-1 🟢 Тексты Telegram-уведомлений только по-русски

**Проблема.** Сообщения пользователю формируются с русскими строками
без учёта языка получателя, хотя интерфейс поддерживает 3 локали.

**Где.** `apps/notifier/internal/telegram/client.go:63-79` (`FormatMessage`),
дисклеймер — строка 12.

**Как чинить.** Прокинуть локаль получателя в событие/подписку и выбрать
шаблон по ней. Потребует поля локали в подписке
(`apps/notifier/internal/mongo/subscriptions.go`) и в доменном событии.

**Важно знать.** Это заметный объём работы (схема события + хранилище +
шаблоны), а notifier сейчас не входит в i18n-контур интерфейса. Разумно
отложить до момента, когда реально появятся неруглоязычные получатели.

отличаться в зависимости от того, что установлено.

<a id="i18n-3"></a>
### I18N-3 ⚪ `/internal/**` полностью по-русски

**Наблюдение, не баг.** Все страницы внутренней ops-консоли содержат
захардкоженные русские строки (`AI-черновики`, `Пересчёт метрик`,
`Пользователи`, `Роль`, подписи форм и т.д.).

**Где.** `apps/web/src/app/internal/**` и `apps/web/src/features/admin-*/**`,
например `apps/web/src/app/internal/users/page.tsx:11`,
`apps/web/src/features/admin-evidence/EvidenceAdminView.tsx:80`.

**Почему это, скорее всего, осознанно.** `apps/web/src/middleware.ts:10-18`
явно выводит `/internal` из-под локального префикса («ops-консоль, никогда
не локализуется») и закрывает её вне dev/local. Это внутренний инструмент
для одной команды.

**Что решить.** Просто зафиксируйте позицию: «внутренняя консоль — только
русский» либо «локализуем и её». Сейчас это нигде не записано, и следующий
разработчик может начать переводить её вразнобой.

### ARCH-2 ⚪ `apps/worker` не упомянут в compose/CI/доках

**Наблюдение.** Поиск по `docker-compose.yml`, `.github/workflows/`,
каталогу dev-tools-раннера и `docs/` не находит ни одного упоминания
worker'а. Оператор, пришедший в проект, не узнает, что сервис существует и
как его запускать.

**Как чинить.** Достаточно малого: строка в
`docs/_arch_/01_Продукт/02_Текущее_состояние_системы.md` и/или запись в
каталоге `utils/dev_tools_scripts_runner/config/` — чтобы он появился в
`python dev_tools_scripts_runner.py`.

**Важно знать.** Тесты worker'а в CI **гоняются** (они лежат в общем
`tests/`), так что регресс поймается. Проблема ровно в обнаруживаемости.

## 4. Что я проверил и **отбросил** как ложные срабатывания

Чтобы вы не наткнулись на то же самое и не потратили время:

- **`Math.random()` в чартах** (`BarColumns`, `DivergingMeter`, `DonutChart`,
  `DriftBoard`, `GaugeArc`, `Heatmap`, `RadarChart`) — выглядит как риск
  гидратации, но весь этот код **внутри `useEffect`** (live-режим
  «дыхания» данных) с корректным `clearInterval`. В рендере ничего
  случайного нет.
- **`target="_blank"` без `rel`** — греп даёт ~10 совпадений, но `rel="noreferrer"`
  во всех случаях стоит **на следующей строке** JSX. Всё безопасно.
- **Поля формы без `aria-label`** — первый прогон дал 22 штуки, но regex
  ломался на `=>` внутри JSX-атрибутов. После корректного разбора осталось
  **14** (они в A-1); например `GlossaryFilters.tsx`, `DecisionWeightSlider.tsx`
  и `CommunityCountryBlock.tsx:219` размечены правильно.
- **«INSERT без ON CONFLICT» в миграциях** — 6 кандидатов, все безопасны:
  `021` защищён блоком `DO $$ ... IF NOT EXISTS`, `010`/`020` — только
  `UPDATE`, `014`/`028`/`055` — только `ALTER`, `006` — delete-then-insert.
- **Мутации «без обработки ошибок»** — обработка есть, но только откат
  кэша. Это не отсутствие обработки, а отсутствие обратной связи — вынесено
  отдельным пунктом UX-1.
- **Hydration на `/countries/russia`** — падало в dev при первой (холодной)
  компиляции маршрута, но в **prod-сборке чисто**. Это артефакт dev-сервера
  Next.js, а не баг приложения. Именно поэтому прогон делался против обеих
  сборок.

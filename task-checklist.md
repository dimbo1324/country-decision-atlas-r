# Чек-лист задачи: Аудит-эпизод 2 — `fix/login-rate-limiting-v1`

Цель: закрыть реальную возможность перебора пароля и злоупотребления
лимитером через подделку заголовков, независимо от `APP_ENV` — P0-3, P1-6,
P3-12, P3-5. См. `docs/_arch_/09_План_устранения_аудита.md`, раздел
«Аудит-эпизод 2».

Примечание: ветка ранее называлась `fix/rate-limiting-v1`; переименована в
`fix/login-rate-limiting-v1`, чтобы не пересекаться со старой (нерабочей)
удалённой веткой того же имени. Реализация начата заново с чистого main
(main уже содержит смердженный Аудит-эпизод 1).

## Подготовка

- [+] Изучены `app_factory.py`/`main.py` (текущий in-memory лимитер,
      per-process, работает только при `APP_ENV=production`) и
      `services/auth.py` (`login_user`/`register_user`, `services/cache.py`

## Реализация

- [+] P1-6: новый модуль `services/rate_limiter.py` — Redis INCR+EXPIRE
      счётчик (fail-open при недоступном Redis), синглтон-клиент
      (`@lru_cache`, без повторения бага P1-5). Отдельный строгий лимит
      `API_RATE_LIMIT_AUTH_PER_MINUTE` на `/auth/login` и `/auth/register`,
      работающий **независимо от `APP_ENV`** — именно это и есть суть
      находки. Общий лимит на весь остальной трафик оставлен prod-only, как
      было изначально (см. «Сознательные отступления» — так и было
      сформулировано в исходном плане episода, но по ошибке снято при первой
      реализации и возвращено после диагностики E2E-нестабильности).
- [+] P0-3: per-account блокировка после
      `AUTH_LOGIN_MAX_FAILED_ATTEMPTS` неудачных попыток (Redis-счётчик,
      окно `AUTH_LOGIN_FAILURE_WINDOW_SECONDS`), растущая длительность
      блокировки (`AUTH_LOGIN_LOCKOUT_SECONDS × 2^excess`, капается
      `AUTH_LOGIN_LOCKOUT_MAX_SECONDS`). События лога (не `audit_events` —
      см. «Сознательные отступления»).
- [+] P3-12: `main.py::_rate_limit_client` — `X-Forwarded-For` учитывается
      только если `request.client.host` входит в `TRUSTED_PROXY_IPS`.
- [+] P3-5: решение — оставлено как есть (см. «Сознательные отступления»),
      без изменения кода.

## Сознательные отступления от первоначального описания находки

- [+] **Найдена и исправлена собственная регрессия при первой попытке
      реализации.** Первая версия миддлвари убирала prod-only проверку
      **для всего трафика**, а не только для auth-путей — хотя сам план
      эпизода прямо говорил «общий лимит — как есть». Это привело к тому,
      что КАЖДЫЙ запрос (не только auth) стал делать синхронный Redis
      round-trip внутри `async`-миддлвари, что при полном прогоне Playwright
      E2E (278 тестов, высокий параллелизм) вызывало нестабильные таймауты
      (14–36 случайных падений на прогон). Диагностировано контрольным
      прогоном на чистом (без AE-2) коде — там все 278 тестов проходили
      стабильно. Исправлено: (а) синхронный вызов обёрнут в
      `starlette.concurrency.run_in_threadpool`, чтобы не блокировать event
      loop; (б) **главное** — общий лимитер возвращён к исходному поведению
      (активен только при `APP_ENV=production`), auth-лимитер и lockout
      остались always-on (это и есть фактическое требование P0-3/P1-6).
      После фикса — 278/278 стабильно, дважды подряд.
- [+] Lockout-события пишутся через `logger.warning(...)`, а не в
      `audit_events`: таблица имеет `CHECK`-constraint на `action` (см.
      миграцию 048) и семантически рассчитана на действия
      **аутентифицированного** актора над сущностью; анонимная неудачная
      попытка логина не вписывается в эту модель без миграции ради одного
      значения enum. Структурные логи достаточны и не создают лишнего
      технического долга (наблюдаемость — тема отдельного эпизода AE-7).
- [+] `docker-compose.yml` и `.github/workflows/quality.yml`: значение
      `API_RATE_LIMIT_AUTH_PER_MINUTE` для CI/локального Docker-стека
      завышено (300 / 5000) относительно продового дефолта (10) — по тому
      же принципу, что уже применялся к общему лимиту (`600` в compose
      против `120` в проде): тестовая инфраструктура легитимно генерирует
      на порядки больше auth-трафика в единицу времени, чем реальный
      пользователь.
- [+] Убрана мёртвая настройка `api_rate_limit_max_clients` (принадлежала
      старому in-memory механизму, полностью заменённому на Redis с TTL —
      отдельная очистка по лимиту клиентов больше не нужна).
- [+] `docker-compose.yml`: `api` теперь зависит от `redis: condition:
      service_healthy` (раньше — только `postgres`); без этого лимитер мог
      бы кэшировать `None`-клиент навсегда при гонке запуска контейнеров
      (Redis ещё не готов на момент первого запроса).

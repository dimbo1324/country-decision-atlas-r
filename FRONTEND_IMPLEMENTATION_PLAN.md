# План реализации фронтенда Country Decision Atlas

Версия: 1.0 · Дата: 2026-07-12
Аудитория: отдел фронтенд-разработки.
Назначение: пошаговый план построения продуктового фронтенда поверх уже готового бэкенда, в визуальном языке «тёплый архив разведки».

---

## 1. Цель и суть проекта

Country Decision Atlas — платформа поддержки решений «в какую страну ехать под конкретную цель» (переезд, ВНЖ/гражданство, удалённая работа, бюджетная жизнь, бизнес, безопасность). Это **не** рейтинговый блог: каждый вывод сопровождается источниками, уровнем уверенности, свежестью данных и прозрачными компромиссами.

Бэкенд полностью готов: FastAPI, ~215 эндпоинтов в 41 домене, версионируемый контракт `contracts/openapi.yaml`, из которого уже генерируются TypeScript-типы (`packages/contracts/generated/types.ts`). Существующий фронтенд `apps/web` (Next.js 15, App Router) уже покрывает бизнес-логику **функционально** — 29 страниц, 23 feature-домена, слой API-обёрток — но выглядит как служебный MVP без дизайна.

**Задача этого плана** — превратить функциональный MVP в законченный продукт: перевести весь интерфейс на дизайн-систему «тёплый архив разведки» (тёмный тёплый угольно-синий фон, золотой акцент, гравюрная типографика, «дышащие» данные), не потеряв ни одной работающей функции.

## 3. Ключевые архитектурные решения

Эти решения приняты на уровне плана и не пересматриваются внутри этапов без отдельного согласования.

### Р-1. База — `apps/web`, стратегия — «рескин, не rewrite»

Прототип остаётся донором компонентов и стилей; продуктовый код живёт в `apps/web`. Причины: в `apps/web` уже есть корректная работа с контрактом, auth-сессиями, локализацией и все продуктовые маршруты; переписывание с нуля гарантированно потеряет краевые случаи (обработка error envelope, feature-гейты, статусы модерации). Миграция идёт **постранично**: новая дизайн-система вводится фундаментом (этапы 1–4), затем страницы переводятся на неё по доменам (этапы 5–12). Старый `styles.css` (3001 строка) выпиливается по мере миграции и удаляется целиком в конце.

### Р-2. Общий пакет дизайн-системы `packages/ui`

Токены, примитивы и чарты извлекаются из прототипа в workspace-пакет `packages/ui` (`@country-decision-atlas/ui`), который импортируют и `apps/web`, и `apps/web-prototype`. Так прототип продолжает жить как «песочница/витрина», а продукт и прототип не расходятся визуально. Внутри пакета: `tokens/` (CSS-переменные + Tailwind-тема), `primitives/`, `charts/`, `shell/`, `hooks/`.

### Р-3. Стилизация — Tailwind CSS v4 + CVA

Прототип уже на Tailwind v4 (`@theme` с токенами в `index.css` — готовый файл темы). В `apps/web` подключается тот же слой темы. Вариативность компонентов — через `class-variance-authority` + `clsx` + `tailwind-merge`. Все цвета — **только** через токены, хардкод HEX в компонентах запрещён (правило §10.8 дизайн-системы).

### Р-4. Данные — типизированный клиент + TanStack Query

- `openapi-fetch` поверх сгенерированных типов `packages/contracts` — типобезопасные вызовы без ручных обёрток (постепенно заменяет самописные модули `shared/api/*`).
- `TanStack Query` — кэш, инвалидация, optimistic updates (watchlist, голосования, чек-листы поездок), retry-политика, состояния загрузки.
- Серверные компоненты (RSC) — для публичных контентных страниц (каталог, досье страны, методология): данные грузятся на сервере, SEO и LCP выигрывают. Клиентские компоненты + Query — для всего интерактивного и авторизованного (кабинет, борд, решения, AI).

### Р-5. Чарты — «custom-first»

Фирменные «дышащие» canvas/SVG-графики прототипа — **и есть бренд**; они не заменяются библиотекой. Библиотеки допускаются только как низкоуровневые помощники (`d3-scale`, `d3-shape` для шкал и дуг) или для экранов, где кастом нерентабелен (админ-графики). Правила производительности из дизайн-системы (§5.6, §10) — обязательны: анимируется только видимый canvas, resize только с debounce, шум генерируется на офскрин-канвасе ¼ разрешения.

### Р-6. Анимации — три уровня

1. **CSS-переходы** (hover, раскрытия, лазерная обводка карточек) — уже в токенах (`--ease-arrive`, `--ease-spring`, `--ease-reveal`).
2. **Motion** (бывш. Framer Motion) — только для presence/layout-переходов: страницы, drawer'ы, списки. Не для «дыхания данных».
3. **Собственный RAF-цикл** (`useCanvasLoop`) — для живых графиков. Всё уважает `prefers-reduced-motion` (готовый хук `useReducedMotion`).

### Р-7. Локализация — locale-in-URL, next-intl для хрома

Контентная локализация уже решена бэкендом (overlay по `locale`-параметру). Для строк интерфейса вводится `next-intl` (ru — основной, en — вторичный), локаль публичных страниц — в URL (`/ru/...`, `/en/...`) согласно принятому владельцем решению. Существующий `localization-labels.ts` мигрирует в словари next-intl.

### Р-8. Состояние в URL — nuqs

Фильтры каталога, выбранные страны сравнения, активный сценарий, вкладки досье — сериализуются в query-параметры через `nuqs`. Любой экран аналитики должен быть shareable по ссылке — это продуктовое требование (паспорта решений уже работают так на бэкенде).

### Р-9. Feature flags — сквозной гейтинг

Бэкенд отдаёт `/api/v1/platform/features`. Каждая новая поверхность оборачивается в гейт (уже есть `shared/lib/features.ts`) — новые экраны выкатываются выключенными и включаются флагом. Это соответствует серверному инварианту «новые поверхности за флагами».

### Р-10. Компонентная мастерская — Storybook

`LibraryOverlay` прототипа остаётся быстрой витриной, но для командной работы вводится Storybook в `packages/ui`: каждый примитив и чарт получает story со всеми вариантами и «шумными» данными. Storybook = приёмочный стенд для этапов 1 и 4.

## 5. Обязательные правила дизайн-системы (выжимка)

Полный эталон — `countryatlas_design_system.md`. Ниже — то, что проверяется на каждом код-ревью:

1. **Палитра.** Фоны `--bg…--bg4`, текст `--c1…--c5`, золото `--gold/gold2/gold3` ≤ 10–15% площади экрана. Вторичные акценты: синий = нейтральные факты, терракота = риск/ухудшение, шалфей = позитив/рост, слива = сообщество. У одного блока — ровно один акцент (3 его оттенка). Чистые `#fff`/`#000` запрещены.
2. **Типографика.** Playfair Display — заголовки и числа-герои; Crimson Text — текст; IM Fell English italic — цитаты/сноски; мономаркировка (Courier New, caps, letter-spacing 2–4px) — только для метаданных (`REF №`, `LIVE`, `CONFIDENCE`). Shimmer — только у главного логотипа.
3. **Движение.** Кривые из токенов; hover 0.25–0.4s; drawer 0.45–0.6s; scroll-reveal 0.8–0.9s однократно. Живые значения — только lerp (коэфф. 0.045–0.06/кадр), никаких дискретных подмен.
4. **Производительность.** Один активный canvas за раз (IntersectionObserver/индекс слайда); resize — debounce 200–300ms; никакого кэширования scroll-офсетов (только свежий `getBoundingClientRect()` на кадр); `overflow-anchor: none` при JS-управлении высотами; шум — офскрин ¼ разрешения.
5. **Доступность.** `prefers-reduced-motion` деградирует всё тяжёлое в статику; `:focus-visible` — золотая обводка; интерактив — только на семантических элементах; контраст текста `--c2`/`--c3` на тёмных фонах проверяется для мелких кеглей (при провале AA — поднимать до `--c1`/`--c2`).
6. **Прозрачность данных.** Рядом с каждым показателем — дата верификации и уверенность; дисклеймер «не юридическая консультация» — в футере и возле любых рекомендаций. Это продуктовый инвариант, а не украшение.
7. **Иконки** — только inline SVG (Lucide), в рамке-квадрате, никаких иконочных веб-шрифтов.

## 7. Этапы реализации

Нумерация сквозная; зависимости указаны в каждом этапе. Этапы 5–12 могут частично параллелиться между разработчиками после завершения 1–4.

---

### Этап 0 — Ревизия, договорённости, инфраструктура работ

**Описание.** Команда знакомится с активами, фиксирует рабочие соглашения, готовит окружение. Кода-продукта на этом этапе нет.

**Цели и задачи.**
- Каждый разработчик читает дизайн-систему (`countryatlas_design_system.md`), открывает живой макет (`countryatlas_platform.html` в браузере) и прототип (`corepack pnpm@9.12.0 --filter @country-decision-atlas/web-prototype dev`), проходит галерею компонентов (кнопка «Библиотека компонентов» в топбаре).
- Инвентаризация `apps/web`: по каждой из 29 страниц — таблица «работает / что показывает / какие фичи гейтятся». Результат — чек-лист миграции, против которого будут закрываться этапы 5–12.
- Поднять локальное окружение данных: либо Docker-стек (`docker compose up -d postgres redis api` + миграции + `restore_demo_countries.py`), либо синтетический mock-сервер из `scripts/synthetic_data` — для работы без БД.
- Договориться о конвенциях: структура `packages/ui`, нейминг компонентов, правила story-файлов, критерии ревью (раздел 5 этого плана — чек-лист ревьюера).

**Библиотеки.** Не добавляются.

**Критерий готовности.** Чек-лист страниц составлен; каждый разработчик запускал прототип и макет; окружение данных поднимается одной командой.

---

### Этап 1 — Фундамент дизайн-системы: `packages/ui`, токены, типографика, фон

**Описание.** Создаётся общий пакет дизайн-системы и переносится «нулевой слой» стиля: цвета, шрифты, фоновые слои, глобальные утилиты. После этого этапа любой экран `apps/web` уже «стоит на тёплом фоне» даже со старыми компонентами.

**Цели и задачи.**
- Создать `packages/ui` (workspace-пакет, tsup/unbuild-сборка или прямые source-импорты через `transpilePackages` в Next — проще второе).
- Перенести из `apps/web-prototype/src/index.css` тему целиком: `@theme`-токены (палитра, шрифты, кривые), утилиты (`border-warm`, `card-glow`, `text-shimmer`, `scrollbar-thin`), keyframes. Это уже готовый, отлаженный файл — переносится почти без изменений.
- Подключить шрифты через `next/font` (Playfair Display, Crimson Text, IM Fell English — Google Fonts, self-hosted сборка next/font; Courier New — системный). Обязательно `display: swap` и подстраховка метрик (fallback adjust), т.к. донагрузка шрифтов сдвигает разметку — а это ровно тот класс багов, что описан в §6.3.1 дизайн-системы.
- Перенести фоновые слои: `BackgroundTexture` (бумажный шум + волокна + пятна, офскрин-генерация) и `BackgroundFX` (золотая пыль + scan-линия). Оба — клиентские компоненты, рендерятся один раз в корневом layout, с `prefers-reduced-motion`-деградацией в статичную текстуру.
- Перенести хуки `useCanvasLoop` (RAF с паузой по видимости) и `useReducedMotion`.
- Настроить Storybook в `packages/ui` с тёмным фоном-обёрткой и шрифтами.

**Компоненты этапа и источники.**

| Компонент | Источник |
|---|---|
| Тема/токены | Прототип: `index.css` — перенос as-is |
| `BackgroundTexture`, `BackgroundFX` | Прототип — перенос |
| `useCanvasLoop`, `useReducedMotion` | Прототип — перенос |
| Шрифтовая настройка | Самим (next/font-обвязка, ~30 строк) |

**Библиотеки.** tailwindcss v4, @tailwindcss/postcss (для Next), clsx, tailwind-merge, class-variance-authority, Storybook.

**Паттерн для разработчика.** Пакет не собирается в dist на первом шаге: Next `transpilePackages: ['@country-decision-atlas/ui']` + `exports` на исходники. Vite-прототип импортирует те же исходники. Один источник токенов — один файл `theme.css`, который импортируют оба приложения.

**Критерий готовности.** В Storybook и в пустой странице `apps/web` виден фирменный фон с текстурой и типографикой; Lighthouse не показывает CLS от шрифтов; переключение `prefers-reduced-motion` останавливает пыль и scan-линию.

### Этап 2 — Библиотека примитивов

**Критерий готовности.** Storybook покрывает 100% примитивов; дизайн-ревью галереи против макета `countryatlas_platform.html`; клавиатурная навигация и фокус-обводка работают в каждом интерактивном примитиве.

### Этап 3 — Каркас приложения, слой данных, auth

**Описание.** Переводится «скелет» `apps/web`: корневой layout, навигация, провайдеры данных и аутентификации, обработка ошибок, аналитика, фиче-флаги. После этого этапа приложение выглядит фирменно на уровне хрома, а страницы внутри ещё старые.

**Цели и задачи.**
- **AppShell**: новый `TopBar` (за основу — прототипный: логотип с shimmer, навигация с letter-spacing-hover, кнопка входа/аккаунта), футер с дисклеймером и REF-маркировкой, мобильная навигация (Vaul-шторка). Прототипный `HorizontalPager` в каркас продукта **не** переносится — это паттерн промо-страницы (см. этап 5).
- **Слой данных**: обёртка `openapi-fetch`-клиента с базовым URL из env, интерсептор error-envelope бэкенда (код + детали → типизированная ошибка), QueryClientProvider с дефолтами (staleTime по типам данных: контент — минуты, live-метрики — секунды), инвалидация по доменным ключам (`['country', slug, 'trust']`).
- **Auth**: рескин Login/Register форм (RHF + Zod), сессия — как сейчас (cookie), `AuthProvider` остаётся, добавляется guard-паттерн для приватных сегментов (`/account`, `/internal`) и редиректы.
- **i18n**: внедрить next-intl, вынести строки хрома в словари ru/en, включить locale-in-URL для публичных сегментов (middleware). Контентные ответы бэкенда продолжают приходить уже локализованными.
- **Feature flags**: провайдер, читающий `/platform/features` (SSR-подгрузка + клиентский кэш), компонент-гейт `<Feature flag="...">`.
- **Аналитика**: хук `useAnalyticsEvent` → `POST /analytics/events` (fire-and-forget, батчинг не нужен на старте).
- **Ошибки**: глобальные `error.tsx`/`not-found.tsx` в фирменном стиле («архивная» страница-заглушка с мономаркировкой кода ошибки).
- **⌘K-палитра** (cmdk): каркас с секциями «Страны / Разделы / Действия», данные — из `/search` (подключение поиска — этап 5, здесь только оболочка).

**Компоненты и источники.**

| Компонент | Источник |
|---|---|
| TopBar | **Прототип** — перенос + продуктовая навигация |
| Футер | **Самим** (простой, по макету full_v1) |
| Мобильная навигация | **Vaul** + самим |
| API-клиент | **openapi-fetch** + самим (обвязка ошибок) |
| Провайдеры Query/Auth/Flags/Intl | **Библиотеки** + самим |
| error/not-found страницы | **Самим** |
| Палитра ⌘K | **cmdk** + стилизация |

**Библиотеки.** @tanstack/react-query, openapi-fetch, next-intl, nuqs, cmdk.

**Паттерн для разработчика (данные).** Один модуль на домен: `entities/country/api.ts` экспортирует query-опции (`countryTrustQuery(slug, locale)`) — фабрики `queryOptions`, используемые и в RSC (prefetch + HydrationBoundary), и в клиенте. Мутации соседствуют с инвалидацией там же. Feature-компоненты не знают про fetch — только про хуки домена.

**Критерий готовности.** Все страницы открываются внутри нового AppShell без регрессий; вход/выход/регистрация работают; локаль переключается и сохраняется в URL; ошибка API рендерит фирменный ErrorState; события аналитики уходят.

### Этап 4 — Библиотека визуализаций данных

**Описание.** Переносятся все фирменные чарты прототипа, унифицируется их API, каждый получает «продуктовый» режим (реальные данные, без синтетического дыхания там, где это неуместно).

**Цели и задачи.**
- Перенести: `RadarChart` (CII-профиль), `SparklineChart`, `DivergingMeter` (velocity/drift ±), `ProgressRing`, `GaugeArc`, `DonutChart`, `Heatmap`, `RankFlow` (bump-chart рангов), `BarColumns`, `chartColors.ts`.
- Перенести составные: `DriftBoard` (live-доска строк со спарклайнами и drawer-досье), `PassportCard` (документ с тумблерами и пересчётом), `Counter`.
- Ввести у каждого чарта два режима: `live` (лёгкое «дыхание» вокруг базовых значений — для витринных/обзорных экранов) и `static` (значение = данные, анимация только на въезде — для аналитических экранов, где пользователь читает точные цифры). Дыхание никогда не искажает подписанные числовые значения — колышется только отрисовка.
- Общий контракт данных чартов: `{ label, value, min?, max?, accent?, verifiedAt?, confidence? }` — чтобы бейджи прозрачности подключались единообразно через `ChartFrame`.
- Недостающие визуализации: **временная шкала правовых сигналов** (горизонтальная ось времени, точки-события с категорией и силой влияния — строится самим на SVG + `d3-scale`; библиотечные таймлайны не дадут нужной эстетики), **мини-карта весов** (горизонтальные полосы вкладов критериев в скор — самим, тривиально).
- Все чарты — client-only (`next/dynamic`, `ssr: false`), с серверным плейсхолдером-скелетоном той же геометрии (без CLS).

**Компоненты и источники.**

| Компонент | Источник |
|---|---|
| Radar, Sparkline, DivergingMeter, ProgressRing, GaugeArc, Donut, Heatmap, RankFlow, BarColumns | **Прототип** — перенос |
| DriftBoard, PassportCard | **Прототип** — перенос |
| Таймлайн сигналов | **Самим** (+ d3-scale, d3-shape как утилиты) |
| Полосы вкладов критериев | **Самим** |

**Библиотеки.** d3-scale, d3-shape (только утилиты; не тянуть весь d3).

**Паттерн для разработчика.** Canvas-чарты подписываются на общий «реестр видимости»: один IntersectionObserver в контексте, чарты регистрируются и получают `isVisible`; RAF-цикл (`useCanvasLoop`) стартует только у видимых. DPR-скейлинг канваса обязателен. Для `static`-режима — одна анимация въезда (lerp от нуля к значению по `--ease-arrive`-профилю) и остановка цикла.

**Критерий готовности.** Все чарты в Storybook с live/static и reduced-motion сториз; страница с 6+ чартами держит 60fps (профилирование в DevTools); при скрытии вкладки RAF-циклы останавливаются.

### Этап 5 — Главная, каталог стран, поиск

**Описание.** Первые продуктовые экраны в новом стиле: «витринная» главная и рабочий каталог. Здесь дизайн-система впервые встречается с реальными данными.

**Цели и задачи.**
- **Главная** (`/`, данные `home/overview`): hero-блок с shimmer-заголовком и мономаркировкой (`REF № CA-…`, live-индикатор), сводка платформы (страны/источники/сигналы — `Counter`), лента «что изменилось» (`SignalTicker` + список), вход в мастер решений (главный CTA — чернильная кнопка), секции-представления модулей со scroll-reveal. Прототипный revolver-deck (`HorizontalPager`) используется здесь **опционально** как промо-секция «возможности платформы» — строго по правилам §6.3 (три зоны, мобильная деградация в scroll-snap, reduced-motion в колонку). Если сроки жмут — deck откладывается без ущерба, обычные секции достаточны.
- **Каталог** (`/countries`): грид карточек стран (Card + laser-glow), в карточке — имя, флаг-эмблема, CII-мини (ProgressRing), значок тренда (шалфей/терракота), watchlist-звезда (optimistic-мутация). Фильтры/сортировка в nuqs. Пагинация — keyset с бэкенда.

- Скелетоны каталога — сетка мерцающих карточек той же геометрии.

**Компоненты и источники.**

| Компонент | Источник |
|---|---|
| Hero, секции главной | **Самим** из примитивов (референс — HeroSlide прототипа и intro макета) |
| Revolver-deck (промо, опция) | **Прототип** `HorizontalPager` + `MobileStack` — перенос |
| Карточка страны каталога | **Самим** из Card/ProgressRing/бейджей |
| Фильтр-панель | **Самим** (Select/Toggle из этапа 2 + nuqs) |

### Этап 6 — Досье страны

**Описание.** Центральный и самый насыщенный экран продукта — страница страны как «архивное досье»: все вычисляемые слои бэкенда на одном полотне. Существующие 10 feature-блоков `apps/web` (country-card, drift, trust, data-journal и т.д.) рескинятся на новую библиотеку.

**Цели и задачи.**
- Шапка досье: название (Playfair, крупно), мономаркировка (`slug`, дата верификации, статус локализации — `LocaleStatusBadge`), действия (watchlist, «в сравнение», share).
- Навигация по досье: липкие Tabs или якорная боковая rail-навигация (по образцу `#rail` из макета) — секции: Обзор · Скоры · CII · Тренды · Доверие · Сигналы · Источники · Маршруты · Журнал данных · Сообщество.
- Секции и их визуализации:
  - **CII** (`/cii`) — RadarChart профиля + разбор по компонентам (полосы вкладов);
  - **Скоры по сценариям** (`/scores`) — MetricCard-грид с GaugeArc и score-label (weak…excellent — цветовая семантика);
  - **Платформенные метрики** (`/platform-metrics`) — DataTable + Sparkline на строку;
  - **Доверие** (`/trust`) — ProgressRing уверенности + DonutChart структуры источников + FreshnessBadge (образец — TrustSlide прототипа);
  - **Дрейф** (`/drift`, `/what-changed`) — DriftBoard с drawer-досье изменения (образец — DriftSlide);
  - **Правовые сигналы** (`/legal-signals`) — TimelineList + связка с реестром этапа 8;
  - **Журнал данных** (`/data-journal`) — TimelineList записей;
  - **Источники** (`/sources`) — EvidenceCard-список с ConfidenceBadge;
  - **Маршруты** (`/routes`) — карточки маршрутов легализации со ступенями (шаги — TimelineList), ссылка на детальную `/routes/{id}`;
  - **Сообщество** (сводка user-stories) — врезка-цитата (IM Fell English) + счётчики.
- Совместимость направлений (`destination-compatibility`, `country-pairs`) — блок «если вы из …» с выбором страны происхождения (Select) и DivergingMeter по осям совместимости.
- RSC-префетч карточки (`/card` — агрегат) + ленивые клиентские подзапросы тяжёлых секций при скролле (IntersectionObserver-триггер query).

**Компоненты и источники.**

| Компонент | Источник |
|---|---|
| Все чарты секций | **Этап 4** (прототип) |
| Rail-навигация досье | **Самим** (референс — `#rail` макета) |

| Drawer досье изменения | **Прототип** DriftBoard/Drawer |
| Блок совместимости | **Самим** из примитивов |

**Библиотеки.** Новых нет.

**Паттерн (ленивая загрузка секций).** Каждая тяжёлая секция — самостоятельный клиентский компонент с собственным query, `enabled: isNearViewport`. Скелетон повторяет геометрию. Так первая отрисовка досье остаётся лёгкой, а полные данные догружаются по мере чтения.

### Этап 7 — Механика решений (ядро продукта)

**Описание.** Сценарий «получить решение»: мастер → расчёт → разбор → сравнение → паспорт. Самый важный пользовательский путь; на нём сходятся персонализация (веса, персоны) и прозрачность (разбор скора).

**Цели и задачи.**
- **Мастер** (`decision/wizard/resolve`): пошаговая форма (шаги: цель/сценарий → страна происхождения → приоритеты/персона → ограничения) с RadioCards и прогресс-индикатором; состояние шагов — в URL (nuqs), чтобы «поделиться незаконченным мастером» работало.
- **Запуск и результат** (`decision/run`): экран результата — ранжированный список стран с разбором (`DecisionBreakdown` из `apps/web` — рескин): полосы вкладов критериев, штрафы/бонусы, confidence. Анимация появления результата — AnalysisOverlay-паттерн прототипа (краткий «прогон анализа» с мономаркировкой стадий) — это фирменный момент продукта.
- **Персонализация**: слайдеры весов (Radix Slider, стилизация «гравированная шкала»), пресеты-персоны (`personas`), сохранение профилей весов (`me/weight-profiles`, CRUD). Изменение весов — живой пересчёт через `decision/compare` с debounce.
- **Сравнение** (`decision/compare`, до 3–4 стран): колонки-досье рядом, DivergingMeter по критериям, подсветка лидера по строке.
- **Матрица** (`countries/matrix`): сводная страны×сценарии (рескин `compare-matrix` фичи), ячейка — мини-скор с цветом семантики; RankFlow как альтернативный вид «движение рангов».
- **Паспорт решения** (`decision/passports`): предметный экран-документ на базе PassportCard прототипа — «архивный документ» с реквизитами, факторами-тумблерами (что учтено), итоговым скором (Counter), QR/ссылкой шаринга; публичная страница `/decision/passports/{token}` — read-only версия того же документа. Экспорт в поездку (`me/trips/from-passport`) — CTA на паспорте.

**Компоненты и источники.**

| Компонент | Источник |
|---|---|
| PassportCard | **Прототип** — перенос и подключение к данным |
| AnalysisOverlay | **Прототип** — перенос |
| Слайдеры весов | **Radix Slider** + стилизация самим |
| DecisionBreakdown, wizard-логика | **apps/web** — рескин |
| RadioCards шагов | **Этап 2** |
| Матрица | **apps/web** compare-matrix — рескин + RankFlow |

**Библиотеки.** Новых нет.

**Паттерн (живой пересчёт).** Изменение весов пишется в локальный стейт мгновенно (слайдер отзывчив), пересчёт — debounce 400–500ms → mutation → результат приходит и **лерпится** от старых значений к новым (числа через Counter, полосы через CSS-transition) — дискретной подмены цифр быть не должно (правило §1.3).

**Критерий готовности.** Полный путь «мастер → результат → сравнение → паспорт → публичная ссылка» проходит в новом стиле без консольных ошибок; E2E-сценарий пути записан в Playwright.

### Этап 8 — Знания и прозрачность: сигналы, источники, методология, глоссарий

**Описание.** «Библиотечный» контур платформы — реестры и справочники, на которых держится доверие: правовые сигналы, источники/доказательства, методология, глоссарий.

**Цели и задачи.**
- **Правовые сигналы** (`/legal-signals`): реестр с фильтрами (страна, категория, влияние, период — nuqs) + **таймлайн-режим** (`legal-signals/timeline`) на новой SVG-шкале времени (этап 4); деталка сигнала — Drawer с доказательствами (`{id}/evidence`) и затронутыми странами.
- **Источники** (`/sources`): реестр с ConfidenceBadge/FreshnessBadge, деталка с evidence-items; перекрёстные ссылки: источник ↔ сигнал ↔ страна (единый паттерн «связанные материалы» — список ссылок с мономаркировкой типа).
- **Методология** (`/methodology`): длинная контентная страница — Playfair-заголовки, Crimson-текст с interline 1.7–1.8, Accordion для разделов, параметры методологии (`/methodology/parameters`) — DataTable с текущими порогами. Врезки-принципы — IM Fell English.
- **Глоссарий** (`/glossary`): алфавитный список терминов, деталка-Popover при упоминании термина в других экранах (обёртка `<GlossaryTerm slug>` — самим, подчёркивание пунктиром + Popover с определением).
- Всё контентное — RSC, статически кэшируемое, идеальное SEO.

**Компоненты и источники.**

| Компонент | Источник |
|---|---|
| Таймлайн сигналов | **Этап 4** (самодельный) |
| Реестры | DataTable + TanStack Virtual (длинные списки) |
| GlossaryTerm-поповер | **Самим** + Radix Popover |
| Контентные страницы | Примитивы этапа 2 |

**Библиотеки.** @tanstack/react-virtual.

**Критерий готовности.** Реестры фильтруются и шарятся по URL; таймлайн читаем на десктопе и мобильном (на мобильном — вертикальная деградация); глоссарные термины подсвечиваются минимум на досье страны и в методологии.

### Этап 9 — Личный кабинет: безопасность, watchlist, лента, поездки

**Описание.** Авторизованный контур пользователя. Много форм и CRUD — здесь окупаются Field-обвязка и мутации этапа 2–3.

**Цели и задачи.**
- **Аккаунт** (`/account`): профиль, привязка Telegram (`auth/telegram/link` — пошаговая инструкция с кодом в мономаркировке), уведомления безопасности (`security-notifications` + ack), **сессии** (`auth/sessions`): DataTable устройств с revoke/revoke-all (терракотовые действия с подтверждением через Dialog).
- **Watchlist** (`/watchlist`): грид карточек-стран (как каталог) + сводка изменений по наблюдаемым; пустое состояние — приглашение в каталог.
- **Подписки и лента** (`me/subscriptions`, `feed`): управление подписками (страна/тип события), лента — TimelineList с группировкой по дням, значками типов (сигнал/дрейф/источник) и ссылками в досье.
- **Планировщик поездок** (`me/trips/*`) — мини-приложение:
  - список поездок → деталка поездки;
  - **waypoints**: упорядоченный список точек с drag-and-drop реорганизацией (`waypoints/reorder`);
  - **чек-лист**: секции, чекбоксы (optimistic), импорт шаблона (`checklist/import`);
  - **напоминания**: список с датами (date-fns форматирование);
  - **предупреждения** (`warnings`) и **what-changed поездки** — терракотовые врезки;
  - **share/export**: публичная ссылка (`trips/shared/{token}` — read-only страница) и экспорт.
- Всё приватное — клиентские компоненты с Query, guard из этапа 3.

**Компоненты и источники.**

| Компонент | Источник |
|---|---|
| Формы кабинета | Field-обвязка этапа 2 (RHF+Zod) |
| Таблица сессий | DataTable |
| Лента | TimelineList (прототип) |
| Drag-and-drop waypoints | **dnd-kit** (лёгкий, headless) — либо кнопки вверх/вниз как fallback |
| Чек-лист, напоминания | Примитивы + самим |

**Библиотеки.** @dnd-kit/core + @dnd-kit/sortable (MIT).

**Паттерн (optimistic-CRUD).** Все переключения чек-листа/звёздочек/подписок: optimistic update → rollback по ошибке + Sonner-тост терракотой. Списки инвалидируются доменными ключами.

**Критерий готовности.** Полный CRUD поездки проходит вручную и в E2E; revoke сессии подтверждается диалогом и реально разлогинивает другое устройство (проверка в двух браузерах).

### Этап 10 — Сообщество: истории, Q&A, борд, авторские метрики, заявки стран

**Описание.** Самый широкий по числу экранов этап — четыре подсистемы UGC. Дизайн-акцент — слива (правило §2.3). Всюду статусы модерации и репутация.

**Цели и задачи.**
- **Истории пользователей** (`user-stories`): лента карточек-историй (цитатная типографика IM Fell для выдержек), рейтинги по 6 осям (`community/user-story-ratings` — компактный DivergingMeter-профиль), маркировка synthetic-историй, форма подачи истории.
- **Вопросы-ответы** (`community/questions`, `answers`, `votes`): список вопросов по стране, ветка ответов, голосование (optimistic), статусы модерации (StatusBadge), форма «сообщить об ошибке в данных» (`data-error-reports`) — доступна с любого показателя досье (иконка-флажок у ChartFrame).
- **Migration Board** (`migration-board/*`, `me/threads`) — «доска попутчиков»:
  - публичная лента постов (фильтры: направление, период, цель);
  - деталка поста + запрос контакта (`contact-requests`);
  - входящие/исходящие запросы, accept/decline/cancel/report, блокировки;
  - треды сообщений (`me/threads/{id}/messages`) — простой чат-вид (без realtime на старте: поллинг Query с refetchInterval 10–15s; realtime — отдельная задача после интеграционного транша);
  - мои посты: черновик → submit → статусы модерации; matches (`posts/{id}/matches`) — подборка совпадений.
- **Авторские метрики** (`me/author-metrics`, `authors/*`): студия — CRUD определения метрики, значения по странам, submit на ревью, fork чужой метрики; публичный профиль автора с репутацией (ProgressRing) и его метриками; на досье страны — секция авторских метрик (уже заложена в этапе 6).
- **Заявки стран** (`me/country-proposals/*`) — «студия контрибуции», самый сложный CRUD: заявка → источники → доказательства → сигналы → метрики → таймлайн → карточка-превью по локали → submit. UI-паттерн: мастер-страница заявки с боковым прогрессом разделов (rail) и readiness-индикатором (ProgressRing по полноте), каждый раздел — своя форма-таблица.
- Все формы UGC — с PII-подсказками (бэкенд валидирует; фронт дублирует мягким предупреждением «не публикуйте контакты в открытом тексте»).

**Компоненты и источники.**

| Компонент | Источник |
|---|---|
| Карточка истории, цитаты | Примитивы + самим |
| Голосование, статусы | Самим + бейджи |
| Чат-тред | **Самим** (простой список + input; библиотека чата не нужна) |
| Мастер заявки страны | **Самим** из примитивов (крупная задача) |
| Профиль репутации | ProgressRing + MetricCard |

**Библиотеки.** Новых нет.

**Критерий готовности.** Каждая подсистема проходит полный жизненный цикл (создать → submit → увидеть статус) на локальном стенде; модерационные статусы отображаются во всех состояниях; терракота/шалфей-семантика согласована.

### Этап 11 — AI-ассистент

**Описание.** Инструментальный AI-контур: вопросы с цитатами, объяснение чисел, распознавание намерения. Бэкенд fake-by-default — фронт строится полностью, провайдер подключится позже без изменений UI.

**Цели и задачи.**
- Страница ассистента (`/assistant`): форма вопроса, карточка ответа с **обязательными цитатами-источниками** (`AICitationsList` — рескин), состояние отказа (`AIRefusalState` — «архивная» вежливая заглушка с объяснением причин), дисклеймер всегда видим.
- **Explain-number**: кнопка «?» у ключевых чисел досье (интеграция в ChartFrame) → Popover/Drawer с объяснением «из чего сложилось число» и ссылками на источники.
- **Decision-intent**: свободный ввод на главной/мастере («хочу уехать с семьёй на юг Европы под удалёнку») → распознанный сценарий+параметры → предзаполнение мастера решений (этап 7).
- Стриминг ответа не требуется (бэкенд синхронный); анимация «печати» — умеренная, с reduced-motion-отключением.

**Компоненты и источники.**

| Компонент | Источник |
|---|---|
| Вся фича-логика | **apps/web** `features/ai-assistant` (8 компонентов) — рескин |
| Explain-кнопка в ChartFrame | Самим |

**Библиотеки.** Новых нет.

**Критерий готовности.** Ответ без цитат не отображается как достоверный (UI-инвариант); refusal-путь красив и понятен; explain-number доступен минимум для CII и скоров сценариев.

### Этап 12 — Внутренний контур (админ/модерация)

**Описание.** Служебные экраны для владельца/модераторов. Стиль — та же система, но плотнее и без витринных анимаций: канцелярия архива, а не витрина. Приоритет — скорость работы оператора.

**Цели и задачи.**
- Каркас `/internal`: отдельный layout (компактный TopBar, боковое меню очередей), role-guard.
- **Очереди модерации** (TanStack Table): community (вопросы/ответы/рейтинги/error-reports), migration board (посты/жалобы), user stories, авторские метрики (approve/reject), заявки стран (полный workflow: readiness-check → request-changes → publish/reject/archive, назначение куратора).
- **Данные**: data-quality report (рескин существующей страницы — статусы чеков, шалфей/терракота), contradiction-candidates, AI-drafts (генерация/статусы), evidence/sources/legal-signals CRUD.
- **Пользователи**: список, роль, статус, сессии, revoke-all.
- **Translation jobs**: очередь, create-missing/stale, process-batch, retry-failed — таблица с батч-действиями.
- **Recompute-панель**: кнопки пересчёта (drift/trust/platform-metrics) с подтверждением и мономаркированным журналом ответов.
- Паттерн всех очередей един: таблица + фильтры статусов + Drawer-деталка + действия с подтверждением + Sonner-результат. Один generic-компонент `ModerationQueue` с колонками-конфигом — строится самим один раз, переиспользуется во всех очередях.

**Компоненты и источники.**

| Компонент | Источник |
|---|---|
| ModerationQueue (generic) | **Самим** поверх TanStack Table |
| Формы действий | Field-обвязка |
| Данные-страницы | apps/web `internal/*` — рескин и расширение |

**Библиотеки.** @tanstack/react-table.

**Критерий готовности.** Модератор проходит полный цикл по каждой очереди без обращения к API вручную; действия логируются тостами; опасные действия (reject, revoke-all, hide) — только через подтверждающий Dialog.

### Этап 13 — Полировка и выпуск

**Описание.** Сквозная доводка качества перед тем, как объявить фронтенд завершённым.

**Цели и задачи.**
- **Доступность**: aудит клавиатурной навигации всех интерактивных путей; контраст-аудит пар токенов (особенно `--c3` на `--bg2` для мелкого текста — по результатам поднять роли); aria-разметка чартов (текстовые summary для скринридеров: «CII 71 из 100, уверенность высокая»); reduced-motion-прогон всех экранов.
- **Производительность**: бюджеты (LCP < 2.5s, INP < 200ms, CLS < 0.1 на ключевых страницах); bundle-аудит (`next build` analyze) — чарты и тяжёлые фичи только в lazy-чанках; изображения-эмблемы через next/image; кэш-политики RSC.
- **i18n-полнота**: прогон ru/en по всем экранам, отсутствие «сирот» вне словарей (lint-правило next-intl).
- **Тесты**: миграция существующего Playwright E2E-набора (284 теста) на новые селекторы — обновлять по мере этапов, здесь — финальная зачистка; Vitest-покрытие утилит и хуков данных; MSW-фикстуры для Storybook interaction-тестов; визуальная регрессия ключевых экранов (Playwright screenshots) — минимум: главная, каталог, досье, результат решения, паспорт.
- **Наблюдаемость фронта**: границы ошибок с отправкой в `analytics/events` (тип `client_error`) — без внешних SaaS до интеграционного транша.
- Удаление legacy: старый `styles.css`, неиспользуемые компоненты MVP, мёртвые импорты.
- Финальный проход по чек-листу этапа 0 — все страницы отмечены «мигрировано».

**Критерий готовности (Definition of Done фронтенда).**
1. Все 29+ маршрутов работают в новой дизайн-системе; legacy-CSS удалён.
2. E2E зелёный; визуальные снапшоты зафиксированы.
3. Бюджеты производительности выполняются на демо-данных.
4. Reduced-motion, клавиатура, контраст — проверены и запротоколированы.
5. Ни одного экрана без: даты верификации данных, confidence там, где он есть в API, и дисклеймера там, где выводится рекомендация.

## 9. Порядок работ, параллелизм, зависимости

```text
Этап 0 ──► Этап 1 ──► Этап 2 ──► Этап 3 ──► Этап 4
                                    │
      ┌──────────────┬──────────────┼──────────────┬─────────────┐
      ▼              ▼              ▼              ▼             ▼
   Этап 5         Этап 6         Этап 7         Этап 8        Этап 9
 (главная,      (досье          (решения)     (знания)      (кабинет)
  каталог)       страны)
      └──────────────┴──────┬───────┴──────────────┴─────────────┘
                            ▼
                 Этап 10 (сообщество) ──► Этап 11 (AI) ──► Этап 12 (админ)
                            │
                            ▼
                        Этап 13 (полировка)
```

- Этапы 1–4 — строго последовательный фундамент (одна пара сильных разработчиков).
- Этапы 5–9 параллелятся по одному разработчику на домен после этапа 4; этап 6 желательно начинать первым (самый большой) .
- Этап 10 крупнее остальных — закладывать двойной объём или дробить на 10a (истории+Q&A), 10b (борд), 10c (метрики+заявки).
- Этап 12 может идти параллельно 10–11 силами отдельного разработчика (generic-очередь изолирована).
- Playwright-обновления делаются **внутри** каждого этапа (страница мигрировала — селекторы обновлены), а не копятся до этапа 13.

## 11. Что считается завершением всего проекта

Фронтенд считается готовым, когда выполнены критерии этапа 13 (Definition of Done), все feature-флаги новых поверхностей включены на демо-стенде, а команда может показать сквозную демонстрацию: **главная → каталог → досье страны → мастер решения → результат → паспорт → поездка из паспорта → лента изменений** — целиком в языке «тёплого архива разведки», на живых данных бэкенда, без единого экрана старого MVP-стиля.

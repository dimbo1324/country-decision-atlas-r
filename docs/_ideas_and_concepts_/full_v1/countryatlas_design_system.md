# CountryAtlas — Дизайн-система макета «Intelligence Platform»

Версия документа: v1.1 · Основа: `countryatlas_platform.html`
Назначение: справочник для фронтенд-разработчиков, чтобы новые страницы/компоненты собирались в едином визуальном языке без обращения к автору макета.

Изменения в v1.1: зафиксирован баг «бесконечного рендера» горизонтального барабана (кэширование scroll-офсета) и его исправление — см. §6.3.1–6.3.2, §10 (пп. 6–7), §11.

---

## 1. Общая концепция

Стиль называем условно **«тёплый архив разведки»** — благородный тёмный интерфейс, вдохновлённый банкнотной гравировкой и архивным досье, но выполненный современными веб-техниками (плавные анимации, живые данные, отзывчивость на курсор).

Три несущих принципа, которые должны сохраняться в любом новом экране:

1. **Тёмная база, тёплый акцент.** Никогда не холодный чёрный и не холодный синий как основной фон — только тёплый угольно-синий. Единственный «горячий» цвет — золото, и он используется расчётливо, не более чем в 10–15% видимой площади экрана.
2. **Воздух важнее плотности.** Между смысловыми блоками — большие отступы (64–110px по вертикали на десктопе). Один экран = один основной посыл. Если кажется, что блоков многовато — значит, лишнее нужно унести на следующий скролл, а не сжимать текущий.
3. **Данные дышат, а не прыгают.** Любая «живая» цифра/график анимируется плавной интерполяцией (lerp) к новому значению, а не дискретной подменой. Амплитуда колебаний всегда небольшая относительно диапазона — ощущение живого потока, а не глюка.

### 2.1 Базовые поверхности (нейтрали)

| Токен | HEX | Назначение |
|---|---|---|
| `--bg` | `#0c0e16` | Базовый фон страницы |
| `--bg2` | `#13161f` | Карточки, панели, поверхности первого уровня |

Принцип: чем «выше» элемент в визуальной иерархии (важнее, ближе к пользователю), тем светлее его фон — но разница всегда небольшая (несколько процентов яркости), никаких резких контрастов между поверхностями.

### 2.2 Текст (тёплая нейтральная шкала)

| Токен | HEX | Назначение |
|---|---|---|
| `--c1` | `#efe6d4` | Основной текст, заголовки — тёплый кремовый, не чистый белый |
| `--c2` | `#c0b6a0` | Вторичный текст, подписи, описания |
| `--c3` | `#827a6a` | Третичный текст, малозначимые подписи |
| `--c4` | `#494436` | Едва заметный текст, лейблы, декоративные подписи |
| `--c5` | `#2a2832` | Границы трек-баров, «пустые» состояния прогресса |

Правило: **никогда не используем чистый белый `#fff`** — это разрушает тёплую атмосферу. Любой светлый текст идёт через `--c1`/`--c2`.

### 2.3 Акцентная палитра

Золото — главный (единственный «премиальный») акцент, используется для CTA, ключевых цифр, hover-состояний навигации, live-индикаторов по умолчанию.

| Токен | HEX | Роль |
|---|---|---|
| `--gold2` | `#9c742c` | Тёмный акцент — бордеры, неактивные состояния |
| `--gold` | `#d8aa4e` | Базовый акцент — иконки, заливки, живые точки |
| `--gold3` | `#f4d089` | Светлый акцент — hover, активный текст, свечение |

Вторичная палитра — для различения категорий/сценариев/модулей друг от друга (не для конкуренции с золотом за внимание):

| Категория | Тёмный | Средний | Светлый | Где используется |
|---|---|---|---|---|
| Синий | `#3a6088` | `#5f93bd` | `#92bce0` | Визы/налоги, «нейтральная» аналитика |
| Терракота | `#8a4e34` | `#c47553` | `#e29a78` | Риски, предупреждения, «ухудшение» |
| Шалфей | `#52684c` | `#7e9876` | `#aac49e` | Позитив, «улучшение», рост |
| Слива | `#5e4660` | `#90709a` | `#b89cc0` | Сообщество, второстепенные модули |

**Правило применения:** у каждого крупного блока/слайда/модуля — ровно один акцентный цвет из таблицы (и его 3 оттенка: тёмный для бордера, средний для иконки, светлый для hover/текста). Цвета никогда не смешиваются внутри одного компонента — это держит визуальную ясность при большом количестве данных.

### 2.4 Семантика цвета (важно закрепить единообразно)

- **Шалфей (sage)** = положительная динамика, рост, «улучшается», совпадение/уверенность.
- **Терракота (terra)** = отрицательная динамика, риск, «ухудшается», предупреждение.
- **Синий (blue)** = нейтральная информация, факты без оценочной окраски.
- **Золото (gold)** = «это важно / это интерактивно / это премиальный элемент».

Не менять местами — пользователь считывает цвет раньше, чем текст.

### 2.5 Границы и прозрачности

Границы почти никогда не сплошные — используется полупрозрачный тёплый белый:

```css
--border:     rgba(239,230,212,0.08–0.10);   /* базовая тонкая граница */
--border-hi:  rgba(239,230,212,0.24–0.28);   /* граница при hover/фокусе */
--glow:       rgba(216,170,78,0.04–0.06);    /* золотая подсветка hover-фона */
```

Правило: базовая граница почти невидима (8–10% непрозрачности), и «проявляется» только при взаимодействии (hover/focus) до 24–28%. Это создаёт эффект «интерфейс просыпается под курсором».

### 3.2 Правила начертания по ролям

- **Заголовки (H1–H2):** Playfair Display, `font-weight: 700`, часто с курсивным акцентным словом внутри (`<em>`) — курсив всегда красится в акцентный цвет блока.

- **Цитаты и сноски:** IM Fell English, всегда `font-style: italic`, цвет `--c2` или акцентный светлый тон.

### 3.3 Логотип/hero-заголовок — приём «shimmer»

Крупный логотип/заголовок использует бегущий градиентный блик через `background-clip: text`:

```css
background: linear-gradient(105deg, var(--c1) 30%, var(--gold3) 45%, var(--c1) 60%);
background-size: 220% auto;
-webkit-background-clip: text;
-webkit-text-fill-color: transparent;
animation: shimmer 6-7s linear infinite;
```

Используется точечно — только для главного логотипа/названия платформы на входном экране, не для рядовых заголовков секций.

## 4. Сетка, отступы, пространство

- **Максимальная ширина контента:** 1020–1080px (`--wrap`), с боковыми паддингами 32–34px.
- **Вертикальный ритм секций:** 52–110px паддинга сверху/снизу у каждой смысловой секции. Чем «спокойнее» контент (вступление, цитата), тем больше воздуха.
- **Сетки карточек:** 3 колонки на десктопе (`repeat(3,1fr)`), схлопывание в 1 колонку на мобильных (`<720–760px`).
- **Общее правило плотности:** если в одном блоке больше 4 самостоятельных смысловых единиц (карточек/метрик) — это повод разбить на подсекции, а не сжимать сетку.

## 5. Компоненты

### 5.1 Карточки (cards / tiles)

- Плоский фон `--bg2`, тонкая граница `--border`.
- Верхняя грань — цветная полоска-акцент (`::before`/`::after`, `height:1–2px`), которая **проявляется через `transform: scaleX(0→1)`** слева направо при hover — не через `opacity`. Это ключевой фирменный микро-приём.
- При hover карточка слегка поднимается: `transform: translateY(-3px…-4px)`, граница светлеет до `--border-hi`.
- Иконка внутри карточки — в отдельной рамке-квадрате (`border:1px solid`), не «голая»; при hover рамка подсвечивается акцентным цветом.

### 5.2 Кнопки (CTA)

**Основная кнопка:** прозрачный фон, тонкая рамка акцентного цвета, текст акцентным цветом. При hover — заливка «наезжает» снизу вверх (`translateY(101%→0)`), а не через `background-color` напрямую (даёт ощущение «чернил», а не CSS-переключателя). Текст меняет цвет на цвет фона (`--bg`), создавая эффект инверсии.

**Второстепенная кнопка (ghost):** без рамки, только текст + иконка; при hover иконка увеличивается (`scale(1.2)`) или сдвигается.

### 5.3 Панели/drawer'ы (выезжающие боковые панели)

Правила:
- Overlay — `rgba` тёмный + `backdrop-filter: blur(2–3px)` (размытие контента позади).
- Панель выезжает через `transform: translateX/Y`, **не** через `display`/`width`, с `cubic-bezier(.22,.68,0,1.05–1.1)` — характерная пружинящая, но не «прыгающая» кривая.
- Кнопка закрытия (✕) поворачивается на 90° при hover (`rotate(90deg) scale(1.1)`).
- Каждая панель/модалка имеет **свой** акцентный цвет из вторичной палитры — так пользователь на глаз различает типы панелей друг от друга.

### 5.4 Табличные данные

- Границы только горизонтальные (между строк), никогда вертикальные между колонками.
- При hover на строке: у нижней границы строки анимированное подчёркивание акцентным цветом (`scaleX` слева направо), текст светлеет с `--c2` до `--c1`.
- Численные значения — всегда `Courier New`, текстовые — `Crimson Text`.

### 5.5 Прогресс-индикаторы

- **Линейные бары:** высота 1–3px, заливка растёт через `transition: width`, easing `cubic-bezier(.22,.68,0,1.1)` (лёгкий overshoot/пружина в конце).
- **Кольца (SVG progress ring):** `stroke-dasharray`/`stroke-dashoffset`, анимация через `transition: stroke-dashoffset 1.2–1.4s`, та же кривая.
- **Счётчики чисел:** покадровая интерполяция через `requestAnimationFrame` с easing `1-(1-p)^3` (ease-out cubic) — цифра «долетает» до значения, не выскакивает мгновенно.

### 5.6 Живые графики (Canvas)

Общий паттерн «дыхания данных», используемый везде (радар, бары, тайм-лайн, heatmap, счётчики):

1. Для каждой метрики хранится `base` (базовое значение), `target` (текущая цель) и `current` (то, что реально рисуется).
2. Раз в 1.5–3 секунды генерируется новый `target` рядом с `base` (± небольшая амплитуда).
3. На каждом кадре `current` лерпится к `target`: `current += (target - current) * 0.045–0.06`.
4. Отрисовывается именно `current`.

Результат — данные плавно «плывут», никогда не дёргаются дискретно. Амплитуда всегда маленькая относительно диапазона шкалы (иначе выглядит как баг, а не как живой поток).

**Важное правило производительности:** если на странице несколько таких живых графиков (например, в горизонтальном барабане), **анимируется и перерисовывается только тот, что сейчас находится в видимой области** (через `IntersectionObserver` или индекс активного слайда). Остальные — на паузе. Это принципиально для FPS при 5+ канвасах на странице.

## 6. Анимации и движение

### 6.1 Тайминги и кривые (единый словарь)

| Кривая/тайминг | Где используется |
|---|---|
| `cubic-bezier(.22,.68,0,1.1)` | Прогресс-бары, заполнение колец, раскрытие аккордеона — «уверенный приход с лёгким нахлёстом» |
| `cubic-bezier(.34,1.36–1.56,.64,1)` | Повороты иконок, магнитные/пружинящие эффекты — «живая пружина» |
| `cubic-bezier(.16,1,.3,1)` | Scroll-reveal появление секций — плавный, без переброса |
| `ease-out cubic` (`1-(1-p)^3`) | Счётчики чисел, программная интерполяция |
| Длительность коротких hover-эффектов | 0.25–0.4s |
| Длительность крупных переходов (drawer, модалка) | 0.45–0.6s |
| Длительность scroll-reveal | 0.8–0.9s |

### 6.2 Scroll-reveal

Все крупные блоки при первом попадании в вьюпорт: `opacity:0; translateY(24–30px)` → `opacity:1; translateY(0)` через `IntersectionObserver` (`threshold: 0.15–0.18`), однократно (`unobserve` после срабатывания). Не повторяется при повторном скролле вверх/вниз.

### 6.3 Фирменный горизонтальный «барабан» (revolver deck)

Если нужно повторить приём — принципиальная схема:

1. **Три зоны:** вертикальное вступление → sticky-секция с горизонтальным треком → классический вертикальный контент. Никогда не начинать горизонтальный скролл с первого пикселя страницы — нужна «зона приземления».
2. Секция-обёртка высотой `N slides × 100vh` (топливо для скролла), внутри — `position: sticky` контейнер на весь экран, внутри него — flex-трек шириной `N × 100vw`.
3. Прогресс скролла конвертируется в `translateX` трека (не блокируется нативный скролл/инерция — важно для доступности и тачпадов).
4. Обязательные UX-элементы: счётчик слайдов (01/05), кликабельные точки-индикаторы, подсказка «скролл = движение вбок» при входе (исчезает после первого движения), подсказка «продолжить чтение» на выходе.
5. **Мобильная деградация:** на ≤768–820px — нативная горизонтальная carousel со `scroll-snap`, без scroll-hijacking (тач-устройства плохо совместимы с этим паттерном).
6. **`prefers-reduced-motion`:** трек превращается в обычную вертикальную колонку, весь hijacking отключается.

#### 6.3.1 Критическое правило реализации: никогда не кэшировать scroll-офсет

Реализация «барабана» без GSAP/ScrollTrigger (чистый JS) один раз уже приводила к серьёзному багу, поэтому фиксируем как обязательное правило.

**Неверный (ломающийся) подход:** один раз при загрузке/resize вычислить и сохранить в переменную абсолютную позицию секции (`deckStart = scrollY + rect.top`), а на каждый скролл использовать этот закэшированный offset для расчёта прогресса.

Проблема: любой последующий сдвиг разметки ВЫШЕ секции (донагрузка веб-шрифта и связанный с этим реflow текста, асинхронная подгрузка контента, даже мелкие изменения высоты от resize, обработанного с небольшой задержкой) делает закэшированное значение неверным. Прогресс скролла начинает рассчитываться от неправильной базы — трек «дёргается»/скачет при каждом кадре скролла. Внешне это выглядит как бесконечный дёрганый рендер вверх-вниз и создаёт сильные лаги, потому что рассинхрон только нарастает.

**Верный (самовосстанавливающийся) подход:** никогда не кэшировать позицию. На каждый тик скролла (через `requestAnimationFrame`-throttling) заново запрашивать у браузера актуальное положение секции через `getBoundingClientRect()` и считать прогресс от него:

```js
function updateDeck() {
  const rect = deckOuter.getBoundingClientRect(); // всегда свежее значение
  const total = Math.max(1, rect.height - innerHeight);
  const p = clamp(-rect.top / total, 0, 1);
  track.style.transform = `translate3d(${-Math.round(p * deckDistance)}px,0,0)`;
}
```

Поскольку `rect.top` запрашивается заново каждый раз, любой посторонний сдвиг разметки автоматически учитывается на следующем же кадре — рассинхрону просто неоткуда взяться. Единственное, что стоит кэшировать/пересчитывать реже — горизонтальную дистанцию прокрутки (`deckDistance`, зависит только от ширины вьюпорта), и то с проверкой «а изменилась ли реально ширина» (игнорировать resize-события, где поменялась только высота — например, показ/скрытие адресной строки на мобильных или пересчёт панели встроенного браузера).

**Дополнительные защитные меры, обязательные для этого паттерна:**
- `overflow-anchor: none;` на `html`/`body` — отключает автокоррекцию скролла браузером при изменении разметки выше вьюпорта; без этого браузер может сам «подрёргивать» scrollY, что дополнительно сбивает расчёты.
- Recalculate горизонтальной дистанции — только когда реально изменилась ширина (`innerWidth !== lastMeasuredWidth`), не при любом `resize`.
- Клики по точкам-индикаторам и стрелки клавиатуры также должны считать целевую позицию от **свежего** `getBoundingClientRect()`, а не от закэшированного старта.

#### 6.3.2 Ресайз-обработчики тяжёлых фоновых эффектов — всегда с debounce

Отдельная, не связанная с «барабаном», но важная защита: любой `resize`-обработчик, который перегенерирует что-то дорогое (шумовая текстура, частицы), обязан идти через debounce (150–300ms), а не выполняться на каждое событие `resize` напрямую. Без этого шквал `resize`-событий подряд (например, пока встроенная панель браузера в редакторе кода донастраивает свой размер сразу после открытия файла) вызывает лавину дорогих перерисовок одна за другой — на глаз это ощущается как зависание/лаги ровно в момент открытия страницы.


### 6.4 Курсорные/интерактивные эффекты

- **3D-наклон карточек** (tilt) за курсором: слушаем `mousemove`, вычисляем позицию курсора относительно центра карточки (0..1 по X/Y), применяем `rotateX/rotateY` в пределах ±7–9°, плюс световое пятно (`radial-gradient`), которое следует за курсором внутри карточки.
- **Магнитные кнопки:** кнопка физически сдвигается на 20–35% расстояния от курсора до своего центра, при уходе курсора — плавно возвращается на место с пружинящей кривой.
- Курсорное **свечение на весь экран** (глобальный cursor-glow) — было в ранних итерациях, **сейчас исключено по требованию заказчика**: держим интерактивность локальной (внутри карточек/кнопок), не глобальной.

### 6.5 Фоновые декоративные слои

- **Бумажная/архивная текстура:** лёгкий шум + «волокна» (кривые Безье) + мягкие тёмные пятна, отрисованные на `<canvas>` поверх фона. Для производительности — генерировать шум на маленьком офскрин-канвасе (¼ разрешения) и растягивать, никогда не через попиксельный `ImageData` в полном разрешении экрана (дорого и триггерит `[Violation] requestAnimationFrame handler took Nms`).
- **Плавающие частицы (золотая пыль):** редкие, медленно поднимающиеся вверх точки с мерцанием (`sin`-модуляция прозрачности). Плотность считается от ширины экрана, с потолком (не более ~70 частиц даже на широких мониторах).
- **Scan-линия:** тонкая горизонтальная линия, медленно (9–11s) сверху вниз пересекающая экран — акцентный «живой» штрих, не более.

## 7. Иконография

- Только **inline SVG** (не иконочные веб-шрифты через CDN) — веб-шрифты ломаются при открытии файла через `file://` и создают зависимость от внешней сети.
- Стиль иконок: line-style (обводка, не заливка), `stroke-width: 1.5`, `stroke-linecap/linejoin: round` — единообразная «тонкая гравировка» на всех иконках.
- Иконка почти всегда живёт в собственной рамке-квадрате (`border:1px solid var(--border)`), а не «голая» рядом с текстом — это часть фирменного архивного стиля.

## 8. Взаимодействие между элементами (микро-паттерны)

| Триггер | Реакция | Кривая/тайминг |
|---|---|---|
| Hover на карточке | Подъём (`translateY`), проявление верхней акцентной полосы, посветление рамки | 0.35–0.5s, ease |
| Hover на строке таблицы | Подчёркивание снизу (`scaleX`), текст светлеет | 0.5s, bezier с нахлёстом |
| Hover на nav-пункте | Расширение `letter-spacing`, подчёркивание снизу | 0.3–0.4s |
| Клик на аккордеон | `max-height` 0→N, стрелка поворот 180° | 0.4–0.5s |
| Клик на тумблер | Кружок сдвигается, фон/рамка окрашиваются в акцент | 0.3–0.35s, пружина |
| Открытие drawer/модалки | `translate`/`scale` из «спрятанного» положения + overlay blur | 0.45–0.6s, пружина |
| Появление секции при скролле | Fade + translateY | 0.8–0.9s, ease-out |
| Обновление «живого» значения | Плавный lerp к новой цели, не мгновенная замена | непрерывно, ~0.045–0.06 коэффициент за кадр |

## 9. Тон и голос интерфейса (текстовые паттерны)

- Технические лейблы — **всегда** капсом, моноширинным шрифтом, с большим трекингом: `REF № CA-2024-IX`, `LIVE`, `CONFIDENCE`.
- Цитаты/сноски — курсивом через IM Fell English, с длинным тире перед атрибуцией.
- Дисклеймер «Это не юридическая консультация» — сквозной элемент, должен присутствовать в футере и рядом с любым выводом, похожим на рекомендацию.
- Даты и версии данных всегда явно показаны рядом с показателем (`Verified 2026-07`, `Confidence 0.88`) — прозрачность источника важнее лаконичности.

## 10. Технические принципы реализации (для консистентности между макетами)

1. **Никаких внешних иконочных шрифтов через CDN** — только inline SVG. Google Fonts через `<link>` — допустимо.
2. **`prefers-reduced-motion` обязателен** для любого scroll-hijacking/тяжёлой анимации — деградация в статичную вертикальную раскладку.
3. **Каждый Canvas-график должен проверять видимость** перед перерисовкой (IntersectionObserver либо индекс активного слайда) — не рисовать то, что не видно.
4. **Никаких попиксельных `ImageData`-генераций в полном разрешении экрана** на resize — либо кэшировать, либо генерировать на уменьшенном канвасе с последующим масштабированием, либо дебаунсить.
5. **`resize`-обработчики — только с debounce** (200–300ms), не выполняются на каждое событие.
6. **Никогда не кэшировать абсолютную scroll-позицию** для расчёта прогресса скролл-анимаций (барабан, parallax и т.п.) — считать прогресс заново на каждый кадр через `getBoundingClientRect()`. Кэш неизбежно рассинхронизируется с любым посторонним сдвигом разметки и превращается в дёрганый/бесконечно нестабильный рендер (см. §6.3.1).
7. **`overflow-anchor: none`** на `html`/`body` при любом JS, управляющем высотой секций во время скролла — иначе браузер может сам подкорректировать scrollY, усугубляя рассинхрон.
8. **Все цвета — через CSS-переменные** (`:root`), никогда хардкодом в компонентах — так дизайн-система остаётся единым источником истины и легко подхватывается новыми страницами.

## 11. Чего избегать (антипаттерны для этого стиля)

- Чистый белый/чёрный текст и фоны.
- Холодные оттенки (голубой/циан/ярко-фиолетовый) как основной акцент — только золото.
- Резкие, ступенчатые переходы (`transition: none` или линейный `ease`) на интерактивных элементах — всегда пружинящая/ease-out кривая.
- Глобальное курсорное свечение на весь экран.
- Плотная вёрстка без воздуха между блоками — «одна мысль на экран».
- Мгновенная (не интерполированная) подмена значений в живых графиках.
- Иконочные веб-шрифты через внешний CDN.
- Кэширование абсолютной scroll-позиции для расчёта прогресса скролл-анимаций — приводит к рассинхрону и «бесконечно» дёрганому рендеру при любом постороннем сдвиге разметки.
- Undebounced `resize`-обработчики, перегенерирующие что-то дорогое (шумовые текстуры, частицы) — вызывают лавину перерисовок и ощущение зависания именно в момент открытия/изменения размера окна.

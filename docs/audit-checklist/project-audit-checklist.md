## 1. Архитектура и границы системы

Когда вы отвечаете за проект целиком, главный вопрос — не «работает ли этот кусок кода», а «правильно ли проведены границы между частями системы». Плохие границы — это когда изменение в одном месте требует правок в пяти других, когда два сервиса знают о внутреннем устройстве друг друга, когда бизнес-логика продублирована и начинает расходиться в разных местах.

Важно оценить баланс: избыточно сложная архитектура (микросервисы там, где хватило бы модуля в монолите) так же вредна, как и монолит, который давно пора распилить. Хорошая архитектура — та, что соответствует текущему масштабу команды и нагрузки, но оставляет разумный путь к росту.

Отдельно стоит смотреть на версионирование контрактов: если у вас Go-сервисы общаются между собой или с фронтендом на TypeScript через REST/gRPC/GraphQL, любое изменение формата данных должно быть обратно совместимым либо сопровождаться понятной стратегией миграции клиентов.

### Чек-лист

- [ ] Есть ли явная карта системы (диаграмма сервисов/модулей и их связей), и совпадает ли она с реальностью?
- [ ] Каждый сервис/модуль отвечает за одну зону ответственности (Single Responsibility на уровне сервиса)?
- [ ] Нет ли дублирования бизнес-логики (например, правило расчёта скидки реализовано и в Go-сервисе, и в Python-сервисе, и на фронте)?
- [ ] Есть ли циклические зависимости между модулями/сервисами (A зависит от B, B зависит от A)?
- [ ] Общаются ли сервисы напрямую с чужими базами данных, минуя API (антипаттерн shared database)?
- [ ] Есть ли версионирование API (v1/v2, заголовки версий) и стратегия deprecation старых версий?
- [ ] Изменения в контрактах (DTO, схемы) обратно совместимы (добавление полей — да, переименование/удаление — только через миграционный период)?
- [ ] Не является ли архитектура избыточно сложной для текущего масштаба команды и нагрузки?
- [ ] Не является ли архитектура, наоборот, «монолитом-который-пора-делить» — есть ли явные точки будущего разреза?
- [ ] Задокументированы ли ключевые архитектурные решения и их причины (ADR)?

## 2. Память и ресурсы

Утечки памяти редко проявляются сразу — они убивают проект через недели стабильной работы, когда сервис внезапно падает по OOM в самый неподходящий момент. Ваша задача как senior+ — закладывать привычку профилирования и code review с прицелом именно на жизненный цикл ресурсов: кто создал объект/соединение/горутину и кто гарантированно его освободит.

**Для Go** типичная проблема — горутины, которые запускаются и никогда не завершаются, потому что ждут канал, в который никто не запишет, или потому что забыли передать `context.Context` для отмены операции при завершении родительского запроса. Также стоит проверять `defer` внутри циклов — если он закрывает ресурсы только в конце функции, а не итерации, ресурсы накапливаются.

**Для Python** утечки чаще связаны с циклическими ссылками (объекты держат друг друга и мешают сборщику мусора), незакрытыми файловыми дескрипторами и сетевыми соединениями (особенно если не используется `with`/контекстный менеджер), а также с генераторами и итераторами, которые держат ссылки на большие структуры дольше, чем нужно.

Общая для всего стека проблема — создание нового соединения к БД, Redis или внешнему API на каждый запрос вместо переиспользования пула. Это не только утечка, но и деградация производительности под нагрузкой.

### Чек-лист

**Go:**
- [ ] Все горутины имеют понятное условие завершения (нет горутин «навсегда»)?
- [ ] Используется `context.Context` для отмены долгих операций при отмене родительского запроса?
- [ ] Каналы закрываются той стороной, которая пишет, а не читает?
- [ ] `defer Close()` не накапливается внутри циклов с большим числом итераций?
- [ ] Проведено профилирование через `pprof` (`go tool pprof`) под реалистичной нагрузкой?

**Python:**
- [ ] Файлы, сетевые соединения, БД-курсоры открываются через `with` (контекстные менеджеры)?
- [ ] Нет циклических ссылок между объектами, которые мешают GC (или явно используется `weakref`)?
- [ ] Генераторы не хранят ссылки на большие объекты дольше необходимого?
- [ ] Проведено профилирование через `tracemalloc` / `memory_profiler` / `objgraph`?

**Общее:**
- [ ] HTTP-клиенты и соединения к БД/Redis переиспользуются через пул, а не создаются заново на каждый запрос?
- [ ] Настроены разумные лимиты пулов (max connections, max idle) и таймауты?
- [ ] Есть мониторинг потребления памяти в проде с алертами на аномальный рост (memory leak trend)?
- [ ] Проведено нагрузочное тестирование с наблюдением за памятью в течение длительного времени (не 5 минут, а хотя бы несколько часов)?

## 3. Конкурентность и целостность данных

Это как раз то, что вы назвали «перезаписыванием данных» — race condition, когда два одновременных запроса читают одно и то же состояние, оба вносят изменения, и один результат перезаписывает другой без учёта изменений соседа. Это одна из самых коварных категорий багов: в тестах и при низкой нагрузке всё работает, а под реальной нагрузкой начинаются «необъяснимые» потери данных.

Для Go есть встроенный инструмент — детектор гонок (`go test -race`), который обязательно должен быть частью CI, а не разовой проверкой.

Ключевой вопрос при работе с конкурентными изменениями данных — какую стратегию блокировки вы используете:
- **Оптимистичная блокировка** — добавляете поле `version` или `updated_at`, и при обновлении проверяете, что оно не изменилось с момента чтения. Подходит, когда конфликты редки.
- **Пессимистичная блокировка** — `SELECT ... FOR UPDATE` в Postgres, блокирующий строку до конца транзакции. Подходит, когда конфликты часты и нужна гарантия.

Отдельная тема — идемпотентность. Если у вас есть очереди сообщений, retry-логика или пользователь может дважды нажать кнопку «Оплатить», важно, чтобы повторный вызов той же операции не создавал дублирующийся эффект (двойное списание денег, дублирующийся заказ и т.д.).

### Чек-лист

- [ ] Для Go-сервисов детектор гонок (`go test -race`) включён в CI и проходит без предупреждений?
- [ ] Определены критичные участки кода, где возможна одновременная запись одних и тех же данных?
- [ ] Для таких участков явно выбрана стратегия — оптимистичная или пессимистичная блокировка — и это осознанное решение, а не случайность?
- [ ] Оптимистичная блокировка: есть поле `version`/`updated_at`, и код проверяет его перед записью?
- [ ] Пессимистичная блокировка: используется `SELECT ... FOR UPDATE` или аналог, и блокировки не держатся дольше необходимого (риск deadlock)?
- [ ] Проверено отсутствие deadlock при одновременном захвате нескольких блокировок (единый порядок захвата в разных частях кода)?
- [ ] Операции, которые могут быть вызваны повторно (retry, повторный клик, повтор сообщения из очереди), идемпотентны?
- [ ] Для идемпотентности используется идемпотентный ключ (idempotency key) там, где это критично (платежи, создание заказов)?
- [ ] Выбраны корректные уровни изоляции транзакций в Postgres под конкретный сценарий (`READ COMMITTED` по умолчанию, `REPEATABLE READ`/`SERIALIZABLE` — где действительно нужно)?
- [ ] Проведено тестирование конкурентного доступа (несколько одновременных запросов к одному ресурсу) под нагрузкой?

## 4. База данных (PostgreSQL)

База данных — это то место, где ошибки архитектуры дороже всего исправлять постфактум, потому что данные уже накоплены. Аудит БД должен покрывать и производительность запросов, и безопасность изменений схемы, и операционную готовность (бэкапы, восстановление).

Индексы — первое, на что стоит смотреть: есть ли они под реальные паттерны запросов приложения, а не «под всё подряд» (лишние индексы замедляют запись). Отдельно проверяйте N+1 — классическую проблему, когда ORM или ручной код делает один запрос для получения списка, а затем по одному запросу на каждый элемент списка для связанных данных.

Миграции схемы должны позволять деплой без даунтайма: добавление колонки с `NOT NULL` без дефолтного значения на большой таблице заблокирует её на время миграции — такие вещи нужно делать поэтапно (сначала nullable-колонка, потом заполнение данных, потом constraint).

Не забывайте про операционную часть: бэкапы, которые никогда не тестировались на восстановление — это не бэкапы, а иллюзия безопасности.

### Чек-лист

- [ ] Индексы покрывают реальные паттерны запросов приложения (проверено через `EXPLAIN ANALYZE` на медленных запросах)?
- [ ] Нет лишних неиспользуемых индексов, замедляющих запись?
- [ ] Проверено отсутствие N+1 запросов (через логирование запросов или APM)?
- [ ] Миграции схемы протестированы на совместимость с текущей версией приложения (zero-downtime deploy)?
- [ ] Тяжёлые миграции (добавление NOT NULL колонки, изменение типа на большой таблице) разбиты на безопасные шаги?
- [ ] Долгие транзакции не блокируют другие операции на тех же таблицах?
- [ ] Настроен connection pooling (например, PgBouncer), лимиты `max_connections` разумны и не исчерпываются под нагрузкой?
- [ ] Для больших/растущих таблиц продумана стратегия партиционирования или архивации старых данных?
- [ ] Бэкапы настроены, и **процедура восстановления реально протестирована** (не только настройка бэкапа)?
- [ ] Есть мониторинг медленных запросов (`pg_stat_statements` или аналог) с регулярным разбором top-N самых тяжёлых?
- [ ] Внешние ключи и constraints используются там, где нужна гарантия целостности на уровне БД, а не только на уровне приложения?

## 5. Безопасность

Как тимлид/senior+ вы отвечаете за то, чтобы безопасность не была «задачей одного человека, который когда-то это проверил», а была встроена в процесс: код-ревью, автоматические проверки зависимостей, регулярный пересмотр прав доступа.

Валидация входных данных должна происходить на всех уровнях, где данные пересекают границу доверия — не только на фронтенде (который легко обойти), но и на бэкенде, и в идеале ещё раз на уровне БД через constraints.

Секреты (пароли, токены, ключи API) никогда не должны попадать в код или систему контроля версий — используйте переменные окружения или секрет-менеджеры (Vault, AWS Secrets Manager и т.д.), и заложите процесс ротации ключей.

Авторизация должна быть централизована — если проверка прав доступа разбросана по десяткам хендлеров с копипастой условий, рано или поздно кто-то забудет добавить проверку в новый эндпоинт.

### Чек-лист

- [ ] Входные данные валидируются на бэкенде независимо от валидации на фронте?
- [ ] Секреты (пароли, API-ключи, токены) хранятся вне кода (env-переменные, секрет-менеджер), не коммитятся в git?
- [ ] Настроена ротация ключей и секретов на регулярной основе?
- [ ] Авторизация (RBAC/ABAC) реализована централизованно (middleware/декоратор), а не дублируется в каждом хендлере?
- [ ] Проведён аудит зависимостей на уязвимости (`npm audit`, `pip-audit`, `govulncheck`, Snyk/Dependabot)?
- [ ] Настроен CORS с конкретным списком разрешённых доменов (не `*` в проде)?
- [ ] Реализован rate limiting на публичных эндпоинтах (защита от брутфорса и DDoS)?
- [ ] Фронтенд защищён от XSS (экранирование пользовательского контента, Content Security Policy)?
- [ ] Реализована защита от CSRF там, где используются cookie-based сессии?
- [ ] SQL-запросы используют параметризацию (нет конкатенации строк с пользовательским вводом)?
- [ ] Логи не содержат чувствительных данных (пароли, токены, номера карт, персональные данные)?
- [ ] Настроен HTTPS везде, включая внутренний трафик между сервисами при необходимости?
- [ ] Есть процесс реагирования на обнаруженные уязвимости (кто чинит, в какие сроки)?

## 6. Frontend (React / Next.js / TypeScript / Vite)

На фронтенде аудит фокусируется на трёх вещах: производительность рендеринга, отсутствие утечек в компонентах и дисциплина типизации.

Для Next.js критично осознанно выбирать стратегию рендеринга под каждую страницу: SSR (Server-Side Rendering) для персонализированного контента, который должен быть свежим при каждом запросе; SSG (Static Site Generation) для контента, который редко меняется; ISR (Incremental Static Regeneration) как компромисс между ними; CSR (Client-Side Rendering) для сильно интерактивных приватных разделов, где SEO не важен. Случайный выбор «по умолчанию SSR для всего» бьёт по производительности сервера, а «всё через CSR» бьёт по SEO и первому рендеру.

В React утечки обычно живут в `useEffect` — если вы подписались на событие, таймер, WebSocket или внешний стор, но забыли вернуть функцию очистки, подписка продолжает жить после размонтирования компонента, накапливаясь при каждом повторном монтировании.

TypeScript в strict-режиме — это не бюрократия, а способ поймать целый класс ошибок на этапе компиляции. Обилие `any` фактически отключает эту защиту в конкретных местах, и важно понимать, сколько именно `any` в проекте и почему.

### Чек-лист

**Next.js:**
- [ ] Для каждой страницы осознанно выбрана стратегия рендеринга (SSR/SSG/ISR/CSR), а не взята «по умолчанию»?
- [ ] Нет hydration errors (несовпадение серверного и клиентского рендера) в консоли?
- [ ] Изображения оптимизированы через `next/image` (или аналог), а не через сырые `<img>`?
- [ ] API routes / Server Actions валидируют входные данные так же строго, как отдельный бэкенд?

**React (общее):**
- [ ] Все `useEffect` с подписками (event listeners, таймеры, WebSocket, интервалы) возвращают функцию очистки?
- [ ] Нет утечек через незакрытые подписки на глобальные стейт-менеджеры при размонтировании?
- [ ] Состояние приложения имеет единый понятный источник правды (не смешаны бесконтрольно Context, локальный стейт и внешний store)?
- [ ] Нет лишних ре-рендеров тяжёлых компонентов (проверено через React DevTools Profiler)?
- [ ] Ключи (`key`) в списках стабильны и уникальны (не индекс массива при динамическом списке)?

**TypeScript:**
- [ ] Включён `strict: true` в `tsconfig.json`?
- [ ] Использование `any` минимально и обосновано (или заменено на `unknown` с последующей проверкой)?
- [ ] Типы для API-ответов синхронизированы с реальным бэкендом (вручную, через OpenAPI-генерацию или общие схемы)?

**Vite / сборка:**
- [ ] Настроен code splitting и lazy loading для тяжёлых частей приложения?
- [ ] Размер финального бандла контролируется (bundle analyzer подключён и просматривается периодически)?
- [ ] Проверено отсутствие случайного попадания dev-зависимостей или больших библиотек целиком (например, импорт всего lodash вместо конкретных функций)?

## 7. Отказоустойчивость

Отказоустойчивость — это ответ на вопрос «что происходит, когда что-то идёт не так», а не «что происходит, когда всё идёт хорошо». Любой внешний вызов (к другому сервису, к БД, к стороннему API) должен иметь таймаут — без него один зависший запрос может исчерпать пул соединений и положить весь сервис.

Retry с backoff нужен, чтобы не заваливать и без того перегруженный сервис повторными запросами мгновенно один за другим — экспоненциальная задержка даёт системе шанс восстановиться. Circuit breaker идёт на шаг дальше: если сервис систематически падает, временно прекращаем попытки к нему, чтобы не тратить ресурсы впустую и дать ему время восстановиться.

Для Go-сервисов отдельно важен graceful shutdown: при получении `SIGTERM` (например, при обновлении деплоя в Kubernetes) сервис должен не оборвать текущие запросы резко, а дождаться их завершения в разумный таймаут, после чего завершиться.

### Чек-лист

- [ ] У всех внешних вызовов (HTTP, БД, очереди) установлены явные таймауты?
- [ ] Реализован retry с экспоненциальным backoff для транзиентных ошибок (сетевые сбои, временная недоступность)?
- [ ] Реализован circuit breaker для критичных внешних зависимостей?
- [ ] Go-сервисы корректно обрабатывают `SIGTERM`/`SIGINT` (graceful shutdown, дожидаются активных запросов)?
- [ ] Настроены health checks (liveness) и readiness probes для оркестратора (Kubernetes и т.д.)?
- [ ] Продумано поведение при недоступности зависимого сервиса (graceful degradation — например, показать кэшированные данные вместо ошибки 500)?
- [ ] Есть fallback-логика для некритичных функций (если рекомендательный сервис недоступен — просто не показываем блок рекомендаций, а не роняем всю страницу)?
- [ ] Проведено тестирование отказа зависимостей (chaos engineering, хотя бы ручное отключение одного из сервисов в staging)?

## 8. Наблюдаемость (Observability)

Если систему нельзя «увидеть» в момент проблемы, её нельзя быстро починить. Наблюдаемость строится на трёх столпах: логи, метрики, трейсинг.

Структурированные логи (JSON, а не свободный текст) с единым `correlation ID`/`trace ID`, который проходит через весь путь запроса — от фронтенда до самого глубокого сервиса — критичны для отладки распределённых систем. Без этого поиск причины бага в цепочке из пяти сервисов превращается в археологию.

Метрики (Prometheus и подобные) должны отражать не только технические показатели (CPU, память, latency), но и бизнес-показатели (количество успешных заказов в минуту, конверсия). Алертинг, настроенный только на технические метрики, может пропустить ситуацию, когда сервер «здоров», а бизнес-процесс сломан (например, оплата тихо перестала проходить).

### Чек-лист

- [ ] Логи структурированы (JSON) и содержат единый `trace ID`/`correlation ID` сквозь весь путь запроса?
- [ ] Уровни логирования используются осознанно (DEBUG/INFO/WARN/ERROR), а не всё подряд как INFO?
- [ ] Подключены метрики (Prometheus/аналог) для технических показателей (latency, error rate, throughput)?
- [ ] Настроены метрики для ключевых бизнес-процессов (заказы, платежи, регистрации)?
- [ ] Подключён distributed tracing (OpenTelemetry/Jaeger/аналог) для распределённых запросов?
- [ ] Алертинг настроен не только на инфраструктурные метрики (CPU/RAM), но и на бизнес-аномалии?
- [ ] Дашборды (Grafana/аналог) отражают реальное состояние системы и понятны не только автору?
- [ ] Логи и метрики не содержат чувствительных данных (см. также раздел «Безопасность»)?

## 9. Тестирование и CI/CD

Тесты — это не «для галочки», а инструмент, который позволяет менять систему без страха. Аудит покрытия тестами должен смотреть не на голый процент, а на то, что именно не покрыто — обычно это самые критичные и самые запутанные части системы, которые бояться трогать.

Три уровня тестов дополняют друг друга: unit-тесты проверяют изолированную логику быстро и дёшево, integration-тесты проверяют взаимодействие с реальной БД/внешними сервисами (часто через test containers), e2e-тесты проверяют пользовательский сценарий целиком, но они дорогие и медленные — их должно быть немного, только на критичные сценарии.

Нагрузочное тестирование перед крупными релизами — то, что отличает «работает у меня локально» от «выдержит реальный трафик». Стратегия деплоя (canary, blue-green) и возможность быстрого отката — это страховка на случай, если что-то пошло не так уже в проде.

### Чек-лист

- [ ] Покрытие тестами отслеживается, и понятно, какие именно критичные части не покрыты?
- [ ] Unit-тесты покрывают ключевую бизнес-логику изолированно, без реальной БД/сети?
- [ ] Integration-тесты проверяют реальное взаимодействие с БД/внешними сервисами (например, через testcontainers)?
- [ ] E2E-тесты покрывают только критичные пользовательские сценарии (не всё подряд, иначе тесты станут неподдерживаемыми)?
- [ ] Тесты для Go включают `-race` флаг в CI?
- [ ] CI прогоняет линтеры и статический анализ (golangci-lint, ruff/mypy, eslint, tsc --noEmit) на каждый PR?
- [ ] Перед крупными релизами проводится нагрузочное тестирование (k6, Locust, аналог)?
- [ ] Стратегия деплоя предполагает постепенный выкат (canary/blue-green), а не мгновенную замену всех инстансов?
- [ ] Есть быстрый и проверенный путь отката (rollback) на предыдущую версию?
- [ ] Инфраструктура описана как код (Terraform/Pulumi/аналог), а не настроена руками через консоль?
- [ ] Секреты в CI/CD хранятся в защищённом хранилище (не в переменных окружения репозитория в открытом виде)?

## 10. Масштабируемость

Масштабируемость — это способность системы выдерживать рост нагрузки без переписывания архитектуры с нуля. Ключевой принцип — по возможности делать сервисы stateless (без хранения состояния сессии в памяти конкретного инстанса), чтобы можно было свободно добавлять и убирать инстансы под нагрузкой.

Кэширование (Redis и подобные) ускоряет частые операции, но требует явной и продуманной стратегии инвалидации — «есть кэш, но никто не знает, когда он протухает» — источник трудноуловимых багов с показом устаревших данных.

Асинхронная обработка через очереди сообщений (Kafka, RabbitMQ, NATS) освобождает синхронный путь запроса от тяжёлых операций, которые не обязаны блокировать пользователя (отправка писем, генерация отчётов, обработка изображений).

### Чек-лист

- [ ] Сервисы спроектированы как stateless там, где это возможно (сессии/состояние вынесены в Redis/БД, а не в памяти процесса)?
- [ ] Есть горизонтальное масштабирование (можно добавить инстансы под нагрузкой без изменений в коде)?
- [ ] Кэширование применяется для часто запрашиваемых, редко меняющихся данных?
- [ ] Стратегия инвалидации кэша явная и понятная (TTL, событийная инвалидация), а не «само как-нибудь протухнет»?
- [ ] Тяжёлые/не требующие мгновенного ответа операции вынесены в асинхронную обработку через очередь?
- [ ] Проведено тестирование поведения системы при росте нагрузки в N раз (не только текущий пик, но и прогноз роста)?
- [ ] Узкие места (bottlenecks) идентифицированы заранее (профилирование, нагрузочное тестирование), а не обнаруживаются впервые в проде?

## 11. Поддерживаемость и технический долг

Последний, но не менее важный аспект — насколько легко в проекте разобраться новому человеку и насколько безопасно вносить в него изменения через полгода-год, когда контекст решений уже забылся.

Architecture Decision Records (ADR) — короткие документы, фиксирующие «почему мы выбрали именно так, а не иначе» — экономят огромное количество времени в будущем, когда кто-то (возможно, вы сами) захочет понять, было ли решение осознанным компромиссом или случайностью.

Технический долг стоит вести не в голове и не в разрозненных комментариях `// TODO: fix this`, а в трекере задач с приоритизацией — иначе он никогда не будет оплачен.

### Чек-лист

- [ ] Ключевые архитектурные решения задокументированы в ADR (что выбрали, какие альтернативы рассматривали, почему)?
- [ ] Новый разработчик может развернуть проект локально и разобраться в основных потоках без «устных знаний» старших коллег?
- [ ] README и внутренняя документация актуальны (не описывают то, что было полгода назад)?
- [ ] Технический долг зафиксирован в трекере задач с приоритетами, а не только в комментариях в коде?
- [ ] Зависимости (библиотеки, фреймворки, версии языков) не устарели критично, есть план их обновления?
- [ ] Есть процесс регулярного код-ревью архитектурных изменений (не только точечных PR, но и обзора направления развития проекта)?
- [ ] Обработка ошибок единообразна по всему проекту (не смешаны разные подходы в разных частях)?
- [ ] Нет «сакральных» частей кода, которые боится трогать вся команда из-за отсутствия документации/тестов?

## 12. Быстрая шпаргалка «с чего начать аудит»

Если времени мало, а нужно быстро оценить состояние проекта, начните в этом порядке:

1. **Быстрый архитектурный обзор** — попросите/нарисуйте диаграмму системы, найдите циклические зависимости и дублирование логики (Раздел 1).
2. **Прогоните `-race` для Go и проверьте конкурентные участки записи в БД** — самый частый источник тихой потери данных (Раздел 3).
3. **Посмотрите top-10 самых медленных запросов в Postgres через `pg_stat_statements`** (Раздел 4).
4. **Проверьте секреты в репозитории** (не закоммичены ли пароли/ключи) и результат последнего `npm audit`/`pip-audit`/`govulncheck` (Раздел 5).
5. **Проверьте, протестировано ли восстановление из бэкапа** — на бумаге это есть у всех, реально работает не у всех (Раздел 4).
6. **Оцените покрытие тестами критичных бизнес-сценариев**, а не общий процент (Раздел 9).
7. **Проверьте наличие таймаутов на всех внешних вызовах** (Раздел 7).
8. **Посмотрите, есть ли трейсинг сквозь сервисы**, или дебаг распределённой системы происходит «по логам вручную» (Раздел 8).

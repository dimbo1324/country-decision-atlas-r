# API Overview

> Conventions for the `apps/api` REST API: versioning, authentication, error format, idempotency, the contract. For how a request travels through the backend's layers, see [../architecture/layers-and-boundaries.md](../architecture/layers-and-boundaries.md). Facts checked against code on 2026-07-21 (`apps/api/app/bootstrap/app_factory.py`, 41 routers).

## 1. Baseline facts

- Base path: `/api/v1/*` — the only version today; the next breaking version
  will get `/api/v2/*` alongside it (coexistence, not replacement), the same
  principle used for `LocaleCode`/`methodology_parameters` — old shapes are
  never deleted, new ones are added alongside.
- API docs are generated by FastAPI and served at `/api/docs` (Swagger UI),
  `/api/redoc` (ReDoc), `/api/openapi.json` (raw JSON) — all three reflect the
  real state of the code at request time.
- The contract's source of truth is `contracts/openapi.yaml` (committed to
  git), regenerated by `pnpm contracts:generate` after any schema change; it
  generates `packages/contracts/generated/types.ts`, consumed by `apps/web`.
  Both files are generated only — hand-edits are forbidden.
- System endpoints outside `/api/v1`: `GET /health` (is the process alive),
  `GET /ready` (is the database reachable — through its own small connection
  pool so it never queues behind application traffic), `GET /metrics` (request
  counters, latency, DB pool gauges).

## 2. Error format

A single envelope for every error, from 4xx validation to 5xx:

```json
{
  "error": {
    "code": "feature_disabled",
    "message": "This feature is not enabled yet.",
    "details": null
  }
}
```

Assembled via `app.core.errors.api_error(status_code, code, message, details=None)`
on the backend and parsed via `isApiError()`/`mutationErrorMessage()` on the
frontend (`apps/web/src/shared/api/http.ts`) — code never treats `message` as a
source of truth (it's free to localize or reword), only `code` is. FastAPI's
own validation errors (422) go through the same envelope, not FastAPI's default
`{"detail": [...]}`.

## 3. Authentication and authorization

- **Session** — an httpOnly + `Secure` (in production) + `SameSite=Lax` cookie.
  For API clients with no cookie (mobile integrations, tests, a future B2B
  surface) — an alternate path via the `Authorization` header.
- **CSRF** — a double-submit token is required for `POST/PUT/PATCH/DELETE` when
  the request carries a session cookie (not `Authorization`); exempt:
  `/auth/login`, `/auth/register` (there's no session yet to forge).
- **Passwords** — PBKDF2-SHA256, 260,000 iterations (`app/core/auth.py`).
- **Session rotation** — the session token changes every
  `auth_session_rotation_interval_minutes` with a grace period on the previous
  one; `GET /auth/sessions` lists active sessions/devices (masked IP);
  `revoke-all` requires step-up — re-entering the password.
- **RBAC** — four roles (`user`/`editor`/`moderator`/`owner`) via
  `require_roles(...)`, plus fine-grained capability grants
  (`author.metrics`, `contributor.countries`, `moderator.<scope>`) via
  `require_capability(...)`/`require_capability_or_roles(...)`
  (`app/core/rbac.py`). Deny-by-default: a router with no matching rights
  rejects the request outright rather than returning a trimmed-down response.
- **Feature flags** — an independent, fourth axis (not to be confused with
  rights): `ensure_feature_enabled(connection, key, message)` → 403
  `feature_disabled` when the capability is off for this access tier
  (`public`/`beta`/`internal`/`admin`). Detail:
  [../architecture/rights-and-roles.md](../architecture/rights-and-roles.md) §1, axis 4.

## 4. Response localization

Public read endpoints accept `locale` as a query parameter
(`app/core/locales.py::LocaleQuery`). Content data (country names, source
text, methodology) is stored in one source language and overlaid with a
translation on read (`overlay_localized_fields`, not a separate column per
language) — the response carries a status per translated field:
`source | translated | fallback | missing`. The `LocaleCode` contract today is
only `en`\|`ru`; for the `es` interface locale (the interface itself supports
three languages, but the backend's content data doesn't), the frontend collapses
the request to `en` itself via `toApiLocale()` before calling the API. Detail on
the interface/content split:
[../architecture/overview.md](../architecture/overview.md) §3.7 (the
whole-project audit) and §7.6.

## 5. Idempotency and listings

- **Create idempotency** — some content-create endpoints (example:
  `POST /me/author-metrics`) accept an `Idempotency-Key` header
  (`app/services/idempotency.py::with_idempotency_key`) — a retry with the same
  key returns the same result instead of creating a duplicate. This matters on
  any endpoint where a client-side network timeout retry would otherwise create
  a duplicate row.
- **Listings** — a standard public listing takes `limit`/`offset` (validated
  ranges, typically `limit` ≤ 100); a handful of internal "deep" listings use
  keyset queries in the repository for stability under concurrent writes —
  deep-offset pagination is unreliable once the underlying list changes between
  pages.
- **Outbox, never direct delivery** — any side effect that "tells someone
  something happened" goes through `domain_events` (the outbox) with
  `event_key` idempotency; it is never sent synchronously from inside a request
  handler. Detail:
  [../architecture/layers-and-boundaries.md](../architecture/layers-and-boundaries.md) §3.

## 6. Public vs `/me/*` vs `/admin/*`

Three endpoint classes by data ownership, not a single rights axis:

| Prefix | Who | Example |
|---|---|---|
| `/api/v1/<domain>` | Public reads of published content (writes may still need a role/grant) | `GET /countries/{slug}` |
| `/api/v1/me/<domain>` | The current user's own data (ownership-gated) | `GET /me/trips`, `PUT /me/author-metrics/{id}/values` |
| `/api/v1/admin/<domain>` | Governance/editorial (a role or grant, not just authentication) | `GET /admin/moderation/actions`, `POST /admin/capabilities` |

Contributor and author flows use `/me/*` even when the end result is public
(e.g. `/me/country-proposals/{id}/...`) — wrappers over the editorial services,
scoped to ownership of one's own proposal/metric, rather than calling
`/admin/*` directly. This keeps the behavior of the existing editorial/admin
endpoints unchanged while opening the surface to the community.

## 7. Changing the contract — the required order

1. Edit `apps/api/app/schemas/*.py` (Pydantic v2).
2. `pnpm contracts:generate` — regenerates `contracts/openapi.yaml` and
   `packages/contracts/generated/types.ts`.
3. Commit both generated files together with the schema change and its
   `apps/web` consumers — as one change, not split across PRs.
4. No field, error code, response shape, or pagination behavior changes
   silently: renaming a field, changing a type, changing a parameter's
   requiredness — all are breaking changes, called out explicitly in the
   task's final report (`.ai/universal/06-quality-and-testing.md`).

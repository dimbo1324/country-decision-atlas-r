# Часть 2. Размышление над вашими технологиями (gRPC / Kafka / Go / MongoDB / Telegram)

Вы сформулировали правильное условие: **не заменять существующую бизнес-логику, а строить новую**. Это ключ. Существующее ядро (FastAPI + PostgreSQL, CII, decision, data-quality) трогать нельзя — оно и есть «source of truth». Все ваши технологии должны жить **на отдельном, асинхронном, слабо связанном контуре**.

## 2.1. Главная идея: все пять технологий складываются в один связный вертикальный срез — «Signals & Alerts»

Самое естественное место, где gRPC + Kafka + Go + MongoDB + Telegram нужны **одновременно и обоснованно**, — это **система подписок и уведомлений о правовых изменениях** (Watchlist + Legal Change Alerts, `Phase 9` дорожной карты, уже частично заложена таблицами `users`/`watchlists`).

Архитектура нового контура:

```
[ FastAPI монолит ]  --(publish event)-->  [ Apache Kafka ]  --(consume)-->  [ Go-микросервис "notifier" ]
   (источник истины)                          legal_signal.published              |          |
   при публикации сигнала                      timeline_event.added               |          |
   эмитит доменное событие                     drift.changed                      v          v
        ^                                                                  [ MongoDB ]   [ Telegram Bot API ]
        |                                                             подписки + лог доставки   доставка + identity
        +------------------ gRPC (subscribe / unsubscribe / list) ----------------+
```

Почему это «правильно», а не «технологии ради технологий»:

1. **Telegram** — естественный первый канал и **одновременно провайдер идентичности**. Подписка привязывается к Telegram user id. Это позволяет построить реальную пользовательскую фичу **вообще без веб-auth/RBAC и без веб-фронтенда** — Telegram сам является UI. Идеально ложится на ваше «сейчас не хочу сложный frontend».

2. **Apache Kafka** — шина доменных событий. Монолит при публикации `legal_signal`/`timeline_event` (это уже существующий lifecycle `draft→published`) дополнительно эмитит событие. Kafka — это **интеграционный лог**, который развязывает «производителя контента» (Python) и «доставщика» (Go). Существующий синхронный read-API не затрагивается — Kafka строго аддитивна. Бонус: эта же шина в будущем обслужит ingestion-пайплайн источников.

3. **Golang-микросервис** (ровно **один значимый микросервис**, как вы и хотели) — `signal-notifier`. Задача I/O-bound и concurrency-friendly (потребление из Kafka, fan-out тысячам подписчиков, ретраи Telegram) — это каноничный сценарий для Go с горутинами. Чёткая граница через Kafka/gRPC означает, что Python и Go **не переплетаются**.

4. **gRPC** — синхронный контракт «запрос-ответ» между монолитом (или фронтендом через монолит) и Go-сервисом: управление подписками, выдача статуса доставки. Важно не путать роли: **gRPC = синхронные команды/запросы, Kafka = асинхронные события**. У них разные задачи, и в этой схеме обе оправданы.

5. **MongoDB** — хранилище **гибкого, write-heavy, нереляционного состояния** Go-сервиса: подписки, настройки доставки, dedup-ключи, журнал отправок, статусы Telegram. Документация прямо запрещает раздувать JSONB в Postgres «как замену реляционной модели» — значит, это состояние и должно жить вне Postgres. **Postgres остаётся источником истины; Mongo хранит производное, eventually-consistent состояние доставки.**

## 2.2. Как это согласуется с «modular monolith, без микросервисов»

Документация предостерегает от **преждевременной фрагментации на множество сервисов** и от перевода ядра в event-driven. Предлагаемая схема этого не нарушает:

- ядро (countries / cards / CII / decision / data-quality / translations) **остаётся монолитом и синхронным**;
- наружу выносится **ровно один** ограниченный, по своей природе асинхронный домен (доставка уведомлений);
- Kafka используется не как «внутренняя шина всего», а как **граница интеграции одного нового контура**.

Это академически защитимая позиция: «strangler-периметр» вокруг монолита, а не распил монолита. Я рекомендую зафиксировать это как явный архитектурный инвариант: **бизнес-решения и чтение — всегда монолит; out-of-process — только асинхронная доставка и тяжёлый ingestion**.

## 2.3. Вторичные (необязательные) применения — если захотите развести технологии «по разным частям»

Вы отметили, что их не обязательно концентрировать в одном месте. Альтернативные обоснованные точки:

- **MongoDB** — также естественный дом для **AI-extraction drafts** и **raw-снимков источников** (ingestion) в будущем AI-эпизоде: полуструктурированные, черновые, недоверенные данные не должны попадать в чистый реляционный core до редакторской проверки.
- **Kafka** — будущий **source-ingestion pipeline** (когда дойдёте до автоматического сбора): `source.fetched → ai.extracted → editor.review_required`.
- **gRPC** — внутренний контракт для будущего **AI-summary сервиса** (если AI-слой тоже вынесете из Python-процесса).

# Часть 3. Предлагаемая последовательность эпизодов (в правильном порядке)

Принципы упорядочивания: (1) data-first — сначала продуктовое ядро данных, потом распределённые надстройки; (2) тонкий frontend в каждом эпизоде; (3) ваши технологии входят на самом раннем этапе, где они **реально оправданы**, а не искусственно; (4) каждый эпизод самодостаточен и заканчивается зелёными гейтами.

# Часть 4. Риски и принципы, которые я рекомендую зафиксировать

1. **Инвариант ядра:** бизнес-решения и чтение — всегда монолит + Postgres (source of truth). Out-of-process (Go/Kafka/Mongo) — только асинхронная доставка и тяжёлый ingestion. Не мигрировать реляционный core в Mongo.
2. **Outbox-паттерн** при эмиссии в Kafka, чтобы событие не рассинхронизировалось с транзакцией Postgres (классическая ловушка dual-write).
3. **Mongo — производное состояние**, eventually-consistent; при потере его можно перестроить из Kafka-лога. Не делать его авторитетным.
4. **gRPC ≠ Kafka:** синхронные команды/запросы — gRPC; события/fan-out — Kafka. Не подменять одно другим.
5. **Операционная сложность растёт** (Postgres + Redis + Kafka + Mongo + 2 сервиса). Держать локально через docker-compose, новый контур — отдельной CI-дорожкой, чтобы он **не блокировал** основные гейты монолита.
6. **«Not legal advice»** обязателен и в Telegram-сообщениях (юридически чувствительный контент).
7. **Тонкий frontend** соблюдается естественно: Эпизод 4 (главный по технологиям) вообще не требует веб-UI — его интерфейс это Telegram.
8. **CII-формулу, scenario-веса, decision-математику не трогаем** — persona/drift строятся отдельными слоями весов поверх.

# Часть 5. Рекомендация по первому шагу

Если приоритет — **продуктовая полнота**: начинайте с **Эпизода 1 → 2 → 3**, затем Эпизод 4.

Если приоритет — **освоение вашего стека (gRPC/Kafka/Go/Mongo/Telegram) как можно раньше**: сделайте **Эпизод 1** (минимальная глубина данных, чтобы было о чём уведомлять), а затем сразу **Эпизод 4** — он технически зависит только от наличия публикуемых legal signals, которые уже есть. Это даст вам реальную пользовательскую фичу на всём желаемом стеке, **без единой строки сложного фронтенда**.

Мой совет: **Эпизод 1, затем Эпизод 4** — это лучший баланс между «двигаться к продуктовым целям», «не нарушать data-first» и «начать получать опыт с gRPC/Kafka/Go/MongoDB/Telegram прямо сейчас на по-настоящему уместном домене».

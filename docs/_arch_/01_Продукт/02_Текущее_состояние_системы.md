## 2. Слои и конвенции кода

| Слой | Где | Правило |
|---|---|---|
| Роутеры | `apps/api/app/api/v1/*.py` | Тонкие: парсинг, зависимости (auth/RBAC/connection), вызов сервиса |
| Сервисы | `apps/api/app/services/` | Вся бизнес-логика; крупные домены — пакетами (`migration_board/`, `decision_engine/`, `data_quality/`) с re-export публичной поверхности через `__init__.py` |
| Репозитории | `apps/api/app/repositories/` | Только SQL (закреплено тестами «repository is SQL-only») |
| Схемы | `apps/api/app/schemas/` | Pydantic v2; OpenAPI-контракт → генерация TS-типов (`pnpm contracts:generate`) |
| Ядро | `apps/api/app/core/` | Settings; database-хелперы; `api_error` → единый формат ошибок `{detail:{error:{code,message,details}}}`; CurrentUser; require_roles |

Конвенции (детально — `CLAUDE.md` в корне):

- **Локализация — overlay на чтении**: `overlay_localized_fields(...)` накладывает переводы поверх строк read-модели; клиент получает статусы `source|translated|fallback|missing`. Легаси-колонки `*_ru/*_en` сохраняются.
- **Пакетная декомпозиция**: файлы >800 строк режутся на пакет; кросс-катящие приватные хелперы — в `helpers.py` с квалифицированным доступом (`helpers.fn()`), иначе monkeypatch в тестах молча перестаёт перехватывать вызовы.
- **Feature flags**: `ensure_feature_enabled(connection, key, message)` → 403 `feature_disabled`; статусы enabled/disabled/internal/deprecated, тиры доступа public/beta/internal/admin, правила по тирам.
- **Тесты**: MagicMock-connection, monkeypatch на модуль репозитория, `_assert_error` для кодов ошибок; migration-тесты читают SQL; каждый файл — однострочный docstring о предмете проверки.

### 3.4 Платформа и пользователь

- **Auth/RBAC**: PBKDF2-SHA256 (260k итераций); сессия — httpOnly+Secure(prod)+SameSite=Lax cookie с CSRF double-submit (плюс `Authorization`-заголовок как альтернативный путь для API-клиентов); токен сессии ротируется каждые `auth_session_rotation_interval_minutes` с grace-периодом на предыдущий; видимость сессий/устройств (`GET /auth/sessions` с меткой устройства и маскированным IP) и уведомление о входе с нового устройства (`GET/POST /auth/security-notifications`); `revoke-all` требует step-up повторный ввод пароля; роли user/editor/moderator/owner; промоция в owner — только owner'ом; регистрация за флагом; Telegram-линковка через gRPC к нотификатору.
- **Watchlists** (за флагом), **What changed** (окно/`since` по стране), **Search** (search_documents + GIN, upsert/remove в той же транзакции при publish-переходах контента + отдельный rebuild-скрипт для полного пересчёта/восстановления, локали раздельно), **Analytics** (hash сессий, санитизация metadata, запрещённые ключи), **Feature flags API**, **Cache** (Redis-бэкенд — синглтон на процесс).
- **Retention**: `scripts/cleanup_retention.py` (идемпотентный, dry-run) удаляет `analytics_events`/`ai_interaction_logs` старше `retention.analytics_days`, `domain_events(status='relayed')` старше `retention.domain_events_days`, `auth_sessions` (expired/revoked) старше `retention.sessions_days` — окна заданы параметрами методологии, не хардкодом.

### 3.5 Сообщество

- **Q&A**: pending→published (модерация); консенсус прозрачной формулой (взвешивание + бонус source-backed + детект «спорного» по разбросу).
- **User stories**: структурированные, verification status, маркировка synthetic; рейтинги по 6 осям; репорты ошибок данных.
- **Миграционная доска**: посты с lifecycle и модерацией (approve/reject/hide), видимость public/members_only/private, PII-детектор публичных текстов (email/телефон/@handle/URL), контакт-запросы по взаимному согласию (лимит 20/день), блокировки, репорты (20/день, дедупликация), companion matching с причинами совпадения, публичная проекция без PII.
- **Треды на контактах** (Эпизод 7, расширение доски): 1:1-переписка автосоздаётся при принятии contact request, статусы `open|closed|frozen`, доставка — polling, лимит сообщений/день (параметр `board.max_thread_messages_per_day`), закрытие любой стороной, заморозка при блокировке контрагента, репорт треда переиспользует репорт контакт-запроса; модераторский доступ к переписке — только по `report_id`, совпадающему с contact_request треда, плюс запрет конфликта интересов и аудит на каждое обращение (не только первое); PII-фильтр не применяется к телу сообщений (приватный канал, не публичная проекция).
- **Авторские метрики** (грант `author.metrics`, за флагом `author_metrics_enabled`): определения метрик с lifecycle draft→review→published→archived(+rejected), премодерация через скоуп `moderator.metrics` (публикация требует методологию не короче порога и покрытие ≥ N стран), значения по странам (источник ИЛИ пометка «личный опыт», проверка шкалы), форки с родословной (`forked_from_id`), подписки на метрику/автора + лента, производная репутация (coverage/freshness/sourcing, пересчёт скриптом). Авторский слой read-only overlay на отдельных эндпоинтах (`/countries/{slug}/author-metrics`, `/authors/{user_id}/metrics`) — платформенный CII/decision/compare не импортирует и не читает `services/author_metrics`.
- **Контрибуция стран** (грант `contributor.countries`, за флагом `country_contribution_enabled`, Эпизод 5): предложение страны (`country_proposals`, lifecycle draft→review→published→archived(+rejected)) создаёт скрытую (`is_active=FALSE`) запись `countries`; контрибутор наполняет карточку/источники/evidence/правовые сигналы/таймлайн/CII-значения через ownership-gated `/me/country-proposals/{id}/...` (обёртки над существующими редакционными сервисами, статус всегда `draft`); куратор-editor закрепляется (`curator_user_id`), прогоняет онбординг-гейт (`evaluate_country_onboarding`, переиспользован как гейт контрибуции), сам вводит сценарные скоринги (`country_scores`/`country_score_breakdowns`, 7 критериев с весами и источниками) и публикует — публикация без куратора или с красным гейтом невозможна (инвариант). Атрибуция контрибутора сохраняется публично.

### 3.6 AI-контур (инструментальный)

Провайдерный шов (Fake-by-default); контекст — только published-контент и сохранённые метрики; `/ai/ask` (цитаты, отказ без контекста), `/ai/explain-number`, `/ai/decision-intent`; драфты → needs_review; contradiction candidates → review; interaction logs с ограничениями metadata.

### 3.7 Редакция и операции

Admin CRUD с lifecycle и audit_events; **data quality** — отчёт valid/invalid, 60+ проверок пакетом по доменам; translation jobs + worker foundation; outbox relay (батчи, max attempts, метрики); rate limits.

## 4. Операционный контур

- **CI**: `.github/workflows/quality.yml`.
- **Периодические джобы** (пока запускаются вручную/через `dev_tools_scripts_runner.py`, не подключены к cron/systemd/CronJob — деплой-цель ещё не выбрана):
  - `scripts/outbox_relay.py` — `domain_events` → Kafka, идемпотентно по `event_key`, батчами.

До появления реального деплоя запускать вручную; на боевом окружении — повесить на регулярное расписание.

### 7.2 Аудит-эпизоды упрочнения (hardening)

Отдельная линия аудита кода (безопасность, ресурсы, конкурентность, БД, наблюдаемость, отказоустойчивость, тестирование/CI) закрыта; её план и полный отчёт (`audit_result.txt`, `09_План_устранения_аудита.md`) выведены из документации после завершения. Затронувшие схему изменения — миграции **052–055**:

| Миграция | Что добавила |
|---|---|
| 052 `session_security_hardening` | Ротация токена сессии, видимость устройств, уведомление о входе с нового устройства (`auth_sessions`) |
| 053 `domain_events_in_flight` | Статус in-flight у `domain_events`: relay освобождает блокировки строк до блокирующего publish в Kafka |
| 054 `retention_config` | Окна retention как параметры методологии (аналитика, `domain_events`, сессии) — не хардкод |
| 055 `recompute_requested_event` | Тип события `recompute_requested` для асинхронного admin-рекомпьюта через outbox |

Изменения без миграций: fail-fast на дефолтных prod-секретах и prod-overlay `docker-compose`, конфигурируемый пул БД; наблюдаемость — request-id трейсинг, JSON-логи, эндпоинт `/metrics` (счётчики запросов, латентность, гейджи пула БД); keyset-пагинация для глубоких публичных списков; идемпотентность на content-create эндпоинтах; единый источник PII-паттернов и lifecycle-статусов; таймауты fetch в API-клиенте `apps/web`; интеграционные тесты репозиториев и `-race` для Go в CI.

### 7.3 Пайплайн синтетических данных (`scripts/synthetic_data.py` + `utils/synthetic_data`)

Автономный dev-инструмент, генерирующий полностью **вымышленный** мир (страны, пользователи, метрики, документы) для разработки и тестов без реальной БД и без реальных стран. Детерминирован по seed; 15 Unicode-локалей; профили и архетипы; рендер документов (PDF/DOCX/XLSX/TXT), SQL-фикстуры с изолированной загрузкой по `dataset_id`, HTML-дашборд, sha256-сайдкары пакетов, diff между датасетами, mock-HTTP-сервер, имитирующий эндпоинты `apps/api`. Точка входа — тонкий shim `scripts/synthetic_data.py`; вся реализация и входные конфиги — в `utils/synthetic_data/` (перенесено из `scripts/synthetic_data`/`docs/synthetic_data`, см. `../SYNTHETIC_DATA_PLAN.md` Этап 0). Этап 1 добавил Synthetic Web Environment (`utils/synthetic_data/web/`): браузируемый детерминированный веб из вымышленных сайтов (один сайт на страну, страницы источников/статей/уведомлений, межсайтовые ссылки, намеренно битые страницы — 404/500/редирект/дубликат/пустая/огромная/битая кодировка), раздаётся локальным FastAPI-сервером (`generate --formats web`, `render-web`, `serve` в CLI). Полная документация — `utils/synthetic_data/README.md`. Собственный набор тестов (341) подключён в quality gate отдельным шагом pytest. Переключение E2E/смоуков продукта на этот набор — отдельная незакрытая задача (Эпизод 8; см. Р-11 в `../08_Открытые_вопросы.md`).

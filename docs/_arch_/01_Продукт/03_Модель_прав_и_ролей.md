# Модель прав и ролей

> Опорный документ по разделению пользователей, прав и модерации. Написан с учётом того, что владелец проектирует такую систему впервые: сначала — понятийный ликбез и типовые ошибки, затем — что уже реализовано, затем — целевая модель и путь к ней. Требование владельца: платформа должна быть минимально зависима от одного человека или узкого круга — этому посвящён раздел 6.

---

## 1. Ликбез: четыре разных механизма, которые нельзя смешивать

Почти все ошибки проектирования прав происходят из смешения четырёх независимых осей. В проекте все четыре уже существуют — важно понимать границы каждой.

### Ось 1. Роль — «кем пользователь является»

Долгосрочная характеристика аккаунта: обычный пользователь, редактор, модератор, владелец. Ролей должно быть **мало** (3–5), они меняются редко и только явным административным действием.

### Ось 2. Право/способность — «что пользователю можно делать»

Конкретное разрешённое действие: «публиковать авторские метрики», «модерировать доску», «предлагать страны». Прав может быть много, и они не обязаны совпадать с ролями: обычный пользователь может получить способность «автор метрик», не становясь редактором.

### Ось 3. Видимость — «кому виден объект»

Свойство не пользователя, а **объекта**: пост доски бывает public / members_only / private; план переезда — private / по ссылке. Права отвечают «можно ли действовать», видимость — «кто увидит результат».

### Ось 4. Feature flag — «включена ли возможность вообще»

Рубильник всей функциональности для этапов раскатки (public/beta/internal/admin-тиры). **Флаг — не право**: флаг отвечает «работает ли механизм на платформе», право — «допущен ли к нему данный пользователь». Типовая ошибка — раздавать доступ флагами; тогда невозможно ни точечно выдать, ни точечно отозвать, ни проаудировать.

### Базовые принципы (закрепляются как инварианты)

1. **Deny by default** — всё, что явно не разрешено, запрещено.
2. **Least privilege** — у каждой роли/гранта минимум прав, достаточный для функции.
3. **Проверка только на сервере** — UI может прятать кнопки, но правда всегда в API (так уже сделано: `require_roles` в зависимостях роутеров).
4. **Каждое привилегированное действие аудируется** — уже сделано для редакции и доски (audit_events), распространяется на всё новое.
5. **Права — это данные, а не код**: выдача/отзыв не должны требовать релиза.

## 2. Что уже реализовано (фактическая база)

| Механизм | Реализация | Оценка |
|---|---|---|
| Роли | `users.role`: `user` / `editor` / `moderator` / `owner`; зависимости `require_roles`, `require_editor`, `require_moderator` | Достаточно; расширять состав ролей не нужно |
| Гранты способностей | `user_capabilities` (Эпизод 3); `has_capability`/`require_capability`/`require_capability_or_roles`; очереди доски и Q&A переведены со `require_moderator` на скоуп-способности без изменения поведения; выдача/отзыв — только владелец, с audit-записью | Реализовано согласно §3 |
| Правила назначения | Промоция/демоция в `owner` — только владельцем; admin-роутер пользователей | База governance есть |
| Сессии | Токены с TTL, revoke-all, PBKDF2-SHA256 | Достаточно |
| Видимость | Доска: public/members_only/private + публичные проекции без PII | Образец для новых доменов |
| Feature flags | Статусы + тиры + правила; `ensure_feature_enabled` | Достаточно; использовать только как раскатку |
| Аудит | audit_events на редакторских действиях, действиях доски, грантах способностей; `GET /admin/moderation/actions` — лента с фильтрами и эвристикой всплеска reject'ов (Эпизод 3) | Расширять на новые привилегированные действия по мере появления |
| Конфликт интересов | Модератор не обрабатывает свои посты/репорты, где является стороной (доска, Эпизод 3); проверяется программно | Реализовано для доски; распространить на новые модераторские домены по мере появления |
| Идентичность | Telegram-линковка (gRPC), email-аккаунты | Достаточно |

Текущее распределение (упрощённо): `user` — личные данные и участие в сообществе; `editor` — контент-ядро и переводы; `moderator` — очереди модерации доски/сообщества; `owner` — всё, включая управление ролями.

Владелец отметил: сейчас он выполняет все роли сразу, и это было сделано намеренно — для отладки разделения прав. Модель ниже проектируется так, чтобы из этого состояния можно было выйти без перестройки.

## 3. Целевая модель: роли + гранты способностей

### 3.1 Проблема, которую предотвращаем: взрыв ролей

Новая линия приносит категории: автор метрик, контрибутор стран, модератор доски, модератор метрик… Если каждую делать ролью, получим комбинаторный взрыв («автор+контрибутор», «модератор доски, но не Q&A») и неуправляемую матрицу. Индустриальное решение — роли остаются базовыми, а точечные разрешения выдаются **грантами способностей**.

### 3.2 Гранты способностей (capability grants)

```sql
CREATE TABLE user_capabilities (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    capability TEXT NOT NULL,          -- пространство имён: 'author.metrics',
                                       -- 'contributor.countries',
                                       -- 'moderator.board', 'moderator.metrics',
                                       -- 'moderator.community'
    granted_by UUID NOT NULL REFERENCES users(id),
    granted_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    revoked_at TIMESTAMPTZ,            -- отзыв — не удаление строки (история сохраняется)
    note TEXT,
    CONSTRAINT uq_user_capability UNIQUE (user_id, capability)
);
```

Правила:

- **Роль разворачивается в набор способностей по умолчанию**: `moderator` автоматически обладает всеми `moderator.*`, `editor` — редакторскими, `owner` — всеми. Проверка: `has_capability(user, cap) = роль покрывает cap ИЛИ есть активный грант`.
- **Гранты расширяют обычного пользователя точечно**: `user` + грант `author.metrics` = автор, не редактор и не модератор.
- Выдача и отзыв — только уровнем governance (владелец; позже — по регламенту), каждая операция — в audit_events.
- Проверка в коде — по образцу существующего RBAC: зависимость `require_capability("author.metrics")` рядом с `require_roles`.

### 3.3 Скоупы модерации

Способности модератора именуются по доменам (`moderator.board`, `moderator.metrics`, `moderator.community`). Пока модераторов один-два — им выдаются все скоупы; модель при этом уже готова к специализации («модерирует метрики, но не переписку доски») без единого изменения схемы.

## 4. Матрица прав (целевая)

Обозначения: A — аноним, U — user, +AM — грант author.metrics, +CC — грант contributor.countries, M — модератор (в своём скоупе), E — editor, O — owner. «Свой» = объект, которым пользователь владеет.

| Действие | A | U | +AM | +CC | M | E | O |
|---|---|---|---|---|---|---|---|
| Читать published-контент ядра | ✔ | ✔ | ✔ | ✔ | ✔ | ✔ | ✔ |
| Decision run / сравнение / паспорта | ✔ | ✔ | ✔ | ✔ | ✔ | ✔ | ✔ |
| Профили весов, watchlist, планы переезда | — | свои | свои | свои | свои | свои | свои |
| Q&A: вопрос/ответ; репорты; рейтинги историй | — | ✔ | ✔ | ✔ | ✔ | ✔ | ✔ |
| Доска: пост, контакт-запрос, блокировки | — | ✔ | ✔ | ✔ | ✔ | ✔ | ✔ |
| Авторские метрики: создать/вести/подать на публикацию | — | — | свои | — | — | — | ✔ |
| Авторские метрики: approve/reject/hide | — | — | — | — | ✔(metrics) | — | ✔ |
| Предложить страну; вести данные предложенной страны | — | — | — | свои | — | — | ✔ |
| Страна: перевод в review/published | — | — | — | — | — | ✔ | ✔ |
| Контент-ядро: CRUD источников/сигналов/карточек | — | — | — | — | — | ✔ | ✔ |
| Очереди модерации доски/Q&A; скрытие; резолюция репортов | — | — | — | — | ✔(скоуп) | — | ✔ |
| Просмотр приватного контекста по репорту | — | — | — | — | ✔(только предмет репорта) | — | ✔ |
| Платформенная методология: новая версия параметров | — | — | — | — | — | — | ✔ |
| Feature flags, выдача/отзыв грантов, роли | — | — | — | — | — | — | ✔ |
| Удаление чужого аккаунта | — | — | — | — | — | — | ✔ |

Строка «просмотр приватного по репорту» — принципиальная: модератор видит приватные данные (например, тред переписки) **только в контексте поданного репорта и только предмет репорта**; это фиксируется в политике приватности и в аудите доступа.

## 5. Институт модераторов

Ответ на запрос владельца: «специальные люди с определёнными правами, которые занимаются модерацией».

### 5.1 Откуда берутся

1. **Фаза 0 (сейчас)** — владелец совмещает роли; модель уже разделена, поэтому совмещение — временная конфигурация, а не архитектура.
2. **Фаза 1** — приглашение владельцем 1–3 активных участников с высокой **производной** репутацией (стаж, вклад в Q&A/доску, отсутствие подтверждённых репортов). Выдача грантов `moderator.<скоуп>` + обязательное принятие регламента модератора.
3. **Фаза 2** — заявки на модерацию с формальными порогами допуска (репутация, стаж) и решением уровня governance.

### 5.2 Что модератор может и чего не может

Может (в своём скоупе): работать с очередями (review метрик, репорты, pending Q&A); approve/reject/hide **с обязательной причиной**; замораживать треды; временно ограничивать пользователя (rate-limit-санкция).

Не может никогда: менять роли и гранты; читать приватные данные вне предмета репорта; редактировать контент-ядро; менять платформенную методологию; удалять аккаунты; видеть платёжные данные (когда появятся).

### 5.3 Подотчётность

- Каждое действие модератора — в audit_events (уже готовый механизм) с причиной.
- Страница governance «действия модераторов» для владельца: лента, фильтры, аномалии (всплеск reject'ов, работа по «своим» объектам).
- Конфликт интересов: модератор не обрабатывает объекты, где он сторона (автор, участник треда) — проверяется программно.
- Отзыв гранта — одно действие; санкции обратимы, необратимое (удаление аккаунта) — только владелец.

## 6. Пирамида автономии: «меньше зависимости от людей»

Прямой ответ на требование владельца сделать платформу автономнее. Каждый уровень разгружает следующий; человек — последний рубеж, а не первый фильтр.

```
Уровень 3 — ВЛАДЕЛЕЦ (governance)          назначения, версии методологии,
     ▲   только то, что нельзя делегировать    необратимые операции
Уровень 2 — МОДЕРАТОРЫ (эскалации)         очереди публикации, спорные репорты,
     ▲   только то, что не решилось ниже       санкции
Уровень 1 — СООБЩЕСТВО (самомодерация)     репорты, консенсус-скоринг,
     ▲   сигналы и пороги                      производная репутация,
     │                                          авто-скрытие по N подтверждённых
     │                                          репортов (с пост-ревью уровня 2)
Уровень 0 — АВТОМАТИКА (правила)           валидация схем, PII-скан, rate limits,
         работает всегда, до человека         data-quality гейты публикации,
                                               дедупликация репортов, k-анонимность
```

Практические следствия:

- Ни одна новая поверхность не запускается без уровня 0 (это уже стандарт: PII-скан, лимиты и dq-проверки существуют как переиспользуемые механизмы).
- Пороги уровня 1 (сколько подтверждённых репортов скрывает объект до ревью) — параметры методологии (Эпизод 1), не константы.
- **Репутация — не право.** Производная репутация (по образцу Trust Score: вклад, свежесть, подтверждённость, жалобы) информирует — бейджи, вес в консенсусе, допуск к заявке в модераторы, — но никогда не выдаёт способности автоматически. Выдача гранта — всегда явное действие governance. Это единственная надёжная защита от накрутки прав.
- Метрика здоровья: доля объектов, прошедших публикацию/резолюцию без участия человека, при неубывающем качестве (контролируется data-quality отчётом).

## 7. Владение данными

| Категория | Владелец данных | Права |
|---|---|---|
| Личное (планы, профили весов, подписки, watchlist) | Пользователь | Полный контроль: экспорт, удаление; private-first по умолчанию |
| Авторское (метрики, шаблоны маршрутов) | Автор | Ведение и архивация — автор; видимость published-версий регулируется lifecycle; при удалении аккаунта — архивация с обезличенной атрибуцией (см. открытые вопросы) |
| Контрибуции в ядро (данные стран) | Платформа (кураторство) | Атрибуция контрибутора сохраняется; после publish правки — через редакционный процесс |
| Ядро (источники, сигналы, методология платформы) | Платформа | Редакция (editor) + governance (owner) |

## 8. Путь реализации

Модель вводится одним эпизодом («Права и роли v2», Эпизод 3 плана) **без ломки существующего**:

1. Таблица `user_capabilities` + зависимость `require_capability` (по образцу `require_roles`); роли продолжают работать как пучки способностей — ни один существующий эндпоинт не меняет поведения.
2. Admin-поверхности governance: выдача/отзыв грантов, лента действий модераторов.
3. Аудит-обвязка: причина обязательна для всех moderate-действий; проверка конфликта интересов.
4. Автосанкции уровня 1: параметризованный порог авто-скрытия по подтверждённым репортам.
5. Дальше гранты используются эпизодами 4 (`author.metrics`, `moderator.metrics`) и 5 (`contributor.countries`).

Тестовая дисциплина: матрица раздела 4 переводится в параметризованные тесты (действие × роль/грант → ожидаемый статус), инвариант «deny by default» — отдельным тестом на каждый новый роутер.

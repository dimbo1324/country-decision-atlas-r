# Рабочий стандарт проекта

Единый рабочий регламент Country Decision Atlas. Применяется к каждой задаче разработки. Это регламент, а не кодовый блок: он фиксирует, как ведётся работа — порядок веток, проверок, слоёв реализации, запретов и формата отчёта, — чтобы любой разработчик или ИИ-помощник работал предсказуемо и одинаково.

Обновлён 2026-07-04 при переустановке базовой линии проекта: ссылки на прежние эпизоды удалены (актуальный план — `02_План/01_План_реализации.md`), архитектурные инварианты вынесены в отдельный реестр (`02_План/02_Реестр_инвариантов.md`), состав quality gate приведён к текущему инструментарию.

---

## 1. Рабочий процесс и дисциплина Git

Каждая задача начинается только от актуального `main`:

```
git checkout main
git pull --ff-only origin main
git checkout -b <branch>
```

Обязательные правила:

- Никогда не работать напрямую в `main` (исключение — документационные задачи по прямому распоряжению владельца).
- Имя ветки соответствует задаче: `feat/…`, `fix/…`, `chore/…`, `refactor/…`, `test/…`, `docs/…`.
- Коммит на каждую логически завершённую под-задачу; сообщение объясняет «зачем», а не только «что».
- Точечные проверки после крупных этапов; полный quality gate — **перед** merge.
- Merge только fast-forward (`git merge --ff-only`), только после зелёного gate.
- Push в `origin/main` — только с явного подтверждения владельца в рамках задачи.
- Финальное состояние — на `main`; отработавшая ветка удаляется.

История задач ведётся git-историей и финальными отчётами (раздел 6); отдельный журнальный файл не используется.

## 2. Единый quality gate

Основной способ запуска — оркестратор в корне репозитория:

```
python dev_tools_scripts_runner.py --profile full     # полный прогон (Docker, E2E)
python dev_tools_scripts_runner.py --profile quick    # быстрый локальный срез
python dev_tools_scripts_runner.py --doctor           # read-only диагностика окружения
```

Эквивалентный ручной набор (его же выполняет CI):

```
python -m pytest
python -m mypy apps packages scripts tests
python -m ruff check apps packages scripts tests
python -m ruff format --check apps packages scripts tests
python -m sqlfluff lint database --dialect postgres
pnpm quality
pnpm web:mvp:check
python -m pre_commit run --all-files
```

Используется именно `ruff format --check`: перед merge нужно доказать, что форматирование уже чистое. Если `pre-commit` что-то изменил — изменения проверяются и коммитятся. Для `pnpm web:mvp:check` нужен поднятый стек (`docker compose up -d api`) с применёнными миграциями.

## 3. Порядок реализации слоёв

Код пишется строго сверху вниз. Проект backend/data-first: сначала данные и контракт, потом UI.

1. `database/migrations/NNN_*.sql` (следующий номер; заголовочный комментарий о назначении)
2. repository — только SQL/доступ к данным
3. service — бизнес-логика (крупный домен — сразу пакетом, см. `CLAUDE.md`)
4. schema — контракты Pydantic v2
5. api/v1 router (тонкий) + `main.py` include_router при новом роутере
6. `contracts/openapi.yaml` → `pnpm contracts:generate` → проверка сгенерированного TypeScript
7. тонкий frontend
8. backend-тесты (unit, API, migration, data-quality)
9. E2E-тесты

Дополнительно для каждого нового домена: feature-флаг (выключен до acceptance), data-quality модуль `_append_<domain>_checks`, события — только через outbox. Полный чек-лист нового домена — в плане реализации, часть III.

## 4. Жёсткие запреты

Действуют всегда:

- без комментариев в коде, docstring и inline-комментариев (исключения: однострочный docstring тестового файла о том, что он проверяет; комментарий-заголовок миграции; случаи, когда «почему» невыводимо из кода);
- без новых README / `.md` / `.txt` в рабочем коде проекта;
- без изменений в `docs/` в рамках продуктовых задач — правки документации только выделенными документационными задачами по распоряжению владельца;
- не трогать математику CII, веса сценариев, scoring decision engine (см. реестр инвариантов);
- не ослаблять data-quality и не маскировать проблемы фейковыми данными;
- миграции — только последовательные, идемпотентные, чистые по sqlfluff; применённые файлы миграций не переименовывать и не редактировать (checksum-трекинг);
- никогда не хранить проездные документы, брони и сканы документов пользователей.

## 6. Формат финального отчёта

После каждой задачи исполнитель даёт отчёт в едином формате:

1. Branch workflow — стартовая ветка, feature-ветка, финальная ветка, был ли push.
2. Implemented — что сделано.
3. Database — изменения миграций.
4. Backend — затронутые слои.
5. API / Contracts — изменения OpenAPI и сгенерированного TS.
6. Frontend — затронутые элементы.
7. Tests — добавленные тесты.
8. Quality gates — результаты всех проверок.
9. Data-quality impact — влияние на качество данных.
10. Not changed — что намеренно не трогали.
11. Risks / notes — риски и заметки.
12. Final git state — финальный статус.

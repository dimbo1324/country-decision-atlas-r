# Открытые вопросы к владельцу

> Вопросы, на которые нужны решения владельца. Ответы будут внесены в документацию следующей документационной задачей. Формат: контекст → вопрос → рекомендация архитектора (если есть) → когда решение становится блокером.

## 1. Судьба тестовых стран (Russia, Uruguay, Argentina)

**Контекст.** Три текущие страны — тестовый набор для ручной проверки; при этом их данные реальны по структуре (источники, сигналы, CII) и используются всеми тестами и сидом.

**Вопрос.** Для каждой из трёх: довести до эталонного качества / пересобрать с нуля через новый пайплайн контрибуции / удалить? Нужны ли они как постоянная «демо-песочница» (флаг `is_demo`)?

**Рекомендация.** Не удалять до Эпизода 5: на них обкатывается onboarding-гейт. В Эпизоде 5 прогнать через гейт и решать по фактическому качеству каждой.

**Блокер для:** Эпизода 5.

## 2. Премодерация авторских методологий

**Контекст.** Владелец описал желаемый институт модераторов, но не выбрал режим модерации авторского контента напрямую.

**Вопрос.** Подтверждается ли зафиксированный в плане режим: премодерация методологий (до публикации) на этапе invite-only авторов, с переходом к постмодерации по мере зрелости уровней 0–1 пирамиды автономии?

**Рекомендация.** Да — премодерация до первых ~50 авторов; критерий перехода: доля отклонений в премодерации стабильно < 10%.

**Блокер для:** Эпизода 4 (заложен режим по рекомендации; при несогласии — правка до старта эпизода).

## 3. Монетизация: модель донатов и юрисдикция

**Контекст.** Решение владельца: донаты в будущем, конкретики нет; сейчас закладывается только фундамент (авторство, подписки) и предохранители (реестр инвариантов §6).

**Предложение архитектора (для обсуждения).** Двухуровневая модель: (а) «поддержать платформу» — добровольные донаты без привилегий; (б) «поблагодарить автора» — донат автору метрики/шаблона с комиссией платформы (например 10–15%). Никаких paywall'ов на данные ядра: доверие — продукт, платность данных его разрушит. Возможное развитие позже — необязательная платная подписка на «удобства» (расширенные лимиты планировщика, экспорт-форматы), но не на данные.

**Вопросы.** (1) Какая юрисдикция оперирования планируется? Это определяет доступных платёжных провайдеров (Stripe/Paddle недоступны для РФ; ЮKassa/CloudPayments — для РФ; криптоплатежи — отдельные риски) — без ответа проектирование платёжного контура невозможно. (2) Приемлема ли модель (а)+(б)? (3) Целевые сроки (влияет только на приоритет, не на текущие эпизоды).

**Блокер для:** отдельного будущего эпизода монетизации; текущей линии 1–7 не блокирует.

## 4. Лицензия пользовательских контрибуций

**Контекст.** С Эпизодов 4–5 пользователи создают контент (методологии, значения, данные стран), который платформа отображает, агрегирует и хранит.

**Вопрос.** Под какой лицензией/соглашением пользователь передаёт контрибуции? Варианты: (а) неисключительная бессрочная лицензия платформе (стандарт для UGC-платформ, минимально достаточно); (б) CC BY-SA для авторских методологий (открытость как ценность, усложняет будущую монетизацию авторов); (в) комбинированно: (а) для данных стран, выбор автора (а|б) для методологий.

**Рекомендация.** Вариант (в).

**Блокер для:** публичного запуска Эпизода 4 (текст пользовательского соглашения); разработку не блокирует.

## 7. Хостинг и продакшен-развёртывание

**Контекст.** Вся инфраструктура сейчас локальная (docker-compose). Планировщик (Эпизод 2) впервые требует always-on контура: напоминания должны отправляться, когда пользователь офлайн.

**Вопрос.** Какая целевая площадка развёртывания (VPS/облако, регион, бюджет)? Достаточно ли одного хоста с docker-compose на старте (рекомендация: да)?

**Блокер для:** публичного запуска Эпизода 2 (разработку не блокирует — fake-режим доставки полноценно тестируется локально).

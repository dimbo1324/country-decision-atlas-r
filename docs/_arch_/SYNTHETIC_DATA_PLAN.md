# Synthetic Data Ecosystem — генеральный план

Дата: 2026-07-17. Статус: утверждённый план, реализация не начата.
Владелец решения: owner. Исполнение: по одному этапу за задачу, каждый
этап — отдельная ветка + `task-checklist.md` + полный quality gate,
по правилам `.ai/universal/01-workflow.md` и `02-task-checklist.md`.

---

## 1. Цель

Полностью уйти от данных реальных стран (Россия, Аргентина, Уругвай в
`database/fixtures/demo_countries/`) к вымышленным синтетическим данным,
на которых приложение живёт целиком: страны, метрики, источники,
документы, пользователи, статьи, комментарии — и, главное, **синтетическая
веб-среда** (Synthetic Web Environment): локальные веб-страницы, по которым
можно переходить, с которых можно скачивать синтетические файлы и на
которые ссылаются источники (`sources`) внутри приложения.

Зачем:

- воспроизводимость: любой баг и любая гипотеза проверяются на
  детерминированном, пересоздаваемом мире;
- нулевые претензии на реальные факты о реальных странах;
- полный контроль над тест-экосистемой: «интернет», который никогда не
  исчезнет и не изменится под ногами;
- один скрипт (одна команда) — и весь мир пересобран.

Анти-цель (сформулирована владельцем явно): **не закопаться**. Модуль
должен помогать, а не превратиться в отдельный вечный проект. Поэтому
план опирается на уже построенное и режет всё лишнее (раздел 7).

## 2. Что уже есть — и что мы переиспользуем

`scripts/synthetic_data/` — зрелый пайплайн (~117 файлов, 302 теста).
Канонический референс — `scripts/synthetic_data/README.md`. Ключевое:

| Уже работает | Где |
| --- | --- |
| Канонический вымышленный мир (страны, метрики, пользователи, авторы, статьи, комментарии, источники, legal signals, сценарии) | `core/world_models.py`, `core/world_generator.py` |
| «Метаданные-ориентиры»: архетипы стран с диапазонами 8 метрик + профили-смещения (`balanced`/`crisis`/`optimistic`/…) | `docs/synthetic_data/input_data/world_config.json`, `core/world_generator.py` |
| Детерминизм: `derived_seed = hash(master_seed, entity, id, purpose)` — PDF, SQL-строка и JSON одной страны никогда не противоречат друг другу | `core/seed.py` |
| Безопасность: deny-list из 195 реальных стран, запрет сети (статический тест), маркер `SYNTHETIC TEST DATA - NOT REAL` в каждом артефакте, production-guard для SQL | `core/world_validation.py`, `tests/test_no_network_imports.py`, `core/sql_loader.py` |
| Документы: PDF/DOCX (copyable / non-copyable «скан» / mixed), XLSX, TXT, JSON, ZIP; 15 локалей, RTL/CJK, встроенные шрифты | `core/document_rendering/`, `assets/fonts/` |
| SQL-фикстуры в реальную БД (`countries`, `country_profiles`, `sources`, `legal_signals`), идемпотентные, с защитой демо-стран | `core/sql_fixture.py`, `core/sql_loader.py` |
| Mock HTTP API — фронтенд работает на синтетике без БД | `mock_server/` |
| CLI: `validate / plan / generate / render / package / list / prune / diff / translate / cii-preview / load-sql / cleanup-sql` | `cli.py` |

**Главный принцип плана: не переписывать это, а (а) переложить на
правильное место, (б) достроить единственный крупный недостающий слой —
веб-среду, (в) довести до замены реальных фикстур.**

## 3. Принципы и гардрейлы (наследуются, не пересматриваются)

1. Никаких реальных стран, людей, документов; deny-list обязателен.
2. Никакой сети и никаких LLM в генераторе (offline-by-default; LLM-seam —
   только отдельным решением владельца в интеграционном транше).
3. Каждый артефакт маркирован как синтетический.
4. Детерминизм: один `seed + profile + config` = один и тот же мир.
   «Свежие данные при каждом запуске» = случайный seed по умолчанию,
   который печатается и всегда воспроизводим.
5. Миграции (`database/migrations/`) неприкосновенны — синтетика входит в
   БД только идемпотентными фикстурами после `apply_migrations.py`.
6. Сгенерированный выход не попадает в git; в git — только код, входные
   конфиги и маленькие тестовые снапшоты.
7. Тяжёлая генерация никогда не включается в дефолтный quality gate —
   только явным флагом/командой.

## 4. Целевая архитектура

### 4.1. Размещение в репозитории (после всех этапов)

```text
scripts/synthetic_data.py            # точка входа, <=10 строк (как dev_tools_scripts_runner.py)
utils/synthetic_data/                # ВСЯ реализация
  __init__.py
  cli.py                             # команды (перенос scripts/synthetic_data/cli.py)
  core/                              # мир, сиды, валидация, рендеры документов (перенос)
  generators/                        # legacy-генераторы форматов (перенос)
  mock_server/                       # mock API (перенос)
  archive/                           # zip (перенос)
  assets/fonts/                      # шрифты (перенос)
  web/                               # НОВОЕ: Synthetic Web Environment (этап 1)
  input_data/                        # входные JSON-конфиги (перенос из docs/synthetic_data/input_data)
    world_config.json                # архетипы, профили, пулы имён, deny-list, блоки документов
    locale_corpus.json               # тексты 15 локалей
    web_config.json                  # НОВОЕ: архетипы сайтов, плотность ссылок, аномалии
    data.json                        # пулы слов/заголовков legacy-генераторов
  tests/                             # перенос scripts/synthetic_data/tests
.synthetic_data/                     # выход генерации (gitignored), бывший docs/synthetic_data/output_data
docs/synthetic_data/                 # УДАЛЯЕТСЯ в конце этапа 0
```

### 4.2. Поток данных

```text
input_data/*.json  +  seed  +  profile  +  scale
        |
        v
  WorldGenerator ----> SyntheticWorld (единственный источник правды)
        |                    |
        v                    | (валидация: deny-list, ссылочная целостность)
  world_validation.py <------+
        |
        +---------+-----------+-----------+------------------+-----------------+
        |         |           |           |                  |                 |
        v         v           v           v                  v                 v
   canonical    SQL-       документы   manifest/ZIP     Synthetic Web     mock API
     JSON      фикстура   PDF/DOCX/...  + отчёты        (сайты, HTML,     (FastAPI)
                                                         граф ссылок,
                                                         скачивание файлов)
```

Веб-слой — **ещё один рендерер поверх того же мира**, как JSON/SQL/PDF.
Он не выдумывает собственных фактов: страницы сайтов рассказывают о тех
же странах, событиях и источниках, что лежат в `SyntheticWorld`, а
`SyntheticSource.url` начинает указывать на реально открывающуюся
локальную страницу.

### 4.3. Запуск (после всех этапов)

```powershell
# напрямую
python scripts/synthetic_data.py generate --seed 42017
python scripts/synthetic_data.py serve --dataset <id> --port 8080

# через оркестратор dev-tools (каталог utils/dev_tools_scripts_runner/config/scripts.json)
python dev_tools_scripts_runner.py synthetic-data generate --scale small
```

## 5. Входные данные: JSON-«БД» с метаданными

Требование владельца «несколько JSON со сложной структурой и метаданными
как ориентирами для алгоритмов» уже реализовано моделью
**archetypes + profiles + recipes** и расширяется, а не изобретается заново:

| Файл | Роль | Метаданные-ориентиры |
| --- | --- | --- |
| `input_data/world_config.json` | «Схема мира»: пулы имён, deny-list, архетипы, профили, блоки/рецепты документов | диапазоны `[min,max]` по 8 метрикам на архетип; профиль = управляемое смещение; ни одна страна не «лучшая во всём» |
| `input_data/locale_corpus.json` | Тексты всех 15 локалей | смысл факта не меняется между локалями — меняется только текст |
| `input_data/web_config.json` (новый, этап 1) | «Схема интернета» | архетипы сайтов (gov-портал/новости/блог/вики/корпоративный), число страниц, плотность внутренних и межсайтовых ссылок, доли аномалий (404/500/циклы/редиректы/дубли/пустые) |
| `input_data/data.json` | Пулы слов legacy-генераторов | как есть |

Правило на будущее: новые «ориентиры» добавляются как декларативные поля
этих JSON, а не как ветвления в Python-коде.

### Этап 0 — переезд в `utils/synthetic_data` (механический, без новых фич)

Цель: та же архитектура, что у `dev_tools_scripts_runner`: тонкая точка
входа + пакет в `utils/`; `docs/synthetic_data/` перестаёт существовать.

Шаги:

1. `git mv scripts/synthetic_data utils/synthetic_data` (включая `tests/`,
   `assets/`, `README.md`).
2. Массовое переименование импортов `scripts.synthetic_data.*` →
   `utils.synthetic_data.*` (весь пакет, тесты, `conftest.py`).
3. Перенос входных данных: `docs/synthetic_data/input_data/*.json` →
   `utils/synthetic_data/input_data/`; правка `core/paths.py`
   (пути к input и дефолтному output).
4. Новый дефолтный выход: `.synthetic_data/` в корне (gitignored).
   В `.gitignore`: удалить строку `docs/synthetic_data/output_data/`,
   добавить `.synthetic_data/`.
5. Создать точку входа `scripts/synthetic_data.py` (<=10 строк):
   `from utils.synthetic_data.cli import main; raise SystemExit(main(sys.argv[1:]))`.
6. Обновить всю обвязку, которая знает старый путь:
   - `scripts/dev_tools/full_check.py` — отдельный pytest-шаг
     `scripts/synthetic_data/tests` → `utils/synthetic_data/tests`;
   - `pyproject.toml` — grep по `scripts/synthetic_data` (per-file-ignores,
     exclude'ы) и правка; `utils/` в ruff/mypy уже подключён;
   - `.pre-commit-config.yaml` — regex уже покрывает `utils/`;
   - `tests/test_no_network_imports.py`-аналог внутри пакета — путь скана;
   - документация: `.ai/project/10-project-map.md`, `11-commands.md`
     (+ `sync-agents`), README пакета.
7. Зарегистрировать в каталоге раннера
   `utils/dev_tools_scripts_runner/config/scripts.json`: title
   `synthetic-data`, filename `synthetic_data.py`, directory `scripts`,
   категория `demo` (Demo Dataset Management), алиасы `synth`,
   `syn-data`; примеры: `generate --scale small`, `validate`, `list`.
8. Опциональный флаг гейта: в `full_check.py` фаза
   `synthetic-data smoke` (`validate` + `generate --scale small --dry-run`),
   включается только `--with-synthetic`, **по умолчанию выключена**.
9. Удалить `docs/synthetic_data/` полностью.

Проверка: `py -3.12 -m pytest utils/synthetic_data/tests -q` зелёный;
`python scripts/synthetic_data.py validate` и `generate --scale small`
работают; полный гейт зелёный; `git grep scripts/synthetic_data` и
`git grep docs/synthetic_data` не находят живых ссылок.

Объём: средний. Риск: низкий (чистая механика + широкая обвязка).

### Этап 1 — Synthetic Web Environment (единственный крупный новый слой)

Цель: генерируемая из того же `SyntheticWorld` сеть локальных сайтов, по
которой можно ходить браузером, скачивать синтетические документы, и на
которую ссылаются источники приложения.

Новые модули `utils/synthetic_data/web/`:

```text
web/
  __init__.py
  models.py         # SyntheticSite, SitePage, LinkEdge, PageAnomaly
  archetypes.py     # загрузка/валидация web_config.json (типы сайтов и их структура)
  graph.py          # детерминированное построение графа страниц и ссылок
                    # (внутрисайтовых и межсайтовых), привязка к странам/событиям/источникам мира
  html_renderer.py  # рендер HTML из stdlib-шаблонов (БЕЗ новых зависимостей: никакого jinja2)
  assets.py         # общий css, изображения-заглушки, favicon
  anomalies.py      # намеренно «плохие» страницы: 404, 500, циклы, редиректы,
                    # дубликаты, пустые/огромные страницы, битая кодировка — доли задаются web_config.json
  validation.py     # каждая ссылка резолвится, КРОМЕ зарегистрированных аномалий;
                    # каждая ссылка «скачать файл» указывает на существующий артефакт манифеста
  server.py         # раздача: один локальный сервер, /sites/<slug>/... и /files/...
                    # (FastAPI StaticFiles поверх mock_server ИЛИ stdlib http.server — решить в задаче этапа)
```

Ключевые решения:

- **Один сервер + path-роутинг** (`http://localhost:8080/sites/<site>/...`),
  без nginx, без контейнера-на-сайт, без локального DNS — ровно вывод
  «легче для Docker» из приложения А. Виртуальные хосты — отложенная
  опция, если когда-нибудь реально понадобится.
- Скачивание: страницы ссылаются на **уже сгенерированные** документы
  датасета (`documents/<locale>/pdf/...` и т.д.), сервер отдаёт их с
  `Content-Disposition: attachment` — «скачать/открыть» работает как в
  настоящем вебе.
- Интеграция с миром: `SyntheticSource.url` (сейчас placeholder) начинает
  указывать на конкретную страницу конкретного синтетического сайта —
  клик по источнику в приложении открывает живую локальную страницу.
- Каждая HTML-страница несёт маркер синтетичности (комментарий + видимый
  футер `SYNTHETIC TEST DATA — NOT REAL`).
- Выходной слой датасета: `<dataset>/websites/<site_slug>/...`; граф
  ссылок и список аномалий — в манифест и `validation_report.json`.

CLI: `generate --formats web` (в общем прогоне) и отдельно
`render-web --dataset <id>`, `serve --dataset <id> --port 8080`.

Проверка: тесты на детерминизм графа, на резолв всех не-аномальных
ссылок, на соответствие аномалий заданным долям, на скачивание файла
через сервер; ручной проход браузером.

Объём: большой (но изолированный — ни одна существующая подсистема не
меняется, кроме `SyntheticSource.url` и манифеста). Риск: средний.

**Статус.** Реализовано полностью: `utils/synthetic_data/web/` (models,
archetypes, graph, anomalies, html_renderer, assets, validation, server —
все 8 модулей из плана), `input_data/web_config.json` (4 архетипа сайтов,
доли аномалий), CLI-команды `generate --formats web` / `render-web` /
`serve` в `cli.py`. Один сервер на FastAPI (уже существующая зависимость
проекта — не `http.server`, выбрано ради `Content-Disposition`,
редиректов и корректных статус-кодов из коробки без ручной реализации
HTTP-протокола). 39 новых тестов (`test_web_*.py`), итого 341 в пакете;
ruff/mypy чистые.

Одно отклонение от буквального текста плана, найденное и
задокументированное в процессе: `SyntheticSource.url` **не изменяется**.
`core/world_validation.py` требует, чтобы это поле всегда начиналось с
`synthetic://` — жёсткий, уже существующий инвариант, а не пока-не-
заполненный placeholder. Вместо мутации поля `web/graph.py`'s
`source_page_urls()` строит отдельный маппинг `source_id -> "/sites/..."`,
записываемый в `websites/source_pages.json` и попадающий в `manifest.json`
— «живая страница источника» резолвится через этот маппинг, а не через
изменение канонической модели мира. Это сохраняет полную обратную
совместимость `world_generator.py` и его golden-фикстуру
(`test_smoke_snapshot.py`) нетронутыми.

### Этап 2 — полный локальный стек на синтетике («ядро миграции данных»)

Цель: одна команда поднимает приложение целиком на синтетическом мире —
включая пользователей и пользовательский контент в реальной БД.

Шаги:

1. Аналитический спайк (обязательный первый шаг): маппинг
   `SyntheticUser/SyntheticArticle/SyntheticComment` на реальные таблицы
   (`users`, community-таблицы миграции 051, user stories). README
   пакета честно фиксирует, что сейчас соответствия нет — где маппинг
   неочевиден, решение выносится владельцу, а не выдумывается.
2. Расширение `core/sql_fixture.py`: пользователи (фиктивные email на
   `example.test`, фейковый password-hash, роли), статьи/комментарии по
   результатам спайка. Те же гарантии: `syn_<dataset_id>`-идентичность,
   идемпотентность, `cleanup-sql` удаляет только своё, production-guard.
3. Новая команда `bootstrap-app`: цепочка
   `apply_migrations.py` → `load-sql` → `rebuild_search_index.py --all` →
   `bootstrap_runtime_read_models.py` — «мир в работающем приложении»
   одной командой.
4. Тесты: идемпотентность новых фикстур, изоляция cleanup, e2e-смоук
   входа синтетическим пользователем на локальном стеке.

Объём: средний. Риск: средний (зависит от спайка; возможны решения владельца).

**Статус.** Реализовано полностью. Аналитический спайк (шаг 1) нашёл, что
в реальной схеме нет концепции «публичные статьи + комментарии» вообще —
владелец решил (см. вопрос в чате): SyntheticArticle/SyntheticComment
остаются JSON-only, а `SyntheticUser` маппится на `users` +
`user_auth_credentials`, все синтетические пользователи получают обычную
роль `role='user'`. Пароль реальный и задокументированный
(`SYNTHETIC_USER_PASSWORD` в `core/sql_fixture.py`), хеш строится
детерминированной солью (dataset_id+user_id), но тем же
PBKDF2-алгоритмом/итерациями, что и реальное приложение — верифицируется
настоящей `verify_password()`. `bootstrap-app` реализована в `cli.py`.

Живая проверка (шаг 4) на реальном `docker compose up -d postgres redis`
+ `docker compose up --build -d api`: полная цепочка `bootstrap-app`
отработала, `POST /api/v1/auth/login` с email синтетического пользователя
и документированным паролем вернул 200 и рабочий токен сессии; неверный
пароль — 401; после `cleanup-sql` тот же логин снова 401 (пользователь
реально удалён). 11 новых тестов на идемпотентность/детерминизм/роль/хеш/
маркировку/изоляцию cleanup — все зелёные (33 в `test_sql_fixture.py`
итого).

**Найден и НЕ исправлен (вне скоупа Этапа 2) реальный баг**: живая
проверка вскрыла, что `country_fixture_ids()` (код Этапа 0/1, не Этапа 2)
даёт `countries.iso2`/`iso3` только по индексу страны в датасете, не по
`dataset_id` — второй датасет, загруженный без `cleanup-sql` первого,
падает на `UniqueViolation` (`countries_iso2_key`). Подробности и
обоснование, почему не чинится прямо здесь — `utils/synthetic_data/README.md`
«Known limitations».

### Этап 3 — замена `database/fixtures/demo_countries` (только с явного одобрения владельца)

Цель: реальные Россия/Аргентина/Уругвай уходят из фикстур; их место
занимает «консервированный» синтетический набор.

Это самый связанный этап: демо-страны защищены инвариантом
(`is_demo`, conservation-логика), на их slug'и завязаны e2e-спеки
(`tests/e2e/web-mvp-*.spec.ts`), рантайм-смоуки `full_check.py`
(`/api/v1/countries/russia/trust` и т.п.) и скрипты
`restore/export_demo_countries.py`. Поэтому — фазово:

1. Зафиксировать seed → сгенерировать эталонный набор из 3 синтетических
   стран; экспортировать его в формате demo-фикстур (тот же
   `scripts/dev_tools/_demo_countries_fixture_spec.py`) в
   `database/fixtures/demo_countries_synthetic/` — **рядом**, не вместо.
2. Параметризовать слуги демо-стран в e2e и смоуках (константа/фикстура
   вместо hardcode «russia»), переключение флагом.
3. Прогнать полный гейт и все e2e на синтетическом наборе; добиться
   зелёного.
4. Переключить дефолт на синтетический набор; удалить
   `database/fixtures/demo_countries/` с реальными данными; переименовать
   синтетический каталог в `demo_countries/`.
5. Обновить документацию (`docs/_arch_/…`, `.ai/project/12-domain-rules.md`
   — правило «conserved demo countries» переформулируется на синтетические
   слуги) через `sync-agents`.
6. `database/migrations/` не трогаются вообще (инвариант); семантика
   `is_demo` сохраняется как есть.

Примечание: `docs/_arch_/08_Открытые_вопросы.md` уже фиксирует намерение
увести e2e со старого фиксированного набора — этап закрывает и его.

Объём: средний-большой, в основном правки тестов. Риск: высокий по
площади затрагивания → выполняется строго отдельной задачей, после
явного «да» владельца, с планом отката (реальные фикстуры остаются в
истории git).

1. Финальная документация: README пакета как канонический референс;
   команды — в `.ai/project/11-commands.md` + `sync-agents`.
2. Ревизия «известных ограничений» README: что закрыто этапами 1–3,

## 7. Чего сознательно НЕ делаем (защита от «закапывания»)

| Идея из первоначального видения | Решение | Почему |
| --- | --- | --- |
| nginx, virtual hosts, локальный DNS, контейнер-на-сайт | нет; один сервер + path-роутинг | тот же реализм для тестов, на порядок меньше эксплуатации |
| Генерация через LLM | нет (только seam на будущее) | правило проекта: offline-by-default, интеграции — отдельным траншем |
| Видео-файлы | нет | приложение их нигде не потребляет |
| Новые реальные страны/языки, AI-перевод | нет | прямой запрет `.ai/project/12-domain-rules.md` |
| «Новые данные при каждом запуске» без seed-контроля | нет; случайный, но печатаемый и воспроизводимый seed | иначе теряется воспроизводимость багов |
| Хранение сгенерированного в git | нет; `.synthetic_data/` gitignored | репозиторий — код и конфиги, не артефакты |
| Синтетика в дефолтном quality gate | нет; только `--with-synthetic` / явная команда | гейт и так ~10 минут |
| Отдельный «synthetic-world» микросервис в docker-compose | не сейчас | статическая генерация + локальный сервер закрывают все текущие сценарии; compose-сервис — только если появится реальная нужда |

## 9. Definition of Done (сквозной)

- [+] Этап 0: пакет живёт в `utils/synthetic_data/`, точка входа
      `scripts/synthetic_data.py`, `docs/synthetic_data/` удалён,
      раннер знает команду `synthetic-data`, весь гейт зелёный.
- [+] Этап 1: `generate --formats web` / `render-web` создают браузируемые
      сайты; ссылки валидируются (`web/validation.py`); аномалии
      управляются `web_config.json`; `serve` раздаёт сайты и файлы с
      корректными HTTP-статусами (404/500/302) для каждого вида аномалии;
      «живая страница источника» резолвится через
      `websites/source_pages.json` (не через мутацию `SyntheticSource.url`
      — см. «Статус» у Этапа 1 выше).
- [+] Этап 2: `bootstrap-app` поднимает приложение на синтетике одной
      командой, включая пользователей в БД (контент — сознательно
      JSON-only, см. «Статус» у Этапа 2 выше). Живой e2e-логин подтверждён.
- [ ] Этап 3 (после одобрения): в `database/fixtures/` нет данных
      реальных стран; e2e и смоуки зелёные на синтетическом наборе.
- [ ] Этап 4: документация консистентна, чистый клон проходит гейт.

Каждый этап дополнительно обязан: тесты добавлены/обновлены, ruff/mypy
чистые, полный гейт зелёный, `task-checklist.md` заполнен `+`/`-`,
финальный отчёт написан.

## 10. Риски

| Риск | Митигация |
| --- | --- |
| Переезд (этап 0) порвёт скрытые ссылки на старые пути | `git grep` по `scripts/synthetic_data` и `docs/synthetic_data` как обязательный шаг проверки; полный гейт до и после |
| Веб-слой начнёт выдумывать собственные факты и разъедется с миром | архитектурное правило: web — рендерер `SyntheticWorld`, без собственных генераторов сущностей; валидация ссылок против манифеста |

| Маппинг статей/комментариев на реальные таблицы окажется неоднозначным | обязательный аналитический спайк, неоднозначности — решением владельца |

| Windows-специфика (кодировки консоли, длинные пути) | уже отработана в пакете (UTF-8 везде, шрифты встроены); тесты гоняются на Windows локально и на ubuntu в CI |

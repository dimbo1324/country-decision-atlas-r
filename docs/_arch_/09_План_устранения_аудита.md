# План устранения замечаний аудита

> Источник: `audit_result.txt` (в этой же папке) — полный 10-разделный аудит
> проекта от 2026-07-08 (ветка `chore/full-project-audit`, слит в `main`
> коммитом `3a3d47c`). Раздел 11 чек-листа («Поддерживаемость и технический
> долг») в аудит не входил — владелец ведёт его отдельно сам.
>
> Этот документ разбивает **все 39 находок** аудита (3×P0, 10×P1, 14×P2,
> 12×P3) на изолированные, логически завершённые аудит-эпизоды. Каждый
> эпизод — отдельная ветка `fix/<slug>-v1`, ведётся по `00_Рабочий_стандарт.md`
> (полный quality gate перед merge, task-checklist.md, финальный отчёт).
> Нумерация — «Аудит-эпизод N» (AE-N), чтобы не путать с продуктовыми
> эпизодами 1–7 из `02_План/01_План_реализации.md`.
>
> **Принцип комплектации:** находки сгруппированы не по разделу чек-листа, а
> по **общей зоне кода и общему риску слияния** — так, чтобы один PR можно
> было тестировать и ревьюить как одно целое. Порядок эпизодов — по
> убыванию критичности и с учётом зависимостей (см. карту ниже).
>
> **Каждая находка закрывается ровно в одном эпизоде.** Если находка требует
> решения владельца (не чисто техническая правка), это явно помечено
> «⚠ Решение владельца» с вариантами на выбор — работа по эпизоду стартует
> только после выбора варианта.

### Аудит-эпизод 1 — `fix/db-index-safety-v1`: индексобезопасность и сессии

**Цель.** Убрать анти-паттерн, который отключает индексы почти на каждом
авторизованном запросе, и сократить число обращений к БД на путь проверки
сессии.

**Находки и файлы.**

- **P0-1** — `WHERE колонка::text = %s` (187 вхождений) в
  `apps/api/app/repositories/auth.py`, `capabilities.py`,
  `author_metrics/*.py`, `country_contribution/*.py`, `admin_content.py`,
  `countries.py:111,150`, `country_pairs.py:93,127`,
  `decision_passports.py:134`, `common.py:75` и др.
- **P3-1** — `common.py:69-81`, `countries.py:101-121`: `id::text = %s OR
  slug = %s` в одном методе.
- **P3-2** — `auth.py:343`: `fetch_all()` вместо `execute()` для UPDATE.
- **P1-8** — `services/auth.py:219`, `repositories/auth.py:343-353`:
  `touch_session` пишет на каждый запрос.
- **P2-14** — `services/auth.py:190-220`: 3–4 отдельных запроса в
  `validate_session_token`.
- **P3-9** — путь-параметры вида `proposal_id`/`definition_id` не
  типизированы как UUID на границе роутера.

**Что делаем.**

1. Механическая замена `(\w+)::text = %s` → `\1 = %s::uuid` по всем
   репозиториям (кроме случаев, где колонка реально `text`). Добавить
   grep-тест-инвариант (по образцу существующих `tests/test_*_mig.py`),
   запрещающий появление `::text = %s` в `apps/api/app/repositories/`.
2. `common.py`/`countries.py`: развести `get_by_id`/`get_by_slug` на два
   явных метода; публичные эндпоинты — только по slug.
3. `touch_session`: сравнивать `last_seen_at` с `NOW() - interval`, писать
   UPDATE только при превышении порога (параметр, например 5 минут);
   заменить `fetch_all` на `execute`.
4. `validate_session_token`: один запрос — JOIN `auth_sessions` × `users`
   по `token_hash`, статус (active/expired/revoked) вычислять из полей
   результата вместо отдельных выборок.
5. Роутеры с path-параметрами `*_id`: сменить тип аннотации на `UUID` —
   FastAPI сам вернёт 422 на невалидный формат.

**Тесты/acceptance.** Полный `pytest tests/` зелёный без правок ожиданий
(поведение не меняется, только SQL и число round-trip'ов). Новый
grep-тест-инвариант проходит. Repository-тесты (если уже есть к этому
моменту из AE-8) — обновить под новые методы `get_by_id`/`get_by_slug`.
Нагрузочно не измеряем (нет baseline) — но `EXPLAIN` на 2–3 горячих запросах
до/после в финальном отчёте.

**Объём.** Без миграции. Правки в 20+ файлах репозиториев (в основном
однотипные), 2 сервисных файла, 1 новый тест-инвариант. Оценка: 1–1.5 дня.

**Статус.** Реализовано на ветке `fix/db-index-safety-v1` (не смерджена,
ждёт explicit push/merge по решению владельца). Все 187 вхождений
`::text = %s` в 37 файлах репозиториев заменены на `= %s::uuid`; добавлен
grep-тест-инвариант `tests/test_repository_index_safety_invariant.py`.
`touch_session` переведён на `execute()` и троттлинг по
`auth_session_touch_interval_minutes` (новая настройка). `validate_session_token`
сведён к одному JOIN-запросу (`repository.get_session_with_user_by_token_hash`);
старые `get_active_session_by_token_hash`/`get_session_by_token_hash` удалены.
Path-параметры `*_id` типизированы как `UUID` в ~40 эндпоинтах (20 файлов
`api/v1/`). Отступление от наброска: при разведке P3-9 та же
id-или-slug-дыра (P3-1) нашлась ещё в 4 репозиторных функциях сверх изначально
указанных в аудите файлов — устранена в этом же эпизоде, а не отложена;
контракты `/countries/{country_id}/profile|scores|legal-signals` сужены до
`{country_slug}` (раскрытый breaking change, ни одного реального вызывающего
места с id не найдено). `contracts/openapi.yaml` и generated types
обновлены. Полный quality gate (Docker + миграции + Playwright E2E) зелёный;
`EXPLAIN` до/после подтверждает переход `Filter` → `Index Cond` на
`auth_sessions`. Подробности — `task-checklist.md` в истории ветки.

### Аудит-эпизод 2 — `fix/login-rate-limiting-v1`: периметр — брутфорс и лимитирование

> Ветка переименована из `fix/rate-limiting-v1` — старая одноимённая ветка на
> удалённом репозитории осталась в нерабочем (до-ребейзном) состоянии, чтобы
> избежать путаницы её удалили из работы, реализация начата заново с чистого
> `main` (уже включающего Аудит-эпизод 1).

**Цель.** Закрыть реальную возможность перебора пароля и злоупотребления
лимитером через подделку заголовков — независимо от `APP_ENV`.

**Находки и файлы.**

- **P0-3** — `services/auth.py` (`login_user`): нет учёта неудачных попыток
  по аккаунту.
- **P1-6** — `bootstrap/app_factory.py:141-162`, `main.py:18-59`:
  in-memory лимитер, работает только при `APP_ENV=production`,
  per-process, умножается воркерами.
- **P3-12** — `main.py:23-32`: `X-Forwarded-For` берётся без проверки
  доверенных прокси.
- **P3-5** — `register_user`: 409 `email_already_registered` допускает
  перечисление email (низкий риск, но чинится тем же PR).

**Что делаем.**

1. Ввести в БД или Redis счётчик неудачных попыток логина per-account:
   после N неудач (параметр методологии) — растущая задержка/временная
   блокировка + audit-событие. N и окно — через существующий
   `methodology_config.py` (не хардкод).
2. Перенести rate limiter на Redis (INCR+EXPIRE или sorted-set скользящее
   окно) — инфраструктура уже в `docker-compose.yml`. Включить независимо
   от `APP_ENV`; отдельный жёсткий лимит на `/auth/login` и
   `/auth/register` (например 5/мин/IP + 20/час/email), общий лимит — как
   есть.
3. `X-Forwarded-For`: валидировать, что `request.client.host` входит в
   список доверенных прокси (новый параметр конфигурации), прежде чем
   доверять заголовку; иначе — использовать `request.client.host` напрямую.
4. Register: оставить 409 (стандартное поведение), но задокументировать
   компромисс в коде одной строкой — без изменения ответа (сознательно
   не блокирующая находка, чинить не обязательно техническими средствами;
   зафиксировать как принятый риск в финальном отчёте эпизода).

**Тесты/acceptance.** Unit: N+1 неудачный логин блокирует N+2-й попыткой с
верным паролем (пока не истечёт окно); лимитер работает при
`APP_ENV=local` и при рестарте процесса (данные в Redis переживают
рестарт); подделанный `X-Forwarded-For` без доверенного прокси не влияет
на лимит. Regression: `NullCache`/redis-недоступен → лимитер fail-open с
логом предупреждения (не блокирует логин полностью).

**Объём.** 1 новая таблица или Redis-структура (без миграции, если Redis),
правки в `auth.py` (service+repository), `app_factory.py`, `main.py`,
~20–25 тестов. Оценка: 1–1.5 дня.

**Статус.** Реализовано на ветке `fix/login-rate-limiting-v1` (не смерджена,
ждёт explicit push/merge). Новый модуль `services/rate_limiter.py`
(Redis INCR+EXPIRE, синглтон-клиент, fail-open). P0-3: per-account
блокировка после `AUTH_LOGIN_MAX_FAILED_ATTEMPTS` неудач с растущей
длительностью (капается `AUTH_LOGIN_LOCKOUT_MAX_SECONDS`); события — через
структурные логи, не `audit_events` (constraint на `action` рассчитан на
действия аутентифицированного актора, см. финальный отчёт). P1-6:
отдельный always-on лимит `API_RATE_LIMIT_AUTH_PER_MINUTE` на
`/auth/login`/`/auth/register`, независимо от `APP_ENV`; общий лимит на
остальной трафик **оставлен prod-only, как и предполагал этот план** (см.
пункт 2 «Что делаем» выше) — при первой попытке реализации это по ошибке
было снято для всего трафика, что вызвало нестабильность Playwright E2E под
нагрузкой (синхронный Redis-вызов на каждый запрос внутри async-миддлвари);
диагностировано контрольным прогоном на чистом коде и исправлено (плюс
вызов обёрнут в `run_in_threadpool` на будущее). P3-12: `X-Forwarded-For`
учитывается только от IP из `TRUSTED_PROXY_IPS`. P3-5: оставлено как есть,
принято как риск. Полный quality gate (Docker + миграции + Playwright E2E
278/278) зелёный. Подробности — `task-checklist.md` в истории ветки.

### Аудит-эпизод 3 — `fix/session-security-v1`: сессии и XSS-периметр

✅ **Решение владельца принято: вариант Б, плюс три меры усиленной защиты**
(короткоживущие ротируемые токены, видимость сессий/устройств с
уведомлением о новом устройстве, step-up повторная аутентификация для
массового отзыва сессий) — см. «Статус» ниже.

**Находки и файлы.**

- **P0-2** — `apps/web/src/shared/auth/session.ts`: токен в `localStorage`,
  TTL 7 суток (`core/config.py: auth_session_ttl_hours = 168`), нет CSP.
- **P2-5** — `bootstrap/app_factory.py:129-139`: устаревший
  `X-XSS-Protection`, нет HSTS/CSP/Permissions-Policy.

**Варианты (выбрать один до начала работы):**

| Вариант | Что делаем | Стоимость | Закрывает риск |
|---|---|---|---|
| **А. Минимальный** | Добавить CSP (`default-src 'self'` и т.д.) в Next `headers()`, убрать `X-XSS-Protection`, добавить `Permissions-Policy`; сократить TTL сессии до 24–48ч со скользящим продлением (`touch_session` уже есть); HSTS — заготовить за флагом, включить при появлении HTTPS-хостинга | 0.5 дня | Сокращает окно эксплуатации XSS, не устраняет саму возможность кражи токена |
| **Б. Полный (рекомендуется)** | Перевести сессию на `httpOnly + Secure + SameSite=Lax` cookie, выдаваемую бэкендом; добавить CSRF-защиту (double-submit token) на мутирующих эндпоинтах; `Authorization`-заголовок оставить как альтернативный путь для API-клиентов; плюс всё из варианта А | 2–3 дня | Устраняет саму возможность кражи токена через XSS (JS не видит cookie) |

**Рекомендация:** вариант Б — вариант А снижает риск, но не устраняет его,
а платформа целенаправленно наращивает поверхность пользовательского
контента (Q&A, доска, авторские методологии), где будущий XSS почти
неизбежен статистически.

**Что делаем (при выборе Б).**

1. Backend: `/auth/login`, `/auth/register` начинают устанавливать
   `Set-Cookie: cda_session=<token>; HttpOnly; Secure; SameSite=Lax; Max-
   Age=...` в дополнение к телу ответа (переходный период — оба канала);
   `validate_session_token` читает токен из cookie, если заголовка
   `Authorization` нет.
2. CSRF: генерировать csrf-токен при логине, отдавать не-httpOnly cookie с
   ним + требовать соответствующий заголовок на POST/PATCH/DELETE
   (double-submit). Middleware проверки — по образцу существующих
   middleware в `app_factory.py`.
3. Frontend: `AuthProvider.tsx`/`session.ts` перестают писать токен в
   `localStorage`; `http.ts` добавляет csrf-заголовок из non-httpOnly
   cookie на мутирующие запросы; `credentials: "include"` на fetch.
4. CSP/HSTS/Permissions-Policy — как в варианте А, добавить в обоих
   случаях.
5. Контракт: если публичный API (внешние клиенты) теряет
   `Authorization`-путь — не терять, оставить как fallback (объявить в
   `contracts/openapi.yaml` оба способа аутентификации).

**Тесты/acceptance.** Логин выставляет httpOnly cookie; JS-доступ к токену
невозможен (тест через `document.cookie` в Playwright); мутирующий запрос
без csrf-заголовка получает 403; E2E-логин/логаут проходят через cookie-
поток. CSP не ломает существующие inline-скрипты (проверить Next.js
`_document`/`next/script` — при необходимости nonce-based CSP).

**Объём.** Backend: auth service+repository+middleware (~1 день). Frontend:
AuthProvider, http.ts, все вызовы с `credentials` (~1 день). CSRF-слой
(~0.5 дня). Тесты ~30–40. Итого 2.5–3 дня при варианте Б, 0.5 дня при
варианте А.

**Статус.** Реализовано на ветке `fix/session-security-v1` (вариант Б плюс три
меры усиленной защиты, согласованные с владельцем сверх базового плана).
P0-2/P2-5: `/auth/login`/`/auth/register` выставляют httpOnly+Secure(prod)+
SameSite=Lax cookie `cda_session` и не-httpOnly cookie `cda_csrf`
(double-submit), сохраняя `token` в теле ответа как переходный
альтернативный путь для API-клиентов; `core/auth.py` читает токен из cookie,
если нет `Authorization`; новый `csrf_protection_middleware` (double-submit,
исключения для login/register и запросов с `Authorization`) в
`app_factory.py`; `CORSMiddleware.allow_credentials=True`; security-headers —
CSP (`script-src 'self' 'unsafe-inline'` — принятый временный компромисс,
см. ниже), Permissions-Policy, HSTS за флагом `security_hsts_enabled`
(выключен по умолчанию, как и предполагал план — «включить при появлении
HTTPS-хостинга»); убран устаревший `X-XSS-Protection`; те же заголовки
добавлены в `next.config.mjs`. Frontend: `session.ts` больше не хранит
токен в `localStorage` (замена — `csrfHeaders()`, читает CSRF-cookie);
`AuthProvider.tsx` полагается на cookie; `http.ts` — `credentials:
"include"` на всех запросах.

Сверх базового плана эпизода (по решению владельца):
1. **Ротация токена сессии.** Токен больше не статичен на весь TTL: при
   каждой валидации, если с последней ротации прошло больше
   `auth_session_rotation_interval_minutes` (по умолчанию 15 мин),
   `validate_session_token` генерирует новый токен, сохраняет предыдущий
   хеш с grace-периодом `auth_session_rotation_grace_seconds` (по умолчанию
   30с — гасит гонку параллельных запросов, использующих ещё не обновлённый
   токен), и возвращает новый токен клиенту через cookie (для браузера) и
   заголовок `X-Session-Token` (для API-клиентов на `Authorization`).
   Миграция `052_session_security_hardening.sql` добавляет
   `previous_token_hash`/`previous_token_expires_at`/`rotated_at`.
2. **Видимость сессий/устройств + уведомление о новом устройстве.**
   `auth_sessions` получил `device_label`/`ip_display`/
   `device_fingerprint_hash` (человекочитаемая метка устройства из
   User-Agent, маскированный IP для отображения, отдельный солёный хеш для
   определения «то же устройство» — раньше существовавшие
   `user_agent_hash`/`ip_hash` никогда не заполнялись роутером, это тоже
   исправлено в этом эпизоде). `GET /auth/sessions` отдаёт метку/IP и
   `is_current`. При входе с устройства, для которого раньше не было ни
   одной сессии пользователя, создаётся запись в новой таблице
   `user_security_notifications` (`event_type='new_device_login'`) и
   расширяется `audit_events_action_check` значением `new_device_login`;
   `GET /auth/security-notifications` +
   `POST /auth/security-notifications/{id}/ack` дают фронтенду баннер «Вход
   с нового устройства — это были вы?» в `AccountView.tsx`. Уведомление
   работает полностью офлайн (без внешней интеграции) — по правилу
   автономной разработки; доставка через Telegram/email осталась бы за
   контуром интеграционного транша, если владелец решит её добавить позже.
3. **Step-up повторная аутентификация.** `POST /auth/sessions/revoke-all`
   (единственное существующее в кодовой базе действие, которое действительно
   меняет состояние безопасности аккаунта — отзыв всех устройств) теперь
   требует `current_password` в теле запроса;
   `service.require_step_up_reauthentication` — переиспользуемая функция,
   готовая для будущих чувствительных действий (смены пароля/email), которых
   в продукте пока нет.

Отступление от наброска: при реализации CSRF-мидлвари обнаружен и попутно
исправлен баг, унаследованный от AE-2, — `CORSMiddleware` регистрировался
до остальных мидлварей и оказывался самым внутренним слоем, из-за чего
любой ранний `error_response` (403 у CSRF, 429 у rate-limit) не получал
CORS-заголовков и в браузере выглядел как неотличимый от сетевого сбоя
`Failed to fetch`, а не как читаемый JSON с кодом ошибки; порядок
регистрации изменён так, что CORS теперь самый внешний слой — оба случая
устранены одним и тем же изменением. Обнаружено и исправлено E2E-тестом
`web-mvp-session-security.spec.ts` в рамках этого эпизода.

Известный принятый компромисс: CSP использует `script-src 'self'
'unsafe-inline'` вместо nonce-based CSP (план явно допускал это как
временный шаг: «HSTS/CSP как в варианте А… при необходимости nonce-based
CSP»); ужесточение до nonce-based CSP — отложенный технический долг,
не блокирующий этот эпизод.

Тесты: unit (ротация — свежий/просроченный/grace-период предыдущего
токена, определение нового устройства, step-up отклоняет неверный пароль,

### Аудит-эпизод 9 — `fix/frontend-resilience-v1`: фронтенд — устойчивость сети

**Цель.** Дать фронтенду базовую защиту от зависшего бэкенда; зафиксировать
судьбу более крупной находки о рендер-стратегии.

**Находки и файлы.**

- **P2-11** — `apps/web/src/shared/api/http.ts`: `apiGet`/`apiPost`/
  `apiPatch`/`apiDelete` без `signal`/таймаута.
- **P2-12** — 22× `export const dynamic = "force-dynamic"`,
  `cache: "no-store"` везде, страница страны делает ~10 отдельных
  клиентских запросов вместо одного агрегата.

**Что делаем (P2-11 — делаем сейчас, независимо).**

1. `http.ts`: добавить `AbortSignal.timeout(N)` (например 10с,
   конфигурируемо через `shared/config/env.ts`) во все четыре метода;
   единообразный маппинг `AbortError` → уже существующий `ErrorState`.

**P2-12 — статус: сверка с продуктовым планом, не самостоятельный фикс.**

Аудит сам отмечает, что полная переработка рендер-стратегии (ISR/
`revalidate`, агрегатный fetch на странице страны, `revalidateTag` по
publish-событиям) — это прямой скоуп уже запланированного **«Визуального
транша»** (`02_План/01_План_реализации.md`, раздел после Эпизода 7).
Переделывать рендеринг сейчас, до начала этого транша, означало бы делать
работу дважды и нарушать принцип «одна задача — один логический кусок».

Решение для этого эпизода: **P2-11 закрывается здесь; P2-12 не
закрывается здесь** — вместо кода эпизод добавляет один абзац в раздел
«Визуальный транш» плана реализации (`02_План/01_План_реализации.md`) с
конкретными техническими шагами из аудита (ISR/revalidate по типу
страницы; один агрегатный `country_read_model`-fetch на странице страны
вместо ~10 блочных; инвалидация по publish-событиям через существующие
хуки `cache_invalidation.py`), чтобы это не потерялось к моменту старта
транша. Если владелец хочет закрыть P2-12 раньше — это отдельное решение
вне текущего пакета (эпизод не блокируется этим выбором).

**Тесты/acceptance.** Зависший mock-бэкенд (задержка > таймаута) даёт
контролируемый `ErrorState`, а не бесконечную загрузку (Playwright-тест).
Существующие E2E не регрессируют.

**Объём.** 1 файл (`http.ts`) + типы ошибок, ~8–10 тестов, 1 абзац в
`01_План_реализации.md`. Оценка: 0.5 дня.

**Статус.** Не начато.

### Аудит-эпизод 10 — `fix/architecture-hygiene-v1`: архитектурная гигиена

**Цель.** Закрыть оставшиеся P2/P3-находки — дублирование правил,
неоптимальная пагинация, мелкая гигиена. Делать последним эпизодом,
после того как более рискованные правки (AE-1..9) уже стабилизировались.

**Находки и файлы.**

- **P2-8** — PII-фильтры продублированы в
  `services/author_metrics/helpers.py:21-26`, migration_board PII-
  детекторе, `repositories/data_quality/author_metrics.py:55-84` (SQL-
  regex — версия уже разошлась с Python); lifecycle-статусы
  продублированы между `services/publication.py`,
  `repositories/author_metrics/definitions.py:298,321,348,368`, и
  `country_cards` (миграция `003_decision_engine.sql:19-27`) имеет свой
  несовпадающий набор статусов.
- **P2-2** — `COUNT(*) OVER()` в `audit.py:80`, `capabilities.py`,
  `author_metrics/definitions.py`, `country_contribution/
  proposals.py:165,192`, `migration_board/*`.
- **P3-3** — `admin_country_contribution.py:27`,
  `admin_author_metrics.py`: `status: str | None = Query(None)` без
  enum-валидации.
- **P3-4** — POST-эндпоинты создания контента без idempotency-защиты от
  двойного сабмита.
- **P3-6** — `main.py:69-84`: `/ready` берёт соединение из общего пула.
- **P3-7** — `apps/notifier/internal/app/app.go:89-158`: gRPC-сервер не
  дожидается `GracefulStop()` симметрично HTTP.
- **P3-8** — публичные списки (sources/evidence/timeline):
  LIMIT/OFFSET без keyset-варианта.
- **P3-10** — контент-сиды в миграциях 002-025 — информационная находка,
  без действия (зафиксировать как принятое архитектурное ограничение в
  `02_Реестр_инвариантов.md`, если там ещё не отражено).
- **P3-11** — дублирование `_total(rows)`/`_audit(...)`/proposal-view
  маппингов между `author_metrics`, `country_contribution`,
  `migration_board`.

**Что делаем.**

1. PII: единый модуль-источник паттернов (например
   `services/pii_patterns.py` — паттерны как данные); Python-код
   импортирует напрямую; SQL-версия для DQ либо генерируется из тех же
   паттернов, либо DQ-проверка переводится на Python-скан выборки вместо
   SQL-regex.
2. Lifecycle: миграция, приводящая CHECK-констрейнт `country_cards` к
   общему enum `PublicationStatus` (маппинг легаси-значений
   `editor_reviewed`/`expert_review_required` → ближайшие валидные
   статусы, с уточнением у владельца, если маппинг неочевиден);
   репозиторные `WHERE status IN (...)`-списки генерировать из единого
   источника `publication._ALLOWED` вместо ручного дублирования.
3. `COUNT(*) OVER()` → keyset-пагинация (`created_at, id`) для
   административных/модерационных списков, либо (если фронтенду
   действительно нужен `total`) — отдельный более редкий COUNT-запрос
   вне горячего пути. **Контрактное изменение** — если меняется форма
   ответа пагинации, обновить `openapi.yaml` и сгенерированные типы,
   явно указать breaking change в финальном отчёте эпизода.
4. `Query(None)` → `Query(None, ...)` с `Literal`/enum-типом для
   status-фильтров.
5. Idempotency: disable-on-submit на фронте для create-форм (минимально)
   и/или `Idempotency-Key`-заголовок на самых чувствительных create-
   эндпоинтах (proposals, definitions) — на усмотрение исполнителя
   эпизода, зафиксировать выбор в отчёте.
6. `/ready`: отдельное короткоживущее соединение или мини-пул,
   не пересекающийся с основным пулом запросов.
7. `app.go`: дождаться `grpcSrv.GracefulStop()` симметрично
   `httpSrv.Shutdown()` перед выходом из `Run()`.
8. Публичные списки sources/evidence/timeline: перевести на keyset
   (не обязательно везде — начать с самых глубоких списков).
9. `_total`/`_audit`-хелперы: вынести в общий `services/list_helpers.py`
   (или аналог), сохраняя доступ через квалифицированный импорт модуля
   (не `from .helpers import thing`) — по правилу house style о
   монкипатч-безопасности.
10. P3-10: если реестр инвариантов уже фиксирует «сиды в миграциях
    неприкосновенны» — оставить как есть, отметить в финальном отчёте
    «без действия, подтверждено».

**Тесты/acceptance.** PII-паттерны: один источник, тест на идентичность
результатов Python- и SQL-путей (или удаление SQL-пути, если он
заменяется на Python-скан). Lifecycle: миграция откатываема или
задокументирован необратимый шаг; существующие тесты `country_cards`
проходят с новыми статусами. Пагинация: контрактные тесты обновлены,
frontend-потребители (если есть) не ломаются — проверить
`apps/web` на использование поля `total`. `/ready` не зависит от
насыщенности основного пула (тест: насытить пул, `/ready` всё равно
отвечает быстро). Go: тест на graceful shutdown (если тестовой
инфраструктуры для этого ещё нет — хотя бы ручная проверка +
комментарий в отчёте).

**Объём.** Самый разнородный эпизод по составу; оценка: 2–2.5 дня,
делать по возможности несколькими более мелкими коммитами внутри одной
ветки (или, если оценка при старте окажется больше ожидаемой — разбить
на AE-10a/AE-10b по факту).

**Статус.** Не начато.

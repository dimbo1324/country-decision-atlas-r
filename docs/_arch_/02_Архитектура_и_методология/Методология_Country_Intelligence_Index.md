──────────────────────────────────────────────────────────────────────────────
  ВОПРОС 1. Финальный список измерений Country Intelligence Index
──────────────────────────────────────────────────────────────────────────────

НАУЧНЫЙ ПОДХОД (OECD Handbook, Step 2: Selecting Variables):
  Переменные должны:
  • Быть концептуально связаны с измеряемым явлением
  • Иметь достаточное покрытие стран
  • Обновляться регулярно
  • Быть воспроизводимыми (не субъективными)
  • Не дублировать друг друга (проверяется через корреляционный анализ)

РЕКОМЕНДУЕМЫЙ ФИНАЛЬНЫЙ СПИСОК (6 основных + 2 специфических):

  Группа A: Стабильность и право (политический фундамент)
  ─────────────────────────────────────────────────────────
  1. Political Stability & Security
     Источник: World Bank WGI (PV.EST)
     Что измеряет: вероятность нестабильного правительства или политического
                   насилия, включая терроризм
     Диапазон источника: от -2.5 до +2.5 (нормализуем в 0–100)

  2. Rule of Law & Institutional Quality
     Источник: World Bank WGI (RL.EST) + Transparency International CPI
     Что измеряет: независимость судов, соблюдение контрактов,
                   уровень коррупции
     Диапазон: нормализованный 0–100

  Группа B: Экономика и жизнь
  ────────────────────────────
  3. Economic Freedom & Business Environment
     Источник: Heritage Foundation Economic Freedom Index (Business Freedom)
               + World Bank Business Ready (бывший Doing Business)
     Что измеряет: простота ведения бизнеса, налоги, торговые барьеры
     Диапазон: 0–100

  4. Affordability Index
     Источник: Numbeo Cost of Living Index + World Bank PPP
     Что измеряет: относительная стоимость жизни (100 = мировой baseline,
                   меньше = дешевле)
     Диапазон: 0–200 (инвертируем при агрегации: ниже = лучше)

  Группа C: Качество жизни
  ─────────────────────────
  5. Human Development & Quality of Life
     Источник: UNDP Human Development Index (Education + Health + Income)
     Что измеряет: качество медицины, образования, уровень доходов
     Диапазон: 0–1 (нормализуем в 0–100)

  Группа D: Специфика для мигрантов (уникальная группа)
  ──────────────────────────────────────────────────────
  6. Residency & Migration Accessibility
     Источник: MIPEX + Henley Passport Index + внутренние legal signals
     Что измеряет: простота получения ВНЖ/ПМЖ, доступность виз,
                   бюрократические барьеры
     Диапазон: 0–100 (частично заполняется вручную)

  Дополнительные (добавлять в фазе 2, не MVP):
  7. Digital Infrastructure & Remote Work Readiness
     Источник: ITU ICT Development Index + Ookla Speedtest
  8. Country Drift (вычисляемое из остальных — см. вопрос 7)

ПРИНЦИП ВЫБОРА НА MVP:
  Начинать с 6 измерений группы A–D. Они покрыты хорошими источниками
  и достаточны для убедительного decision output. Добавлять 7–8 позже,
  когда есть данные.

НАУЧНЫЙ ОТВЕТ (когнитивная психология):
  Miller's Law (1956): рабочая память человека удерживает 7 ± 2 объекта.
  Для визуальных объектов на radar chart — ещё меньше из-за угловых искажений.

  Исследования UI/UX показывают:
  • До 5 осей — хорошо читается на любом устройстве
  • 6 осей — оптимальный баланс информации и читаемости
  • 7–8 осей — приемлемо на десктопе, плохо на мобильном
  • 9+ осей — диаграмма теряет смысл, превращается в визуальный шум

РЕКОМЕНДАЦИЯ:
  MVP: 6 осей (группы A–D из вопроса 1)
  Десктоп v2: 8 осей (добавить Digital Infrastructure + Drift)
  Мобайл: всегда 5–6 осей максимум

  Технически: компонент читает оси из API-ответа и рендерит столько,
  сколько пришло. Не захардкоживать число осей в коде.


──────────────────────────────────────────────────────────────────────────────
  ВОПРОС 3 и 4. Какие метрики "выше = лучше", какие "выше = хуже"
──────────────────────────────────────────────────────────────────────────────

НАУЧНЫЙ ТЕРМИН: Polarity (полярность индикатора)
Источник: OECD Handbook, раздел 5.3 "Dealing with polarity"

ПРАВИЛО: все индикаторы должны быть приведены к единой полярности
ДО агрегации. Стандарт — "выше = лучше". Инвертируемые переводятся
через формулу: нормализованное_значение = max - x (или 1/x для ratio).

ТАБЛИЦА ПОЛЯРНОСТИ ДЛЯ НАШЕГО CII:

  Измерение                    Полярность    Обоснование
  ─────────────────────────────────────────────────────────────────────
  Political Stability          Выше = лучше  Чем стабильнее — тем лучше
  Rule of Law                  Выше = лучше  Чем сильнее институты — тем лучше
  Economic Freedom             Выше = лучше  Чем свободнее экономика — тем лучше
  Affordability Index          Выше = ХУЖЕ   Чем выше стоимость жизни — тем хуже
                               → ИНВЕРТИРОВАТЬ при агрегации
  Human Development            Выше = лучше  HDI: выше = лучше по определению
  Residency Accessibility      Выше = лучше  Чем доступнее ВНЖ — тем лучше
  Digital Infrastructure       Выше = лучше  Чем лучше связь — тем лучше
  Country Drift                Выше = лучше  Положительный drift = улучшение

КАК РЕАЛИЗОВАТЬ В КОДЕ:
  В таблице cii_metric_definitions добавить поле:
    higher_is_better BOOLEAN DEFAULT true

  Перед агрегацией:
    IF NOT higher_is_better:
      normalized_value = 100 - normalized_value
      -- или для ratio-метрик: normalized_value = 100 / raw_value


──────────────────────────────────────────────────────────────────────────────
  ВОПРОС 5. Какие веса использовать по умолчанию
──────────────────────────────────────────────────────────────────────────────

НАУЧНЫЙ ПОДХОД (OECD Handbook, глава 6: Weighting):
  Существует три основных метода взвешивания:

  Метод 1: Equal Weighting (равные веса)
    Каждое измерение получает вес 1/N (для 6 измерений = 0.167 каждое)
    Используют: большинство индексов на старте
    Плюсы: прозрачно, легко объяснить, нет субъективности
    Минусы: предполагает, что все измерения одинаково важны

  Метод 2: Principal Component Analysis (PCA)
    Статистический метод, который сам определяет веса
    на основе корреляций между переменными
    Используют: Environmental Performance Index (Yale)
    Плюсы: данные сами определяют структуру
    Минусы: сложно объяснить пользователю, нужны данные по 30+ странам

  Метод 3: Analytic Hierarchy Process (AHP)
    Эксперты ранжируют попарно важность измерений
    Используют: многие государственные индексы
    Плюсы: экспертное мнение формализовано
    Минусы: субъективно, зависит от выбора экспертов

РЕКОМЕНДАЦИЯ ДЛЯ ПРОЕКТА:

  MVP-фаза: Equal Weighting (метод 1)
    Все 6 измерений = вес 0.167 каждое
    Честно и прозрачно. Документировать явно: "равные веса, v1.0"

  Фаза 2: Scenario-Adjusted Weighting
    Для разных сценариев — разные веса. Это наш главный UX-выигрыш:

    Сценарий "Переезд/ВНЖ":
      Residency Accessibility   0.35  ← главное для этого сценария
      Political Stability       0.20
      Affordability             0.20
      Rule of Law               0.10
      Human Development         0.10
      Economic Freedom          0.05

    Сценарий "Бизнес/Самозанятость":
      Economic Freedom          0.35  ← главное для этого сценария
      Rule of Law               0.25
      Political Stability       0.15
      Digital Infrastructure    0.15
      Affordability             0.05
      Residency Accessibility   0.05

    Сценарий "Безопасность/Политический риск":
      Political Stability       0.45  ← главное для этого сценария
      Rule of Law               0.30
      Human Development         0.15
      Affordability             0.05
      Economic Freedom          0.03
      Residency Accessibility   0.02

    И так для каждого из 5 сценариев. Веса хранятся в БД
    (в таблице scenario_metric_weights), не в коде.

  Фаза 3 (будущее): User-Adjusted Weighting
    Пользователь сам двигает слайдеры важности.
    "Мне важнее безопасность, чем стоимость жизни."
    CII пересчитывается в реальном времени.

КАК РЕАЛИЗОВАТЬ В КОДЕ:
  Таблица scenario_metric_weights:
    scenario_id  UUID
    metric_id    UUID
    weight       NUMERIC  -- от 0 до 1, сумма по сценарию = 1.0
    version      VARCHAR  -- 'v1.0', чтобы менять без потери истории


──────────────────────────────────────────────────────────────────────────────
  ВОПРОС 6. Как считать общий индекс страны (агрегация)
──────────────────────────────────────────────────────────────────────────────

НАУЧНЫЙ ПОДХОД:
  Два стандартных метода агрегации:

  Метод 1: Линейная агрегация (взвешенное среднее)
    CII = Σ(weight_i × normalized_score_i)
    Используют: большинство индексов
    Плюс: просто, понятно
    Минус: компенсаторный — очень высокий score по одному измерению
           может скрыть катастрофически низкий по другому.
           Например: богатая, но крайне опасная страна может иметь
           высокий CII из-за экономики, несмотря на нулевую безопасность.

  Метод 2: Геометрическая агрегация
    CII = Π(normalized_score_i ^ weight_i)
    Используют: UNDP HDI (с 2010 года)
    Плюс: НЕ компенсаторный — низкий score по любому измерению
           тянет весь индекс вниз. Более честно для пользователя.
    Минус: сложнее объяснить, чувствителен к нулям (нужна защита)

РЕКОМЕНДАЦИЯ:
  Использовать ГЕОМЕТРИЧЕСКУЮ агрегацию, как UNDP HDI.
  Это методологически сильнее и честнее для принятия решений о жизни.

  Формула:
    CII(country, scenario) =
      (S1^w1 × S2^w2 × S3^w3 × S4^w4 × S5^w5 × S6^w6) ^ (1/Σwi)

  Где Si = нормализованный score измерения i (0–100)
      wi = вес измерения i для данного сценария

  ВАЖНО: если любой Si = 0, геометрическое среднее = 0.
  Нужна защита от нулей:
    protected_score = max(Si, 0.01)
  Или использовать нижнюю границу нормализации = 1, а не 0.

КАК РЕАЛИЗОВАТЬ В КОДЕ:
  Сервис CIIScoreAggregator:
    1. Получить нормализованные scores всех измерений для страны
    2. Применить полярность (инвертировать где higher_is_better = false)
    3. Получить веса для конкретного сценария
    4. Применить формулу геометрической агрегации
    5. Округлить до 1 знака после запятой
    6. Записать в country_cii_scores с formula_version_id


──────────────────────────────────────────────────────────────────────────────
  ВОПРОС 7. Как считать Country Drift
──────────────────────────────────────────────────────────────────────────────

НАУЧНЫЙ ПОДХОД:
  Country Drift — это темп изменения составного индекса во времени.
  В финансовом анализе аналог — momentum indicator.
  В статистике — это первая разность временного ряда (first difference).

ФОРМУЛА (трёхлетнее скользящее окно):

  Drift(country, year) =
    CII(country, year) - CII(country, year-3)

  Трёхлетнее окно выбрано потому что:
  • Годовые изменения содержат много шума (выборы, кризисы)
  • Пятилетнее окно — слишком медленная реакция
  • Три года — стандарт в политической науке для оценки тренда

НОРМАЛИЗАЦИЯ DRIFT:
  Drift выражается в пунктах CII (-100 до +100)
  Дополнительно: категоризировать для визуала:

    > +10 пунктов   = "Активно улучшается" (зелёный 🚀)
    +3 до +10        = "Умеренное улучшение" (светло-зелёный ↗)
    -3 до +3         = "Стагнация" (серый →)
    -10 до -3        = "Умеренное ухудшение" (жёлтый ↘)
    < -10 пунктов   = "Активно деградирует" (красный 🔻)

РАСШИРЕННАЯ ВЕРСИЯ (Acceleration):
  Можно также считать вторую производную — ускорение изменения:
    Acceleration = Drift(year) - Drift(year-3)
  Если Drift позитивный и Acceleration позитивная — страна улучшается
  и этот процесс ускоряется. Это очень сильный сигнал.

КАК РЕАЛИЗОВАТЬ В КОДЕ:
  CountryDriftCalculator:
    1. Проверить, что есть CII scores для year и year-3
    2. Если year-3 данных нет → drift = null, показывать "N/A"
    3. Считать drift = CII(year) - CII(year-3)
    4. Считать acceleration = drift(year) - drift(year-3) если доступно
    5. Записать в country_cii_scores.drift и .drift_acceleration


──────────────────────────────────────────────────────────────────────────────
  ВОПРОС 8. Как считать confidence
──────────────────────────────────────────────────────────────────────────────

НАУЧНЫЙ ПОДХОД (OECD Handbook, глава 8: Uncertainty Analysis):
  Confidence — это не субъективная оценка "насколько мы уверены".
  Это формализованная мера неопределённости данных.

  Три компонента confidence:

  Компонент 1: Data Freshness Score (свежесть данных)
    Данные свежее 1 года          → freshness = 1.0
    Данные 1–2 года               → freshness = 0.85
    Данные 2–3 года               → freshness = 0.65
    Данные 3–5 лет                → freshness = 0.40
    Данные старше 5 лет           → freshness = 0.15

  Компонент 2: Source Reliability Score (надёжность источника)
    Официальный источник (ООН, ВБ, правительство)  → reliability = 1.0
    Авторитетный независимый институт              → reliability = 0.90
    Экспертная/юридическая фирма                   → reliability = 0.75
    СМИ / журналистское расследование              → reliability = 0.55
    Сообщество / форум / Reddit                    → reliability = 0.35
    Синтетические данные / AI-generated            → reliability = 0.10

  Компонент 3: Coverage Score (покрытие данными)
    Измерение рассчитано на основе 3+ источников   → coverage = 1.0
    На основе 2 источников                         → coverage = 0.80
    На основе 1 источника                          → coverage = 0.60
    Экспертная оценка (нет формального источника)  → coverage = 0.30

  ФОРМУЛА ИТОГОВОГО CONFIDENCE:
    Confidence(metric) = freshness × reliability × coverage

    Пример:
      Данные 2-летней давности (0.65) × ВБ источник (1.0) × 1 источник (0.60)
      = 0.65 × 1.0 × 0.60 = 0.39  → низкий confidence, нужно обновить

  ИТОГОВЫЙ CONFIDENCE CII (агрегация):
    НЕ брать среднее! Использовать взвешенный минимум:
      CII_confidence = Σ(weight_i × confidence_i)
    И дополнительно: если любое confidence_i < 0.30 → показывать
    предупреждение "Часть данных требует проверки"

КАК РЕАЛИЗОВАТЬ В КОДЕ:
  В country_metric_values:
    freshness_score   NUMERIC(3,2)
    reliability_score NUMERIC(3,2)
    coverage_score    NUMERIC(3,2)
    confidence        NUMERIC(3,2)  -- вычисляемое произведение трёх

  При любом обновлении данных пересчитывать freshness автоматически
  (функция в БД или в сервисе).

  ВОПРОС 9. Какие данные пока вводим вручную

ВВОДИТЬ ВРУЧНУЮ НА MVP (нет хорошего автоматического источника):

  1. Residency & Migration Accessibility
     MIPEX не имеет открытого API. Henley — датасет раз в год.
     Реальная оценка доступности ВНЖ сильно зависит от деталей,
     которые нельзя взять из одного числа.
     → Вводить вручную через admin panel, ссылаться на MIPEX + Henley.

  2. Legal Signal Impact Scores
     Каждый legal signal имеет impact_level и impact_direction.
     Это редакторская оценка, не алгоритм.
     → Вводить вручную, с обязательной ссылкой на источник.

  3. User Story Verification Status
     Верифицированность истории пользователя нельзя автоматизировать.
     → Вручную, редактором.

  4. Scenario Weights (пока не сделан PCA)
     На MVP — веса задаются вручную командой продукта.
     → Через admin, с фиксацией версии.

  5. Affordability Details для конкретных городов
     Numbeo платный. Стоимость жизни в Москве ≠ Уфа ≠ Сочи.
     → На MVP использовать World Bank PPP как грубую оценку,
       детализировать вручную для приоритетных стран.

НАУЧНЫЙ ПОДХОД (UX research, F-pattern / Z-pattern eye-tracking):

    → Полный текстовый профиль

ПОЧЕМУ SPIDER CHART ПЕРВЫМ:

НАУЧНЫЙ ПОДХОД (Decision Science, Tversky & Kahneman):
  Люди плохо оценивают абсолютные значения, но хорошо воспринимают
  относительные сравнения. "67 из 100" — абстрактно.
  "Уругвай лучше России по безопасности в 3.5 раза" — конкретно.

  Принцип: decision page должна давать ОТНОСИТЕЛЬНОЕ сравнение,
  а не набор абсолютных чисел.

РЕКОМЕНДУЕМАЯ СТРУКТУРА:

  Часть 1 — Визуальный вердикт (выше fold):
    → Два spider chart наложены друг на друга (страны разным цветом)
    → Winner banner: "Для сценария Переезд/ВНЖ: Uruguay 74 vs Russia 42"
    → Confidence indicator: "Высокая уверенность / 3 источника"

  Часть 2 — Разбор по измерениям:
    → Grouped bar chart: для каждого измерения — бар страны A vs страны B
    → Значок 🏆 у победителя в каждом измерении
    → Разница в пунктах (+/- X)

  Часть 3 — Факторы решения:
    → Strengths страны A (3–5 пунктов)
    → Weaknesses страны A
    → Risk Warnings (критические legal signals)

  Часть 4 — Источники (третий уровень):
    → По клику: откуда взяты данные для каждого вывода

  ЧАСТЬ 2. УНИКАЛЬНЫЕ МЕТРИКИ, КОТОРЫХ НЕТ НИ У КОГО
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Все стандартные индексы (WGI, HDI, GPI, EFI) измеряют текущее состояние
страны. Никто из конкурентов не измеряет то, что описано ниже.
Это наш потенциальный moat — уникальные аналитические продукты.


──────────────────────────────────────────────────────────────────────────────
  УНИКАЛЬНАЯ МЕТРИКА 1: Legal Velocity Index (LVI)
──────────────────────────────────────────────────────────────────────────────

ЧТО ИЗМЕРЯЕТ:
  Скорость изменения законодательства. Страна, где законы меняются
  слишком быстро — рискованная для долгосрочного планирования,
  даже если текущие законы хорошие.

  Парадокс, который никто не измеряет: страна с улучшающимися законами,
  но высокой волатильностью — плохой выбор для долгосрочной жизни,
  потому что непредсказуема.

ФОРМУЛА:
  LVI(country, year) =
    count(legal_signal_events в данном году) / исторический_baseline

  Где baseline = среднее количество изменений за последние 5 лет.

  LVI = 1.0  → нормальная скорость изменений
  LVI > 2.0  → законодательная нестабильность (вдвое выше нормы)
  LVI < 0.5  → законодательная заморозка (очень мало изменений)

НОРМАЛИЗАЦИЯ:
  Optimal range: 0.5–1.5 (стабильные реформы без хаоса)
  Inverted: LVI далеко от 1.0 в любую сторону = хуже

  Score = 100 × e^(-|LVI - 1.0| × 2)  -- колоколообразная кривая
  При LVI = 1.0 → Score = 100 (идеально)
  При LVI = 0.0 или 2.0 → Score → 0

КАК ИСПОЛЬЗОВАТЬ В ПРОЕКТЕ:
  Источник данных: собственная таблица legal_signal_events
  (не нужны внешние API — считается из наших данных!)

  Показывать на: карточке страны в блоке Risk Indicators
  Показывать на: decision page в блоке Stability Risk
  Уникальный текст: "Законодательство меняется в 2.3 раза быстрее нормы"


──────────────────────────────────────────────────────────────────────────────
  УНИКАЛЬНАЯ МЕТРИКА 2: Expat Reality Gap (ERG)
──────────────────────────────────────────────────────────────────────────────

ЧТО ИЗМЕРЯЕТ:
  Разрыв между тем, что говорят официальные индексы, и тем,
  что реально испытывают переехавшие люди.

  Это метрика, которую принципиально нельзя рассчитать без
  пользовательских историй. Поэтому она уникальна для нашей платформы.

  Пример: официальный Rule of Law у страны = 72/100.
  Но переехавшие предприниматели в user stories описывают коррупцию
  как серьёзную проблему. ERG = разрыв между этими данными.

ФОРМУЛА:
  ERG(country, dimension) =
    official_score(dimension) - user_experience_score(dimension)

  user_experience_score считается как средний рейтинг по user stories,
  которые касаются данного измерения (из поля scenario + outcome + problems)

  ERG = 0    → официальные данные соответствуют реальности
  ERG > +15  → официальные данные завышены ("бумажно хорошо, на деле нет")
  ERG < -15  → страна лучше, чем говорят официальные индексы

НОРМАЛИЗАЦИЯ:
  Absolute ERG Score = |ERG| / 100  (насколько велик разрыв)
  Направление ERG = знак (официальное завышено или занижено)

КАК ИСПОЛЬЗОВАТЬ В ПРОЕКТЕ:
  Показывать как "Reality Check" индикатор на карточке страны:
    "⚠️ Официальные данные могут быть завышены: -18 пунктов разрыва"
    "✅ Данные соответствуют реальному опыту переехавших"

  Источник: собственные user stories + официальные scores
  (снова: не нужны внешние API, считается из наших данных!)

  По мере накопления user stories — метрика становится точнее.
  Это incentive для пользователей добавлять свои истории.


──────────────────────────────────────────────────────────────────────────────
  УНИКАЛЬНАЯ МЕТРИКА 3: Bureaucracy Friction Score (BFS)
──────────────────────────────────────────────────────────────────────────────

ЧТО ИЗМЕРЯЕТ:
  Реальное время и стоимость прохождения ключевых процедур для мигранта.
  Стандартный "Ease of Doing Business" измеряет бизнес-процедуры.
  Мы измеряем процедуры, важные именно для переезжающего человека.

НАБОР ПРОЦЕДУР:
  1. Время получения первичной визы/разрешения на въезд (дни)
  2. Время получения ВНЖ после въезда (дни)
  3. Количество документов, требуемых для ВНЖ
  4. Количество ведомств, которые нужно посетить
  5. Наличие цифровых процедур (онлайн vs офлайн)
  6. Средняя стоимость всего процесса (USD)
  7. Необходимость юридической помощи (да/нет)
  8. Время открытия банковского счёта для нерезидента (дни)

ФОРМУЛА:
  Каждая процедура нормализуется по глобальному min-max.
  BFS = среднее нормализованных значений, инвертированное (меньше = лучше).

КАК ИСПОЛЬЗОВАТЬ В ПРОЕКТЕ:
  Источник: вводится вручную через admin, обновляется редко (раз в год)
  Подтверждается: ссылками на официальные сайты миграционных ведомств

  Показывать как: "Оценка бюрократии: 3.2/10" + breakdown по шагам
  Это конкретно и actionable — пользователь понимает, что его ждёт.


──────────────────────────────────────────────────────────────────────────────
  УНИКАЛЬНАЯ МЕТРИКА 4: Openness Momentum (OM)
──────────────────────────────────────────────────────────────────────────────

ЧТО ИЗМЕРЯЕТ:
  Если Country Drift — это скорость изменения (первая производная),
  то Openness Momentum — это ускорение (вторая производная).

  Аналогия из физики:
    CII score      = позиция
    Country Drift  = скорость
    OM             = ускорение

  Аналогия из финансов: MACD (Moving Average Convergence Divergence)

ФОРМУЛА:
  OM(country, year) = Drift(year) - Drift(year-3)

  OM > 0: страна улучшается и этот процесс ускоряется → "набирает ход"
  OM = 0: страна улучшается, но скорость не меняется → "стабильный рост"
  OM < 0: страна ухудшается и деградация ускоряется → "в свободном падении"

  ОСОБЕННО ЦЕННЫЕ КОМБИНАЦИИ:
    Drift > 0 и OM > 0: страна разгоняется → лучшее время для переезда
    Drift > 0 и OM < 0: страна улучшается, но замедляется → пик достигнут
    Drift < 0 и OM > 0: деградация замедляется → возможный разворот
    Drift < 0 и OM < 0: деградация ускоряется → опасный сигнал

КАК ИСПОЛЬЗОВАТЬ В ПРОЕКТЕ:
  Показывать как дополнительный индикатор рядом с Drift badge:
    🚀 Drift +12 / Momentum ↑ "Разгоняется"
  Или предупреждение:
    ⚠️ Drift +8 / Momentum ↓ "Темп улучшения замедляется"

  Источник: вычисляется из country_cii_scores (собственные данные)


──────────────────────────────────────────────────────────────────────────────
  УНИКАЛЬНАЯ МЕТРИКА 5: Contradiction Score (CS)
──────────────────────────────────────────────────────────────────────────────

ЧТО ИЗМЕРЯЕТ:
  Степень противоречия между разными источниками по одной теме.
  Когда официальный сайт говорит одно, юридическая фирма — другое,
  а user stories показывают третье.

  Высокий Contradiction Score = "данные по этой стране ненадёжны,
  ситуация неоднозначна, рекомендуем перепроверить".

ФОРМУЛА:
  CS(country, topic) =
    std_dev(score_по_разным_источникам_на_одну_тему) / max_возможное_отклонение

  Чем выше стандартное отклонение между источниками → тем выше CS.

  CS = 0.0–0.2  → источники согласуются → высокое доверие
  CS = 0.2–0.5  → умеренное расхождение → проверить детали
  CS > 0.5      → серьёзные противоречия → красный флаг

КАК ИСПОЛЬЗОВАТЬ В ПРОЕКТЕ:
  Источник: наша таблица evidence_items + user_stories
  (поле contradicts_evidence_id, которое мы уже запланировали)

  Показывать как: индикатор на карточке страны
    "⚠️ По налоговой теме обнаружены противоречия в источниках"
    [Посмотреть подробности] → список противоречащих друг другу источников

  Это одна из самых уникальных функций — ни один конкурент не показывает
  пользователю, когда данные противоречат друг другу.


──────────────────────────────────────────────────────────────────────────────
  УНИКАЛЬНАЯ МЕТРИКА 6: Scenario-Specific Risk Score (SSRS)
──────────────────────────────────────────────────────────────────────────────

ЧТО ИЗМЕРЯЕТ:
  Стандартные индексы дают один риск-score для всей страны.
  Но риск для digital nomad принципиально отличается от риска для семьи
  с детьми или для инвестора.

  SSRS — это агрегация только тех legal signals и evidence items,
  которые помечены как affecting конкретную группу.

ФОРМУЛА:
  SSRS(country, scenario) =
    Σ(impact_level × recency_weight) для legal_signals
    где affected_groups CONTAINS scenario_group

  Где recency_weight:
    Сигнал последних 6 месяцев  → weight = 1.0
    Сигнал 6–18 месяцев         → weight = 0.7
    Сигнал 18–36 месяцев        → weight = 0.4
    Сигнал старше 3 лет         → weight = 0.1

КАК ИСПОЛЬЗОВАТЬ В ПРОЕКТЕ:
  Источник: наши legal_signals с полем affected_groups (уже есть)
  Не требует внешних API — считается из наших данных.

  Показывать на decision page:
    "Для вашего сценария (Переезд/ВНЖ): Risk Score 7.2/10"
    "Основные риски: [список релевантных legal signals]"

  Это прямо отвечает на вопрос пользователя "а мне-то насколько рискованно?"


──────────────────────────────────────────────────────────────────────────────
  УНИКАЛЬНАЯ МЕТРИКА 7: Cross-Border Life Compatibility (CBLC)
──────────────────────────────────────────────────────────────────────────────

ЧТО ИЗМЕРЯЕТ:
  Насколько легко поддерживать связи с родной страной после переезда.
  Это вопрос, который не задаёт НИ ОДИН существующий индекс.

  Многие люди переезжают, но хотят: регулярно возвращаться, держать
  счета дома, поддерживать бизнес-связи. Для них совместимость
  "страна происхождения + страна назначения" — критически важна.

КОМПОНЕНТЫ:
  1. Visa-free / easy return (можно ли легко вернуться на родину)
  2. Banking compatibility (можно ли держать счёт в двух странах)
  3. Tax treaty existence (есть ли соглашение об избежании двойного налогообложения)
  4. Flight connectivity (прямые рейсы и частота)
  5. Time zone difference (разница часовых поясов для удалённой работы)
  6. Internet call quality (возможность видеозвонков без ограничений)

ФОРМУЛА:
  CBLC(origin, destination) =
    взвешенное среднее 6 компонентов, нормализованных в 0–100

ОСОБЕННОСТЬ:
  Это бинарная метрика — она зависит от ПАРЫ стран, а не от одной.
  Для каждой пары (origin, destination) — свой CBLC.

КАК ИСПОЛЬЗОВАТЬ В ПРОЕКТЕ:
  Источник: частично из наших данных, частично из Henley + открытых реестров
  Показывать на: decision page, когда пользователь указал origin country
    "Для граждан России → Уругвай: совместимость 62/100"
    "⚠️ Нет прямых рейсов. ✅ Есть налоговое соглашение."

  Это делает decision engine персонализированным — результат зависит
  от того, откуда именно переезжает пользователь.

──────────────────────────────────────────────────────────────────────────────
  3.1 ПРИНЦИП СЛОЁНОГО ВНЕДРЕНИЯ
──────────────────────────────────────────────────────────────────────────────

Не внедрять всё сразу. Порядок важен:

  СЛОЙ 0 — Данные (приоритет абсолютный):
    Без данных нет метрик. Без метрик нет визуала.
    Сначала — наполнить БД хотя бы базовыми значениями вручную.
    Только потом строить визуализации.

  СЛОЙ 1 — Стандартные метрики (CII из внешних индексов):
    Political Stability, Rule of Law, Economic Freedom, HDI, Affordability
    Источники: World Bank, UNDP, Heritage — всё бесплатно.
    Это то, что делает CII достоверным и сравнимым.

  СЛОЙ 2 — Специфические метрики платформы:
    Residency Accessibility (вручную), Country Drift (вычисляемое),
    Legal Velocity Index (из наших legal signals).
    Не требуют внешних API — считаются из наших данных.

  СЛОЙ 3 — Уникальные метрики (наш differentiator):
    Expat Reality Gap, Contradiction Score, SSRS, CBLC.
    Они появляются по мере накопления user stories и evidence.
    Запускать только когда есть достаточно данных для расчёта.

  СЛОЙ 4 — AI-enhanced метрики (будущее):
    Автоматическое обнаружение противоречий, NLP sentiment из форумов,
    автоматическая классификация legal signals по affected groups.

  3.2 СХЕМА ТАБЛИЦ ДЛЯ ВСЕЙ МЕТРИЧЕСКОЙ СИСТЕМЫ
──────────────────────────────────────────────────────────────────────────────

  -- Реестр всех метрик (расширяемый без ALTER TABLE)
  CREATE TABLE cii_metric_definitions (
    id               UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    slug             VARCHAR UNIQUE NOT NULL,   -- 'political_stability'
    name_en          VARCHAR NOT NULL,
    name_ru          VARCHAR,
    description_en   TEXT,
    category         VARCHAR,   -- 'stability', 'economy', 'lifestyle', 'unique'
    metric_type      VARCHAR,   -- 'standard', 'computed', 'unique_platform'
    higher_is_better BOOLEAN DEFAULT true,
    value_min        NUMERIC DEFAULT 0,
    value_max        NUMERIC DEFAULT 100,
    data_source      VARCHAR,   -- 'world_bank_wgi', 'computed', 'manual'
    update_frequency VARCHAR,   -- 'annual', 'quarterly', 'real_time'
    is_active        BOOLEAN DEFAULT true,
    display_order    INT,
    introduced_in    VARCHAR,   -- 'v1.0', 'v1.1' — в какой версии добавлена
    created_at       TIMESTAMPTZ DEFAULT now()
  );

  -- Значения метрик по странам и годам
  CREATE TABLE country_metric_values (
    id               UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    country_id       UUID REFERENCES countries(id) ON DELETE RESTRICT,
    metric_id        UUID REFERENCES cii_metric_definitions(id),
    year             INT NOT NULL,
    value            NUMERIC(7,3),   -- нормализованное 0–100
    raw_value        NUMERIC(10,4),  -- оригинальное из источника
    raw_unit         VARCHAR,        -- '%', 'score', 'days', 'USD'
    freshness_score  NUMERIC(3,2) DEFAULT 1.0,
    reliability_score NUMERIC(3,2) DEFAULT 1.0,
    coverage_score   NUMERIC(3,2) DEFAULT 1.0,
    confidence       NUMERIC(3,2) GENERATED ALWAYS AS
                       (freshness_score * reliability_score * coverage_score) STORED,
    source_id        UUID REFERENCES sources(id),
    ingestion_method VARCHAR,  -- 'api', 'manual', 'computed', 'csv_import'
    ingested_at      TIMESTAMPTZ DEFAULT now(),
    ingested_by      VARCHAR,  -- 'world_bank_ingester', 'admin:user_name'
    notes            TEXT,
    deleted_at       TIMESTAMPTZ NULL,
    UNIQUE (country_id, metric_id, year)
  );

  -- Веса метрик по сценариям (разные веса для разных сценариев)
  CREATE TABLE scenario_metric_weights (
    id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    scenario_id  UUID REFERENCES scenarios(id),
    metric_id    UUID REFERENCES cii_metric_definitions(id),
    weight       NUMERIC(4,3) NOT NULL,  -- 0.000–1.000
    formula_version VARCHAR NOT NULL,   -- 'v1.0', 'v1.1'
    effective_from DATE NOT NULL,
    effective_to   DATE NULL,           -- NULL = текущая версия
    created_at     TIMESTAMPTZ DEFAULT now(),
    UNIQUE (scenario_id, metric_id, formula_version)
  );

  -- Итоговые CII scores (кеш агрегированных значений)
  CREATE TABLE country_cii_scores (
    id                UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    country_id        UUID REFERENCES countries(id),
    scenario_id       UUID REFERENCES scenarios(id) NULL, -- NULL = общий
    year              INT NOT NULL,
    cii_score         NUMERIC(5,2),   -- итоговый 0–100
    cii_confidence    NUMERIC(3,2),   -- агрегированный confidence
    country_drift     NUMERIC(6,2),   -- delta vs year-3
    drift_acceleration NUMERIC(6,2),  -- delta drift vs предыдущий период
    formula_version   VARCHAR,
    calculated_at     TIMESTAMPTZ DEFAULT now(),
    UNIQUE (country_id, scenario_id, year)
  );

  -- Уникальные метрики платформы (хранятся отдельно)
  CREATE TABLE country_platform_metrics (
    id                     UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    country_id             UUID REFERENCES countries(id),
    year                   INT,
    legal_velocity_index   NUMERIC(5,3),  -- LVI
    contradiction_score    NUMERIC(4,3),  -- CS
    bureaucracy_friction   NUMERIC(5,2),  -- BFS
    openness_momentum      NUMERIC(6,2),  -- OM
    calculated_at          TIMESTAMPTZ DEFAULT now()
  );

  -- CBLC — для пар стран
  CREATE TABLE country_pair_compatibility (
    id               UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    origin_id        UUID REFERENCES countries(id),
    destination_id   UUID REFERENCES countries(id),
    year             INT,
    cblc_score       NUMERIC(5,2),   -- 0–100
    tax_treaty       BOOLEAN,
    direct_flights   BOOLEAN,
    timezone_diff_h  INT,
    banking_compat   VARCHAR,  -- 'easy', 'moderate', 'hard'
    notes            TEXT,
    source_id        UUID REFERENCES sources(id),
    updated_at       TIMESTAMPTZ DEFAULT now(),
    UNIQUE (origin_id, destination_id, year)
  );

  3.4 API ЭНДПОИНТЫ ДЛЯ ВИЗУАЛЬНОЙ СИСТЕМЫ
──────────────────────────────────────────────────────────────────────────────

  GET /api/v1/countries/{slug}/cii
    → Все метрики CII для страны (текущий год)
    → Spider chart данные: [{metric_slug, value, confidence, label}]
    → Country Drift + Acceleration
    → Scenario scores (все 5 сценариев)

  GET /api/v1/countries/{slug}/cii/history?years=5
    → Исторические данные для Drift Timeline
    → [{year, cii_score, drift, metrics: {...}}]

  GET /api/v1/countries/compare?
        countries=russia,uruguay&scenario=relocation_residence
    → CII данные для группы стран в одном запросе
    → Для spider overlay и grouped bar chart

  GET /api/v1/world-map?scenario=relocation_residence&year=2026
    → Лёгкий ответ: [{iso2, cii_score, drift}] для всех стран с данными
    → Специально оптимизирован для карты (только необходимые поля)

  GET /api/v1/countries/matrix?
        countries=russia,uruguay,georgia&scenarios=all
    → Матрица для heatmap: {[country_slug]: {[scenario_slug]: score}}

  GET /api/v1/metrics/definitions
    → Список активных метрик с названиями, category, display_order
    → Frontend использует для динамического построения chart осей

  GET /api/v1/countries/{slug}/platform-metrics
    → Уникальные метрики: LVI, ERG, CS, BFS, OM

  3.5 ПОРЯДОК ВНЕДРЕНИЯ УНИКАЛЬНЫХ МЕТРИК

  ЭТАП 1 (MVP, сейчас):
    ✓ Legal Velocity Index — считается из наших legal_signal_events
      Минимум данных: 10+ legal signals для страны
      Реализация: 1–2 дня backend

    ✓ Scenario-Specific Risk Score — из наших legal_signals
      Зависит от: поля affected_groups (уже есть в модели)
      Реализация: 2–3 дня backend + 1 день frontend

    ✓ Contradiction Score — из наших evidence_items
      Нужно: добавить поле contradicts_evidence_id в evidence_items
      Реализация: 2 дня backend

  ЭТАП 2 (после накопления 50+ user stories):
    ✓ Expat Reality Gap
      Зависит от: достаточного количества user stories с ratings
      Нужно: добавить поля dimension_ratings в user_stories
      Реализация: 3–4 дня

    ✓ Openness Momentum
      Зависит от: наличия данных за 3+ года
      Реализация: 1 день (уже есть Drift, нужна вторая производная)

  ЭТАП 3 (после добавления 5+ стран):
    ✓ Cross-Border Life Compatibility
      Зависит от: данных по парам стран
      Нужно: заполнить country_pair_compatibility вручную для топ-пар
      Реализация: 3 дня backend + 2 дня frontend

  ЭТАП 4 (будущее, с AI):
    ✓ Bureaucracy Friction Score (детальная версия с AI-валидацией)
    ✓ NLP-enhanced Reality Gap (из форумов и отзывов)

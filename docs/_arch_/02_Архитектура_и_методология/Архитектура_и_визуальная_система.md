  РАЗДЕЛ 0. КОНТЕКСТ И ПРОБЛЕМА, КОТОРУЮ МЫ РЕШАЕМ
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Сейчас пользователь приходит в приложение и видит: текст, карточки, описания,
числа в таблицах. Он вынужден читать, чтобы понять. Это создаёт высокую
когнитивную нагрузку и замедляет принятие решения.

Цель этого документа — описать систему, при которой:

  → Пользователь сначала ВИДИТ (карта, паук, числа, цвет)
  → Затем ПОНИМАЕТ (сравнение, динамика, breakdown)
  → Затем при необходимости УГЛУБЛЯЕТСЯ (текст, источники, evidence)

Этот принцип называется Progressive Disclosure. Он пронизывает всю
архитектуру, которая описана ниже.

  РАЗДЕЛ 1. ВИЗУАЛЬНАЯ КОНЦЕПЦИЯ — ЧТО СТРОИМ
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1.1 ГЛАВНАЯ СТРАНИЦА — "МИР С ВЫСОТЫ"
──────────────────────────────────────

Первое, что видит пользователь — интерактивная карта мира (choropleth map),
где страны окрашены по общему score для выбранного сценария:
зелёный = высокий score, красный = низкий, серый = нет данных.

Над картой — выбор сценария: Переезд / ВНЖ / Бизнес / Безопасность и т.д.
При смене сценария карта перекрашивается в реальном времени.

Под картой — три блока:
  • TOP-5 стран для выбранного сценария
  • WORST-5 стран
  • BIGGEST MOVERS — страны с наибольшим Country Drift за последний год

Технология: react-simple-maps + GeoJSON + D3.js для раскраски.


1.2 КАРТОЧКА СТРАНЫ — "COUNTRY INTELLIGENCE DASHBOARD"
──────────────────────────────────────────────────────

Карточка страны полностью переосмысливается. Вместо текстовых секций —
визуальный дашборд, разделённый на уровни.

  └─────────────────────────────────────────────┘

ВТОРОЙ ЭКРАН (после прокрутки):
  • Country Drift Timeline — sparkline за 5 лет по каждому измерению
  • Legal Signals — последние 3, с цветовой кодировкой 🔴🟡🟢
  • Evidence Summary — краткая сводка доказательной базы

ТРЕТИЙ ЭКРАН (по клику "Читать подробнее"):
  • Полный текстовый профиль
  • Все источники и evidence
  • User stories
  • Полный список legal signals


1.3 СТРАНИЦА DECISION — "СРАВНЕНИЕ В РЕАЛЬНОМ ВРЕМЕНИ"
──────────────────────────────────────────────────────

Decision page перестаёт быть формой с результатом.
Она становится живым сравнительным дашбордом:

  • Два spider chart наложены друг на друга (синий vs оранжевый)
  • Grouped bar chart по всем измерениям рядом для двух стран
  • Цветной winner badge для каждого измерения
  • Блок Strengths / Weaknesses / Risk Warnings
  • Источники по каждому выводу (кликабельные)

Возможность добавить третью страну для сравнения — P2 приоритет.


1.4 LEGAL SIGNALS — "POLICY TIMELINE"
──────────────────────────────────────

Вместо списка — хронологическая вертикальная лента:

  2026  🔴  Закон об ограничении зарубежных счетов
            Impact: высокий | Affects: мигранты, инвесторы
            [Source: ФНС РФ → ссылка]

  2025  🟡  Изменение порядка ВНЖ для фрилансеров

            Impact: позитивный | Affects: предприниматели

Фильтры: по impact direction, по affected group, по году, по типу сигнала.


1.5 СТРАНИЦА COMPARE — "МАТРИЦА СТРАН"
───────────────────────────────────────

Таблица-heatmap: строки = страны, столбцы = сценарии.
Каждая ячейка окрашена по score. Клик на ячейку → детальное сравнение.

           Переезд  Бизнес  Бюджет  Безопасность
Russia        42      34      61        22
Uruguay       74      65      58        79
Georgia       81      88      70        65
Portugal      69      72      45        80

Это позволяет пользователю охватить взглядом весь выбор за секунды.

  РАЗДЕЛ 2. COUNTRY INTELLIGENCE INDEX (CII) — СИСТЕМА МЕТРИК
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

CII — единая система числовых измерений, которая питает все визуализации.
Рассчитывается для каждой страны. Все charts, карта и comparisons
берут данные из одного источника — таблицы metric values.


2.1 ВОСЕМЬ БАЗОВЫХ ИЗМЕРЕНИЙ
──────────────────────────────

  #  Измерение                Что измеряет                    Диапазон
  ─────────────────────────────────────────────────────────────────────
  1  Political Stability      Стабильность, риск конфликтов   0–100
  2  Rule of Law              Суды, коррупция, соблюд. законов 0–100
  3  Economic Freedom         Бизнес, налоги, барьеры         0–100
  4  Cost of Living Index     Стоимость жизни (100 = baseline) 0–200
  5  Residency Friendliness   Простота ВНЖ/ПМЖ/виз            0–100
  6  Quality of Life          Медицина, образование, HDI       0–100
  7  Digital & Remote Work    Интернет, nomad infrastructure   0–100
  8  Country Drift            Куда движется страна             -100…+100

Country Drift — главный уникальный показатель платформы.
Отрицательный Drift = страна ухудшается (закрывается, деградирует).
Положительный Drift = страна улучшается (реформы, открытость).
Этого нет ни у одного конкурента.


2.2 ИСТОЧНИКИ ДАННЫХ И АВТОПОДГРУЗКА
──────────────────────────────────────

Все источники имеют публичные API или открытые датасеты:

  Измерение              Источник                         Тип доступа
  ──────────────────────────────────────────────────────────────────────
  Political Stability    World Bank WGI                   Бесплатный API
                         Global Peace Index (IEP)         CSV-датасет
  Rule of Law            World Bank WGI (Rule of Law)     Бесплатный API
                         Transparency Intl CPI            CSV ежегодно
                         Freedom House                    Dataset
  Economic Freedom       Heritage Foundation EFI          CSV публичный
                         World Bank Business Ready        Бесплатный API
  Cost of Living         Numbeo                           Платный API ~$50/мес
                         World Bank PPP                   Бесплатный API
  Residency Friendly     MIPEX                            Dataset
                         Henley Passport Index            CSV
                         Внутренние legal signals         БД проекта
  Quality of Life        UNDP HDI                         Бесплатный API
                         WHO Health Index                 Dataset
  Digital/Remote Work    ITU Digital Development          Dataset
                         Ookla Speedtest Global           CSV
  Country Drift          Внутренний расчёт из WGI         Вычисляемое поле
                         + динамика legal signals

Ключевые API-эндпоинты:

  World Bank:
    https://api.worldbank.org/v2/country/{iso2}/indicator/{code}?format=json

  UNDP HDI:
    https://hdrdata.org/api/composite/4

  Numbeo:
    https://www.numbeo.com/api/ (требует API key)

РЕКОМЕНДАЦИЯ: создать DataIngestionService, который раз в квартал
подтягивает данные из этих источников, нормализует в диапазон 0–100
и сохраняет в БД с версионированием и указанием источника.


2.3 ФОРМУЛА COUNTRY DRIFT
──────────────────────────

  Drift(country, year) =
    Σ( weight_i × (metric_i(year) - metric_i(year-3)) ) / Σ(weight_i)

  Где year-3 используется для сглаживания краткосрочного шума.

  Веса по умолчанию:
    Political Stability    0.25
    Rule of Law            0.25
    Economic Freedom       0.20
    Residency Friendliness 0.15
    Quality of Life        0.10
    Digital Infrastructure 0.05

  Интерпретация результата:
    +15…+100  Активно улучшается   🚀 Зелёная стрелка вверх
    +5…+15    Умеренное улучшение  ↗ Светло-зелёная
    -5…+5     Стагнация            → Серая
    -15…-5    Умеренное ухудшение  ↘ Жёлтая
    -100…-15  Активно деградирует  🔻 Красная стрелка вниз

  РАЗДЕЛ 3. АРХИТЕКТУРНЫЕ РЕКОМЕНДАЦИИ — КАК СТРОИТЬ ПРАВИЛЬНО
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

3.1 METRIC REGISTRY PATTERN — РАСШИРЯЕМЫЕ МЕТРИКИ БЕЗ МИГРАЦИЙ
────────────────────────────────────────────────────────────────

ПРОБЛЕМА ФИКСИРОВАННЫХ КОЛОНОК:
  Если хранить метрики как отдельные колонки в таблице:
    political_stability NUMERIC,
    rule_of_law         NUMERIC,
    economic_freedom    NUMERIC,
    ...
  — то каждая новая метрика требует ALTER TABLE + миграцию БД + изменение
  кода репозитория, схем, типов TypeScript. Это дорого и опасно.

ПРАВИЛЬНОЕ РЕШЕНИЕ — две таблицы:

  Таблица 1: cii_metric_definitions (реестр метрик)
  ───────────────────────────────────────────────────
  slug            VARCHAR UNIQUE    -- 'political_stability', 'rule_of_law'
  name_en         VARCHAR           -- человекочитаемое название
  name_ru         VARCHAR
  description_en  TEXT
  category        VARCHAR           -- 'stability', 'economy', 'lifestyle'
  weight          NUMERIC           -- вес в итоговом CII score
  value_min       NUMERIC           -- нижняя граница (обычно 0)
  value_max       NUMERIC           -- верхняя граница (обычно 100)
  higher_is_better BOOLEAN          -- для инверсии (Cost of Living)
  data_source     VARCHAR           -- 'world_bank_wgi', 'undp_hdi'
  update_frequency VARCHAR          -- 'annual', 'quarterly'
  is_active       BOOLEAN DEFAULT true
  display_order   INT               -- порядок осей на spider chart
  created_at      TIMESTAMPTZ
  updated_at      TIMESTAMPTZ

  Таблица 2: country_metric_values (значения)
  ─────────────────────────────────────────────
  id          UUID PRIMARY KEY
  country_id  UUID REFERENCES countries(id)
  metric_id   UUID REFERENCES cii_metric_definitions(id)
  year        INT
  value       NUMERIC           -- нормализованное значение
  raw_value   NUMERIC           -- оригинальное значение из источника
  confidence  NUMERIC           -- 0.0–1.0
  source_id   UUID              -- откуда взято
  ingested_at TIMESTAMPTZ
  UNIQUE(country_id, metric_id, year)

ЧТО ЭТО ДАЁТ:
  + Добавить новую метрику = 1 INSERT в cii_metric_definitions
  + Нет ALTER TABLE, нет миграций схемы
  + Spider chart читает оси динамически из API
  + Старые данные не ломаются при добавлении новых метрик
  + Деактивировать метрику = UPDATE is_active = false
  + Изменить вес метрики = UPDATE weight (пересчёт scores автоматически)

ВАЖНО ДЛЯ ФРОНТЕНДА:
  Компоненты Spider chart, bar chart, compare matrix должны принимать
  массив метрик из API-ответа, а не иметь захардкоженные оси.
  Тогда при добавлении 9-й метрики фронтенд обновится автоматически.


3.2 ВЕРСИОНИРОВАНИЕ ФОРМУЛ И ВЕСОВ
────────────────────────────────────

ПРОБЛЕМА:
  Если в 2027 году изменить вес political_stability с 0.25 на 0.20 —
  исторические scores станут несравнимы с текущими. График Drift
  за 5 лет окажется некорректным.

РЕШЕНИЕ — таблица снапшотов формул:

  cii_formula_versions
  ─────────────────────
  id            UUID PRIMARY KEY
  version_tag   VARCHAR           -- 'v1.0', 'v1.1', 'v2.0'
  description   TEXT              -- что изменилось и почему
  weights_json  JSONB             -- снапшот всех весов на момент публикации
  effective_from DATE
  effective_to   DATE NULL        -- NULL = текущая версия
  created_by    VARCHAR
  created_at    TIMESTAMPTZ

  При пересчёте scores всегда записывать formula_version_id,
  по которой они рассчитаны.

  Тогда при сравнении исторических данных система знает,
  по какой формуле они считались, и может либо показать
  предупреждение, либо пересчитать по единой методологии.


3.3 ВЕРСИОНИРОВАНИЕ ВНЕШНИХ ДАННЫХ
────────────────────────────────────

  external_index_snapshots
  ─────────────────────────
  id              UUID PRIMARY KEY
  source_slug     VARCHAR           -- 'world_bank_wgi', 'transparency_cpi'
  country_id      UUID
  indicator_code  VARCHAR           -- 'PV.EST', 'RL.EST' и т.д.
  year            INT
  raw_value       NUMERIC           -- как пришло из API
  unit            VARCHAR           -- '%', 'score', 'index'
  fetched_at      TIMESTAMPTZ       -- когда подтянули
  source_url      VARCHAR           -- точный URL запроса
  api_version     VARCHAR           -- версия API на момент запроса

  ВАЖНО: никогда не перезаписывать raw данные — только добавлять
  новые строки. Это позволяет отследить, как менялись официальные
  данные с течением времени (ретроактивные правки индексов — нередкое явление).


3.4 SOFT DELETE ВЕЗДЕ
──────────────────────

Во всех значимых таблицах добавить поле:

  deleted_at TIMESTAMPTZ NULL DEFAULT NULL

  Удаление = UPDATE deleted_at = NOW()
  Выборка = WHERE deleted_at IS NULL

ПОЧЕМУ ЭТО ВАЖНО:
  Когда появятся внешние партнёры, B2B API, marketplace — физическое
  удаление сломает исторические ссылки. Также важно для audit trail:
  нужно знать, что было удалено и когда.

  Применить к: countries, legal_signals, sources, evidence_items,
  user_stories, metric_definitions, country_profiles.


3.5 EVENT SOURCING ДЛЯ LEGAL SIGNALS
──────────────────────────────────────

ПРОБЛЕМА:
  Legal signals меняются со временем. Закон принят → изменён → отменён.
  Если обновлять одну запись через UPDATE — история теряется.

РЕШЕНИЕ:
  Хранить историю изменений как цепочку событий:

  legal_signal_events
  ────────────────────
  id              UUID PRIMARY KEY
  signal_id       UUID REFERENCES legal_signals(id)
  event_type      VARCHAR     -- 'created', 'amended', 'repealed', 'superseded'
  event_date      DATE        -- когда произошло изменение в реальном мире
  description     TEXT        -- что именно изменилось
  source_id       UUID        -- источник, подтверждающий это изменение
  recorded_at     TIMESTAMPTZ -- когда мы это зафиксировали
  recorded_by     VARCHAR

  Таблица legal_signals хранит ТЕКУЩЕЕ состояние.
  Таблица legal_signal_events хранит ПОЛНУЮ ИСТОРИЮ.

  Это позволяет строить Policy Timeline с реальной историей изменений,
  а не только текущий статус.


3.6 DOMAIN SEPARATION — НЕ СМЕШИВАТЬ ДОМЕНЫ
─────────────────────────────────────────────

Когда появятся новые функции, важно держать их как отдельные домены:

  /apps/api/app/domains/
    countries/          -- страны, профили, карточки
    metrics/            -- CII, метрики, external indices
    decisions/          -- decision engine, scenarios, runs
    legal/              -- legal signals, events, timeline
    content/            -- sources, evidence, user stories
    translations/       -- translation units, jobs, variants
    users/              -- аккаунты (когда появятся)
    watchlist/          -- подписки и алерты (отдельный домен)
    marketplace/        -- консультанты (отдельный домен)
    notifications/      -- emails, alerts (отдельный домен)

  ПРАВИЛО: домены не должны напрямую импортировать друг друга.
  Взаимодействие — через чёткие интерфейсы и сервисы.
  Это позволит в будущем при необходимости вынести домен
  в отдельный микросервис без переписывания всего.

4.1 ДАННЫЕ И КОНТЕНТ
─────────────────────

• УСТАРЕВАНИЕ КОНТЕНТА
  Юридические данные устаревают. Добавить в таблицы:
    reviewed_at   TIMESTAMPTZ   -- когда последний раз проверялось
    expires_at    TIMESTAMPTZ   -- рекомендуемая дата следующей проверки
    staleness_level VARCHAR     -- 'fresh', 'aging', 'stale', 'critical'
  Нужна система автоматических напоминаний редакторам.
  Контент старше 12 месяцев по юридическим вопросам = критично устарел.

• ОРИГИНАЛЬНЫЙ ЯЗЫК ИСТОЧНИКА
  Источник на португальском — НЕ то же самое, что его машинный перевод.
  Никогда не заменять оригинал переводом. Хранить оба.
  Поле source_language_original обязательно.

• РЕТРОАКТИВНЫЕ ПРАВКИ ИНДЕКСОВ
  World Bank и другие провайдеры иногда задним числом меняют исторические
  данные. Нельзя просто перезаписывать raw values — нужно сохранять
  дату получения данных (fetched_at) и всю историю запросов.

• CONFIDENCE НА АГРЕГИРОВАННОМ УРОВНЕ
  Если одно из восьми измерений CII имеет низкий confidence —
  итоговый CII score должен это отражать. Нельзя усреднять confidence:
  итоговый confidence = min(confidence всех измерений) или взвешенная формула.

• SYNTHETIC DATA МАРКИРОВКА
  Сейчас user stories помечены is_synthetic. В будущем это важно сохранять
  и для других видов данных: AI-generated summaries, estimated values,
  interpolated data. Никогда не смешивать synthetic и verified контент
  без явной маркировки.


4.2 ПОЛЬЗОВАТЕЛИ И ПЕРСОНАЛИЗАЦИЯ
───────────────────────────────────

• USER CONTEXT ВЛИЯЕТ НА ВСЁ
  Когда появятся аккаунты, релевантность данных кардинально изменится.
  Страна для одинокого фрилансера и для семьи с детьми — совершенно разные.
  Заложить в архитектуру: decision engine должен принимать user_context
  как параметр (гражданство, семейный статус, доход, профессия),
  а не только scenario и locale.

• ИСТОРИЯ ЗАПРОСОВ — СОБИРАТЬ С ПЕРВОГО ДНЯ
  Какие сценарии пользователь запускал, какие страны смотрел, что сравнивал.
  Это топливо для персонализации и продуктовой аналитики.
  Если не собирать сейчас — потом не восстановить.
  Даже без аккаунтов можно собирать анонимно через session_id.

  decision_run_log (или analytics_events)
  ─────────────────────────────────────────
  session_id    VARCHAR
  scenario_slug VARCHAR
  countries     JSONB
  locale        VARCHAR
  result_hash   VARCHAR   -- для деперсонализированного трекинга
  created_at    TIMESTAMPTZ

• WATCHLIST И АЛЕРТЫ — ОТДЕЛЬНЫЙ ДОМЕН
  Не навешивать на existing таблицы. Watchlist — это:
    user_id + country_id + alert_conditions + delivery_method
  Это отдельный сервис с собственной логикой и инфраструктурой.


4.3 ПЕРЕВОДЫ И МУЛЬТИЯЗЫЧНОСТЬ
────────────────────────────────

• TRANSLATION GLOSSARY
  Когда появятся 5–10 языков, одни термины будут переводиться по-разному
  в разных местах интерфейса. Нужен единый glossary утверждённых переводов:
    glossary_terms: term_en, term_ru, term_es, context, domain
  Особенно критично для юридических терминов:
  ВНЖ, ПМЖ, tax residency, beneficial owner — они должны переводиться
  одинаково везде в приложении.

• TRANSLATION MEMORY
  Если строка изменилась в оригинале — перевод становится outdated.
  Нужен механизм source_hash: хранить хэш оригинального текста.
  При изменении оригинала — пересчитать хэш, сравнить, пометить
  перевод как stale. Это уже частично реализовано, не ломать.

• LOCALE В URL
  Сейчас: /countries/russia
  Нужно думать сейчас: /ru/countries/russia  или  /countries/russia?locale=ru
  Переделать роутинг после запуска дорого и влияет на SEO.
  Решение принять ДО начала масштабирования языков.

• ЮРИДИЧЕСКИ ВАЖНЫЙ КОНТЕНТ
  Machine translation → published truth — НЕЛЬЗЯ.
  Для legal signals, country profiles, legal descriptions обязателен:
    original source → summary → machine draft → human review → published
  Нельзя автоматически публиковать машинный перевод юридического текста.


4.4 МАСШТАБИРОВАНИЕ И ИНФРАСТРУКТУРА
──────────────────────────────────────

• КЕШИРОВАНИЕ ДАННЫХ
  Данные по странам меняются редко — раз в квартал или реже.
  При 50+ странах и 20+ языках отдавать их из PostgreSQL при каждом
  запросе — дорого и медленно. Нужна стратегия кеширования:
    - Redis для горячих данных (country card, scores)
    - CDN для статически генерируемых страниц
  Важно: при обновлении данных инвалидировать кеш корректно,
  иначе пользователи будут видеть устаревшие данные.

• ПОИСК — НЕ ВСТРАИВАТЬ ГЛУБОКО
  Сейчас поиска нет. Когда стран станет 50+, legal signals — тысячи,
  понадобится полнотекстовый поиск. PostgreSQL FTS справится на MVP.
  НО: не встраивать поисковую логику глубоко в бизнес-код.
  Поиск — отдельный слой, который можно заменить на Elasticsearch
  или Typesense без переписывания domain logic.

• RATE LIMITING ДЛЯ API
  Когда появится B2B доступ, понадобится rate limiting.
  Заложить сейчас через абстракцию: все запросы проходят через
  единый API Gateway, где можно навесить throttling без изменения
  бизнес-логики.

• ЛОКАЛИЗОВАННЫЕ URL ДЛЯ SEO
  /countries/russia плохо ранжируется в поиске для русскоязычных.
  В будущем: /ru/страны/россия — это серьёзный SEO-выигрыш.
  Решение: сейчас использовать slug-based роутинг с locale prefix,
  который легко настроить в Next.js через internationalized routing.


4.5 МОНЕТИЗАЦИЯ И B2B
──────────────────────

• FEATURE FLAGS С ПЕРВОГО ДНЯ
  Что бесплатно, что платно — это решение влияет на архитектуру.
  Добавить абстракцию FeatureFlag или AccessTier как можно раньше.
  Не хардкодить логику "показывать или нет" в компонентах.
  Пример правильной архитектуры:
    can_access(user, feature='country_drift') → boolean
  Логика внутри этой функции может быть любой — freemium, subscription,
  trial — без изменения frontend компонентов.

• B2B API — ОТДЕЛЬНЫЙ КОНТРАКТ
  Внутренний API платформы и внешний B2B API должны быть разными.
  Внутренний может меняться быстро. Внешний требует стабильности,
  версионирования и SLA. Не открывать внутренний API напрямую партнёрам.

• MARKETPLACE КАК ОТДЕЛЬНЫЙ ДОМЕН
  Когда появится marketplace консультантов — это совершенно другая
  бизнес-логика: репутация, транзакции, отзывы, escrow, disputes.
  Не смешивать с данными о странах. Отдельный домен, отдельные таблицы,
  возможно — отдельное приложение.

• AUDIT ДЛЯ МОНЕТИЗАЦИИ
  Все действия, связанные с деньгами, должны быть неизменяемыми.
  Финансовые audit logs — отдельная таблица, только INSERT, никогда UPDATE.


4.6 КАЧЕСТВО ДАННЫХ И ДОВЕРИЕ
───────────────────────────────

• CONTRADICTION DETECTOR — УНИКАЛЬНАЯ ФУНКЦИЯ
  Когда официальный источник говорит одно, а практика — другое,
  система должна это обнаруживать и показывать пользователю.
  Заложить в evidence модель поле:
    contradicts_evidence_id UUID NULL
    contradiction_note TEXT NULL
  Это позволит в будущем строить автоматическую детекцию противоречий.

• CONFIDENCE НА КАЖДОМ УРОВНЕ
  Confidence должен быть не только у sources, но и у:
    - отдельных claim в evidence
    - legal signal в целом
    - scenario score
    - итогового CII score
  И агрегироваться снизу вверх — итоговый score не может быть
  высокой уверенности, если хотя бы одно измерение с низким confidence.

• DATA FRESHNESS ИНДИКАТОРЫ
  Пользователь должен всегда видеть, насколько свежи данные.
  Не скрывать устаревший контент — показывать с явным предупреждением.
  last_verified_at на всех публичных данных обязателен.

  РАЗДЕЛ 6. ЧТО ТОЧНО НЕ ДЕЛАТЬ (АНТИПАТТЕРНЫ)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  ✗ Не хранить метрики как фиксированные колонки в таблице
    → Использовать Metric Registry Pattern (раздел 3.1)

  ✗ Не перезаписывать raw данные из внешних источников
    → Только INSERT новых строк с fetched_at

  ✗ Не делать физическое удаление записей
    → Только soft delete через deleted_at

  ✗ Не публиковать машинный перевод юридического контента автоматически
    → Обязательный human review для legal content

  ✗ Не встраивать поисковую логику в бизнес-код
    → Поиск как отдельный заменяемый слой

  ✗ Не смешивать домены: countries + marketplace + notifications
    → Отдельные домены с чёткими границами

  ✗ Не хардкодить оси Spider chart во фронтенде
    → Динамическое чтение метрик из API

  ✗ Не открывать внутренний API напрямую партнёрам
    → Отдельный B2B API контракт

  ✗ Не усреднять confidence бездумно
    → Итоговый confidence = взвешенный минимум, не среднее

  ✗ Не начинать Marketplace, пока не устойчив core product
    → Это отдельный продукт, требует отдельных ресурсов

     Принимаются редакционно (фиксированы командой) или задаются

     Второй вариант — мощная UX-функция, но сложнее в реализации.

  2. COUNTRY DRIFT — КТО СЧИТАЕТ?
     Автоматически из внешних данных (World Bank delta) или
     редактируется вручную через admin с обоснованием?
     Рекомендация: автоматически + возможность editorial override.

  3. NUMBEO — ПЛАТНО ИЛИ НЕТ?
     Numbeo API стоит ~$50/мес. Альтернативы: World Bank PPP (бесплатно,
     менее детально) или ручное внесение данных на старте.

  4. КАРТА МИРА — ВСЕ СТРАНЫ ИЛИ ТОЛЬКО С ДАННЫМИ?
     Показывать все страны серым и постепенно окрашивать по мере
     добавления данных? Или только те, по которым есть данные?

  5. SPIDER CHART — 8 ОСЕЙ ИЛИ 6?
     8 осей = полная картина, но читается хуже на мобильном.
     6 осей = лучше читается, но меньше информации.
     Рекомендация: 6 на мобильном, 8 на десктопе.

  6. LOCALE В URL — РЕШИТЬ СЕЙЧАС
     /countries/russia  vs  /ru/countries/russia
     Это влияет на SEO и роутинг. Переделать после запуска дорого.
     Рекомендация: Next.js internationalized routing с locale prefix.

  7. ПРИОРИТЕТ ПЕРВОГО ЭКРАНА
     Spider chart на карточке страны или WorldMap на главной?
     Что важнее для первого визуального релиза?

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

ЧТО СТРОИМ:
  Единую визуальную систему CII на основе 8 числовых измерений,
  которая пронизывает все страницы приложения — карту мира, spider chart,
  timeline, heatmap сравнения. Пользователь сначала видит, потом читает.

ГЛАВНОЕ УНИКАЛЬНОЕ:
  Country Drift Index — числовое выражение того, куда страна движется.
  Этого нет ни у одного прямого конкурента.

ПРИНЦИП UX:
  Progressive Disclosure — три уровня глубины. Визуал → понимание → текст.
  Каждый переход явный, по клику пользователя.

КЛЮЧЕВЫЕ АРХИТЕКТУРНЫЕ РЕШЕНИЯ:
  • Metric Registry Pattern — метрики в двух таблицах, не в колонках
  • Soft delete везде
  • Event sourcing для legal signals
  • Версионирование формул и raw данных
  • Domain separation — marketplace, watchlist, notifications = отдельные домены
  • Feature flags для монетизации с первого дня
  • История запросов пользователя — собирать с первого дня

ИСТОЧНИКИ ДАННЫХ (автоподгрузка):
  World Bank API (бесплатно) → главный источник
  UNDP HDI (бесплатно)
  Transparency International CPI (CSV, ежегодно)
  Heritage Foundation Economic Freedom (CSV, ежегодно)
  Numbeo (платно, ~$50/мес) — рассмотреть после MVP

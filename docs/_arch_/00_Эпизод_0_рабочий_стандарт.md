# Эпизод 0 — Глобальный рабочий стандарт проекта

Это единый рабочий регламент Country Decision Atlas. Он принят и применяется к каждому следующему эпизоду разработки, начиная с `feat/editorial-publication-foundation-v1`. Эпизод 0 — это регламент, а не кодовый блок: он не добавляет продуктовый код, миграции или схемы, а фиксирует то, как ведётся вся дальнейшая работа.

Цель — чтобы любой разработчик или ИИ-помощник работал предсказуемо и одинаково: один и тот же порядок веток, проверок, слоёв реализации, запретов и формата отчёта.

---

## 0.1. Рабочий процесс и дисциплина Git

Каждый эпизод начинается только от актуального `main`:

```
git checkout main
git pull --ff-only origin main
git checkout -b <branch>
```

Обязательные правила:

- Никогда не работать напрямую в `main`.
- Имя ветки соответствует задаче (`feat/…`, `fix/…`, `chore/…`, `refactor/…`, `test/…`).
- Запись `task_started` в `CHANGELOG.json` — **до** реализации.
- Точечные проверки после крупных этапов.
- Полный quality gate — **перед** merge.
- Запись `task_completed` в `CHANGELOG.json` — **после** проверок.
- Merge только fast-forward, только после зелёного gate.
- `git push` ветки `main`.
- Финальное состояние — на `main`.

`CHANGELOG.json` — append-only, `schema_version: 1`. Его нельзя переписывать; можно только добавлять записи.

## 0.2. Единый quality gate

Перед каждым merge обязаны проходить все проверки:

```
python -m pytest
python -m mypy apps packages scripts tests
python -m ruff check .
python -m ruff format --check .
python -m sqlfluff lint database --dialect postgres
pnpm quality
pnpm web:mvp:check
python -m pre_commit run --all-files
```

Важно: используется `ruff format --check` (а не просто `ruff format`), потому что перед merge нужно доказать, что форматирование уже чистое. Если `pre-commit` что-то изменил — изменения проверяются и коммитятся. Для `pnpm web:mvp:check` нужен поднятый стек (`docker compose up -d api`) с применёнными миграциями, иначе страницы с данными отвалятся по таймауту.

## 0.3. Порядок реализации слоёв

В каждом эпизоде код пишется строго сверху вниз. Проект backend/data-first: сначала данные и контракт, потом UI.

1. `database/migrations/NNN_*.sql`
2. repository — только SQL/доступ к данным
3. service — бизнес-логика
4. schema — контракты Pydantic v2
5. api/v1 router
6. `main.py` include_router (если роутер новый)
7. `contracts/openapi.yaml`
8. `pnpm contracts:generate`
9. проверка сгенерированного TypeScript
10. тонкий frontend
11. backend-тесты
12. E2E-тесты

## 0.4. Жёсткие запреты

Действуют всегда:

- без комментариев в коде, docstring и inline-комментариев;
- без новых README / `.md` / `.txt` в рабочем коде проекта;
- без изменений в `docs` в рамках продуктовых эпизодов;
- не трогать математику CII, веса сценариев, scoring decision engine;
- не ослаблять data-quality и не маскировать проблемы фейковыми данными;
- миграции только последовательные, идемпотентные и чистые по sqlfluff;
- `CHANGELOG.json` — append-only.

> Примечание. Настоящий раздел `docs/_arch_` — временное хранилище и приведён в порядок отдельной задачей по прямому распоряжению владельца проекта. Запрет «не трогать docs» относится к продуктовым эпизодам, а не к этой разовой уборке.

## 0.5. Архитектурные инварианты

Главные решения, которые нельзя случайно нарушить в будущих задачах:

- **Жизненный цикл** контента: `draft → review → published → archived`, плюс `rejected`. Состояние `in_review` не вводится нигде. Применяется к sources, evidence_items, legal_signals, country_cards, country_profiles.
- **Route v1** (Эпизод 2, миграция 027): только описание и флаги пригодности (`yes/no/unknown`). Никакого числового score у маршрута.
- **Persona** (Эпизод 3, миграция 028): модификатор поверх весов сценария, а не отдельная матрица «персона × сценарий». Модификаторы clamp `>= 0`, итоговые веса ренормируются к сумме 1.0.
- **domain_events** (outbox, Эпизод 1, миграция 026): вводится до Kafka. Событие эмитится только при переходе в `published`. `event_key = '{aggregate_type}:{aggregate_id}:{event_type}'` обеспечивает идемпотентность. Историческое/seed-содержимое не backfill-ится в pending-события (без «шторма» уведомлений); seed-строки `notifiable = FALSE`.
- **Drift v1** (Эпизод 5): считается только из `legal_signal_events`, без time-series CII.
- **Ядро** остаётся на FastAPI + PostgreSQL как единый источник правды. Go / Kafka / MongoDB (Эпизод 4, Telegram-алерты) — только асинхронная доставка, не источник правды.
- **Telegram-подписки** (Эпизод 4): резервируют nullable-колонку `web_user_id` (веб-аутентификация ещё не реализована).

Порядок семи эпизодов: 1) editorial-publication-foundation → 2) route-explorer → 3) persona-layer → 4) signal-alerts-telegram → 5) country-direction-drift → 6) search-foundation → 7) ai-summary-contradiction.

## 0.6. Формат финального отчёта

После каждого эпизода исполнитель даёт отчёт в едином формате:

1. Branch workflow — стартовая ветка, feature-ветка, финальная ветка, был ли push.
2. Implemented — что сделано.
3. Database — изменения миграций.
4. Backend — затронутые слои.
5. API / Contracts — изменения OpenAPI и сгенерированного TS.
6. Frontend — затронутые элементы.
7. Tests — добавленные тесты.
8. Quality gates — результаты всех проверок.
9. Data-quality impact — влияние на качество данных.
10. Not changed — что намеренно не трогали.
11. Risks / notes — риски и заметки.
12. Final git state — финальный статус.

Стандарт зафиксирован в `CHANGELOG.json` (`chore/global-project-operating-standard-v0`, 2026-06-24) на зелёной базовой линии `main`.

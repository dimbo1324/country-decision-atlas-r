# План реализации

> Исполнительный план новой линии разработки. Реализованная система (`../01_Продукт/02_Текущее_состояние_системы.md`) — базовая линия; нумерация эпизодов начата заново с 1. Каждый эпизод ведётся по рабочему стандарту (`../00_Рабочий_стандарт.md`): отдельная ветка `feat/<slug>-v1`, миграции продолжают счётчик (046+), новые поверхности за feature-флагами (выключены до acceptance), data-quality пакет с первого дня, полный quality gate перед merge.
>
> Решения владельца, зафиксированные в плане (2026-07-04, полный журнал — `../08_Открытые_вопросы.md`): порог k-анонимности = 20; владелец не выступает «автором №1» (холодный старт — раздел Эпизода 4); синтетические истории вытесняются реальными; монетизация — донаты в будущем (юрисдикция Уругвай), сейчас только фундамент; институт модераторов — по модели `../01_Продукт/03_Модель_прав_и_ролей.md`; премодерация методологий до ~50 авторов (переход при доле отклонений < 10%); лицензии контрибуций — вариант «в»; тестовые страны не удаляются — консервируются демо-набором (Эпизод 5).
>
> **Режим автономной разработки (решение Р-10):** до интеграционного транша ни один эпизод не подключает реальные внешние сервисы — вся линия 1–7 реализуется и тестируется офлайн через fake-швы. Acceptance-критерии эпизодов сформулированы соответственно (например, доставка напоминаний проверяется по delivery log fake-режима).

### Эпизод 1 — `feat/flexible-methodology-v1`: гибкая методология

**Статус.** Реализовано (миграция 046, `services/methodology_config.py`, `services/weight_profiles.py`). Инвентаризация «жёсткости» из §5 базовой линии перенесена в `methodology_parameters`; таблица в `02_Текущее_состояние_системы.md` обновлена.

**Цель.** Убрать пороги интерпретации и продуктовые лимиты из кода в версионируемую конфигурацию; дать пользователю сохраняемые профили весов. Нулевое изменение поведения по умолчанию.

**Модель данных (миграция 046).**

```sql
CREATE TABLE methodology_parameters (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    version TEXT NOT NULL DEFAULT 'v1.0',
    param_key TEXT NOT NULL,          -- 'score_label.weak_below', 'strength.min_score',
                                      -- 'board.max_active_posts', 'flows.k_anonymity', ...
    value_numeric NUMERIC,
    value_json JSONB,
    description TEXT NOT NULL,
    effective_from TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    CONSTRAINT uq_methodology_param UNIQUE (version, param_key)
);
-- сид: инвентаризация «жёсткости» из 02_Текущее_состояние_системы.md §5,
-- значения = текущие константы; flows.k_anonymity = 20

CREATE TABLE user_weight_profiles (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    name TEXT NOT NULL,
    scenario_slug TEXT,               -- NULL = применим к любому сценарию
    weights JSONB NOT NULL,
    is_default BOOLEAN NOT NULL DEFAULT FALSE,
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    CONSTRAINT uq_profile_name UNIQUE (user_id, name)
);
```

**Сервисный слой.** Новый `services/methodology_config.py`: типизированное чтение активной версии с кэшем; отсутствие ключа — жёсткая ошибка (никаких тихих дефолтов вторым источником правды). Потребители (`score_labels`, `decision_engine`, `_recommend`, лимиты доски) переходят на конфигурацию; чистые функции получают параметры аргументами. Новый `services/weight_profiles.py`: CRUD; валидация — переиспользовать `decision_personalization.validate_custom_weights`.

**API.** `GET/POST/PATCH/DELETE /me/weight-profiles`; `decision/run` принимает `weight_profile_id` (взаимоисключимо с inline-весами); все decision-ответы и паспорта несут `methodology_version`; паспорт снапшотит применённые веса (значениями — воспроизводимость). `GET /methodology/parameters` — публичное чтение активной версии (прозрачность).

**Тесты/acceptance.** Регрессия «бит-в-бит»: существующий набор тестов проходит без правок ожиданий. dq: активная версия существует и полна, значения в диапазонах. Acceptance: профиль сохраняется, применяется, паспорт воспроизводит расчёт после смены профиля. **Не делаем:** пользовательские пороги, влияющие на чужие просмотры; UI-редактирование платформенных параметров.

**Объём.** 1 миграция, 2 сервиса, правки 4–5 потребителей, ~25–35 тестов.

### Эпизод 2 — `feat/trip-planner-v1`: планировщик переезда

**Статус.** Реализовано (миграция 047, пакет `services/trip_planner/`, `scripts/dispatch_trip_reminders.py`, зарегистрирован в `dev_tools_scripts_runner.py`).

**Цель.** Личный контур ежедневного использования: план → точки → чек-лист → предупреждения правилами → напоминания в Telegram → экспорт/шаринг. Без карты-UI (визуальный транш).

**Модель данных (миграция 047).**

```sql
CREATE TABLE trips (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    title TEXT NOT NULL,
    scenario_slug TEXT,
    origin_country_id UUID REFERENCES countries(id),
    status TEXT NOT NULL DEFAULT 'draft',            -- draft|active|completed|abandoned
    confidence_tier TEXT NOT NULL DEFAULT 'declared',-- declared|active|confirmed (для Эпизода 6)
    visibility TEXT NOT NULL DEFAULT 'private',      -- private|link
    share_token_hash TEXT UNIQUE,
    share_token_prefix TEXT,
    created_at/updated_at/completed_at TIMESTAMPTZ
);

CREATE TABLE trip_waypoints (
    id UUID PK, trip_id UUID NOT NULL REFERENCES trips(id) ON DELETE CASCADE,
    position INT NOT NULL,
    country_id UUID NOT NULL REFERENCES countries(id),
    city TEXT,
    kind TEXT NOT NULL DEFAULT 'destination',        -- transit|destination|stopover
    planned_from DATE, planned_to DATE,
    notes TEXT,
    CONSTRAINT uq_trip_position UNIQUE (trip_id, position)
);

CREATE TABLE trip_checklist_items (
    id UUID PK, trip_id UUID NOT NULL REFERENCES trips(id) ON DELETE CASCADE,
    waypoint_id UUID REFERENCES trip_waypoints(id) ON DELETE SET NULL,
    title TEXT NOT NULL, description TEXT,
    due_date DATE,
    status TEXT NOT NULL DEFAULT 'todo',             -- todo|in_progress|done|skipped
    origin_kind TEXT NOT NULL DEFAULT 'manual',      -- manual|route_template|author_template
    origin_ref UUID,
    position INT NOT NULL,
    created_at/updated_at TIMESTAMPTZ
);

CREATE TABLE trip_reminders (
    id UUID PK, trip_id UUID NOT NULL REFERENCES trips(id) ON DELETE CASCADE,
    checklist_item_id UUID REFERENCES trip_checklist_items(id) ON DELETE CASCADE,
    remind_at TIMESTAMPTZ NOT NULL,
    channel TEXT NOT NULL DEFAULT 'telegram',
    status TEXT NOT NULL DEFAULT 'scheduled',        -- scheduled|sent|cancelled
    sent_at TIMESTAMPTZ
);

CREATE TABLE trip_annotations (                      -- пользовательские пометки
    id UUID PK, trip_id UUID NOT NULL REFERENCES trips(id) ON DELETE CASCADE,
    waypoint_id UUID REFERENCES trip_waypoints(id) ON DELETE CASCADE,
    kind TEXT NOT NULL,                              -- note|item_to_bring|warning_ack
    body TEXT NOT NULL,
    position INT
);
```

**Сервисный слой — пакет `services/trip_planner/`:** `trips.py` (CRUD, статусные переходы, «я переехал» → крючок Эпизода 6), `checklist.py` (CRUD; импорт шагов из published `route_checklist_items`), `warnings.py` (правило-движок: сегменты плана → `country_pair_compatibility` + `legal_signals` → предупреждения `{code, severity, message, source_ids}`; серьёзность — параметры Эпизода 1; **без AI**), `reminders.py`, `sharing.py` (токен по паттерну паспортов; PII-безопасная проекция по паттерну `_public_post`), `exports.py` (JSON полный; ICS по дедлайнам — stdlib-генерация; GeoJSON точек), `helpers.py`.

**Напоминания — через существующий контур.** Скрипт `scripts/dispatch_trip_reminders.py` (по образцу recompute-скриптов): due-выборка → `domain_events(event_type='trip_reminder_due', event_key='trip_reminder:{id}', notifiable=TRUE)` → relay → Kafka → нотификатор: новый тип события + рендер → Telegram (нет линковки — фиксация в DLQ-лог без падения).

**API (флаг `trip_planner_enabled`).** `GET/POST /me/trips`; `GET/PATCH/DELETE /me/trips/{id}`; waypoints CRUD+reorder; `GET …/warnings`; checklist CRUD + `POST …/checklist/import {route_id}`; reminders POST/DELETE; share POST/DELETE + `GET /trips/shared/{token}`; `GET …/export?format=json|ics|geojson`.

**Интеграции.** Watchlist: страны плана добавляются идемпотентно (алерты drift/сигналов работают на план). `GET …/what-changed` — прокси существующего механизма по странам плана. «Создать план из паспорта решения». AI-кнопки («объясни предупреждение», «черновик чек-листа») — отдельными эндпоинтами через существующий шов, помечены, опциональны.

**Тесты/acceptance.** Unit warnings-движка (матрица: пара с нотой / без пары / сигнал high-impact); PII-фильтр shared-проекции — тест-инвариант на перечень запрещённых полей; ICS-валидность; reminder → payload события. Acceptance: полный цикл «создал план → предупреждение с источником → импорт шагов → напоминание (fake-режим: запись в delivery log) → отметка выполнения → экспорт ICS» без AI. **Не делаем:** карту-UI, совместное редактирование, OAuth-календари, публичный каталог планов.

**Объём.** 1 миграция (5 таблиц), пакет ~7 модулей, 1 скрипт, +1 тип события в нотификаторе, ~50–70 тестов.

### Эпизод 3 — `feat/rights-capabilities-v1`: права и роли v2

**Статус.** Реализовано (миграция 048, `services/capabilities.py`, `require_capability`/`require_capability_or_roles` в `core/rbac.py`, `/admin/capabilities`, `/admin/moderation/actions`). Очереди доски и Q&A переведены на скоуп-способности без изменения поведения существующих ролей; добавлен конфликт-интересов на модераторских действиях доски и автосанкция уровня 1 (авто-скрытие поста при N подтверждённых репортах, параметр `board.auto_hide_report_threshold`).

**Цель.** Ввести гранты способностей и институт модераторов по модели `../01_Продукт/03_Модель_прав_и_ролей.md`, не меняя поведения существующих ролей.

**Модель данных (миграция 048).** `user_capabilities` (см. документ модели прав, §3.2).

**Сервисный слой.** `services/capabilities.py`: `has_capability` (роль-пучок ИЛИ активный грант), зависимость `require_capability(cap)` по образцу `require_roles`; выдача/отзыв с audit-записью; проверка конфликта интересов для moderate-действий (модератор — не сторона объекта). Автосанкция уровня 1: авто-скрытие объекта при N подтверждённых репортов (N — параметр Эпизода 1) с постановкой в очередь пост-ревью.

**API.** Governance (owner): `GET/POST/DELETE /admin/capabilities` (выдача/отзыв, с причиной); `GET /admin/moderation/actions` — лента действий модераторов (фильтры, аномалии). Модераторские очереди уже существуют (доска, Q&A) — переводятся с `require_moderator` на скоуп-способности без изменения поведения (moderator покрывает все скоупы).

**Тесты/acceptance.** Параметризованная матрица «действие × роль/грант → статус» (из §4 документа прав); deny-by-default тест на каждый привилегированный роутер; конфликт интересов; отзыв гранта действует немедленно. Acceptance: user с грантом `moderator.board` работает с очередью доски и не имеет доступа к очередям других скоупов; все его действия видны в ленте governance. **Не делаем:** заявки на модерацию (фаза 2), автоматическую выдачу прав по репутации (запрещено инвариантом).

**Объём.** 1 миграция, 1 сервис + правки зависимостей роутеров, 2 admin-поверхности, ~30–40 тестов.

**Статус.** Реализовано (миграция 049 — 4 таблицы + `author_metrics_enabled` +
пороги `author_metrics.min_methodology_length`/`author_metrics.min_country_coverage`;
пакеты `repositories/author_metrics/` и `services/author_metrics/`
(`definitions.py`, `values.py`, `subscriptions.py`, `reputation.py`, `overlay.py`,
`forks.py`, `helpers.py`); публичный + `/me` роутер `api/v1/author_metrics.py` и
`api/v1/admin_author_metrics.py`; `scripts/recompute_author_reputation.py`
зарегистрирован в `dev_tools_scripts_runner.py`; DQ-проверки подключены в
`services/data_quality/report.py`; контракт и `packages/contracts/generated/types.ts`
обновлены; ~85 тестов, включая инвариант-снапшот CII/decision против
`services/author_metrics`). Фронтенд (витрина авторских метрик на странице
страны) не входит в этот эпизод — по прецеденту эпизодов 1–3 визуальный слой
относится к отдельному визуальному транчу после эпизода 7.

**Цель.** Открыть систему метрик сообществу: создание, премодерация, публикация, подписки, форки, производная репутация авторов. Платформенный CII неприкосновенен.

**Модель данных (миграция 049).**

```sql
CREATE TABLE author_metric_definitions (
    id UUID PK,
    author_user_id UUID NOT NULL REFERENCES users(id),
    slug TEXT NOT NULL,
    name_en/name_ru TEXT NOT NULL,
    methodology_en/methodology_ru TEXT NOT NULL,     -- обязательна к публикации
    polarity TEXT NOT NULL,                          -- positive|negative
    scale_min NUMERIC NOT NULL DEFAULT 0,
    scale_max NUMERIC NOT NULL DEFAULT 100,
    license TEXT NOT NULL DEFAULT 'platform',        -- platform|cc_by_sa (выбор автора
                                                     -- при публикации; решение Р-4)
    status TEXT NOT NULL DEFAULT 'draft',            -- существующий lifecycle
    visibility TEXT NOT NULL DEFAULT 'public',
    forked_from_id UUID REFERENCES author_metric_definitions(id),
    version INT NOT NULL DEFAULT 1,
    created_at/updated_at/published_at TIMESTAMPTZ,
    CONSTRAINT uq_author_metric UNIQUE (author_user_id, slug)
);

CREATE TABLE author_metric_values (
    id UUID PK,
    metric_id UUID NOT NULL REFERENCES author_metric_definitions(id) ON DELETE CASCADE,
    country_id UUID NOT NULL REFERENCES countries(id),
    value NUMERIC NOT NULL,
    source_url TEXT,
    is_personal_experience BOOLEAN NOT NULL DEFAULT FALSE,
    note TEXT,
    valid_as_of DATE NOT NULL,
    updated_at TIMESTAMPTZ,
    CONSTRAINT uq_metric_country UNIQUE (metric_id, country_id),
    CONSTRAINT chk_provenance CHECK (source_url IS NOT NULL OR is_personal_experience)
);

CREATE TABLE author_subscriptions (
    id UUID PK, user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    metric_id UUID REFERENCES author_metric_definitions(id) ON DELETE CASCADE,
    author_user_id UUID REFERENCES users(id) ON DELETE CASCADE,
    created_at TIMESTAMPTZ,
    CONSTRAINT chk_target CHECK (metric_id IS NOT NULL OR author_user_id IS NOT NULL)
);

CREATE TABLE author_reputation (                     -- производная, пересчитываемая
    author_user_id UUID PRIMARY KEY REFERENCES users(id),
    coverage_score/freshness_score/sourcing_score NUMERIC,
    subscriber_count INT, published_metric_count INT,
    computed_at TIMESTAMPTZ, methodology_version TEXT
);
```

**Сервисный слой — пакет `services/author_metrics/`:** `definitions.py` (CRUD + lifecycle submit→review→publish; PII-скан текстов; порог публикации: методология не короче минимума, покрытие ≥ N стран — параметры Эпизода 1), `values.py` (bulk-upsert, валидация шкалы), `subscriptions.py`, `reputation.py` (recompute-скрипт по образцу trust), `overlay.py` (авторские слои для страницы страны/сравнения — **отдельные эндпоинты**, не примесь к CII), `forks.py` (форк копирует определение и методологию с `forked_from_id`; значения автор заполняет сам), `helpers.py`.

**Права.** Создание — грант `author.metrics`; премодерация — скоуп `moderator.metrics` (до зрелости института — владелец). Выдача первых грантов — invite-only (см. холодный старт).

**Холодный старт (владелец — не «автор №1»).** (1) Платформенные методологии (CII, Trust, Drift) отображаются в том же витринном формате «методология + версия + обновлено» — образец жанра существует с первого дня; (2) шаблоны из route checklists — авторский контент без авторов; (3) invite-only гранты активным участникам с высокой производной репутацией; (4) открытая подача — после обкатки модерации.

**Ключевые инварианты (закрепить тестами).** Ни один платформенный эндпоинт CII/decision/compare не меняет ответ от существования авторских данных (снапшот-тест до/после сида); публикация без методологии невозможна; каждое значение — источник ИЛИ явная пометка «личный опыт»; форк несёт родословную.

**API (флаг `author_metrics_enabled`).** `GET/POST/PATCH /me/author-metrics` (+submit/archive); `PUT /me/author-metrics/{id}/values` (bulk); `POST /author-metrics/{id}/fork`; `GET /authors/{user_id}/metrics`; `GET /countries/{slug}/author-metrics`; подписки `POST/DELETE /me/subscriptions` + `GET /me/subscriptions/feed`; admin-очередь `GET /admin/author-metrics?status=review`.

**Фундамент донатов (без реализации).** Никаких платёжных таблиц; достаточно `author_user_id` + подписок. Инвариант: деньги не влияют на репутацию/права/выдачу (см. реестр). Конкретная модель — после ответов на открытые вопросы.

**Acceptance.** Полный путь draft→review→published; слой подписчика на странице страны с подписью «автор, версия, обновлено»; инвариант-тест ядра зелёный. **Не делаем:** монетизацию, «топы авторов», комментарии к метрикам, примешивание к CII.

**Объём.** 1 миграция (4 таблицы), пакет ~7 модулей, 1 recompute-скрипт, admin-роутер, ~60–80 тестов.

### Эпизод 5 — `feat/country-contribution-v1`: контрибуция стран сообществом

**Цель.** Открыть контент-ядро: пользователи с грантом предлагают страны и ведут их наполнение по формальному стандарту под кураторством редакции. Решить судьбу тестовых стран.

**Основа.** Стандарт онбординга страны уже формализован кодом: dq-проверки `build_country_onboarding_dq_results` и тесты onboarding-стандарта описывают, что должна иметь страна (карточка, источники, сигналы, CII-метрики, evidence). Этот стандарт становится **автоматическим гейтом контрибуции** — программным чек-листом готовности.

**Модель данных (миграция 050).**

```sql
CREATE TABLE country_proposals (
    id UUID PK,
    proposer_user_id UUID NOT NULL REFERENCES users(id),
    slug TEXT NOT NULL UNIQUE,
    name_en/name_ru TEXT NOT NULL,
    justification TEXT NOT NULL,                 -- зачем страна нужна платформе
    status TEXT NOT NULL DEFAULT 'draft',        -- существующий lifecycle
    curator_user_id UUID REFERENCES users(id),   -- закреплённый editor
    readiness_snapshot JSONB,                    -- последний прогон onboarding-гейта
    created_at/updated_at/published_at TIMESTAMPTZ
);
```

Страна-предложение создаёт скрытую (не listed) запись `countries` со статусом черновика; наполнение (карточка, источники, сигналы, значения метрик) идёт существующими редакционными механизмами, но правки контрибутора ограничены его страной (грант `contributor.countries` + владение предложением). Перевод в review/published — только editor-куратор после зелёного onboarding-гейта.

**Консервация демо-набора (решение Р-1: тестовые страны не удаляются никогда).** Текущие RU/UY/AR в этом эпизоде: (а) прогоняются через onboarding-гейт как первые «пациенты» пайплайна; (б) вводится флаг `countries.is_demo` — демо-страны скрываются со всех публичных поверхностей (листинги, поиск, решения), но остаются в БД; (в) полный связный дамп демо-набора (страны, карточки, источники, сигналы, метрики, маршруты) экспортируется в JSON-фикстуры `database/fixtures/demo_countries/`; (г) создаётся dev-инструмент `scripts/dev_tools/restore_demo_countries.py` — идемпотентное восстановление демо-набора из фикстур одной командой (регистрируется в оркестраторе `dev_tools_scripts_runner.py` одной записью `ScriptInfo`). Разработчик в любой момент получает рабочий набор данных для локальной проверки.

**Права.** `contributor.countries` — invite-only грант; кураторство — editor; publish — editor/owner. Атрибуция контрибутора сохраняется публично (страница страны: «данные при участии …»).

**Тесты/acceptance.** Гейт: страна без обязательного набора не проходит в review (автоматически); контрибутор не может править чужие страны/ядро; куратор обязателен для publish; демо-страны не видны ни на одной публичной поверхности; `restore_demo_countries.py` восстанавливает набор идемпотентно, и восстановленный набор проходит onboarding-гейт. Acceptance: полная страна-кандидат проходит путь proposal→наполнение→гейт→review→published и появляется во всех поверхностях (решения, сравнение, поиск) без ручных правок кода. **Не делаем:** открытую подачу без гранта; автоматическую публикацию без куратора; массовый импорт стран; удаление демо-данных.

**Объём.** 1 миграция, сервис-пакет `country_contribution/`, admin/куратор-поверхности, экспорт-фикстуры + restore-скрипт, ~45–60 тестов.

### Эпизод 6 — `feat/migration-flows-v1`: потоки и статистика

**Цель.** Агрегированная картина направлений «кто куда движется» с жёсткой анонимизацией (k=20); уровни достоверности планов без документов; «дневник переезда» и вытеснение синтетики.

**Модель данных (миграция 051).**

```sql
CREATE TABLE migration_flow_aggregates (      -- материализуется джобой
    id UUID PK,
    origin_country_id UUID NOT NULL REFERENCES countries(id),
    destination_country_id UUID NOT NULL REFERENCES countries(id),
    period TEXT NOT NULL,                     -- '2026-Q3'
    trip_count INT NOT NULL,                  -- строка не пишется, если < k
    weighted_count NUMERIC NOT NULL,          -- с весами достоверности
    computed_at TIMESTAMPTZ, methodology_version TEXT,
    CONSTRAINT uq_flow UNIQUE (origin_country_id, destination_country_id, period)
);
```

**Уровни достоверности** (`trips.confidence_tier`, без документов — инвариант): `declared` (создан) → `active` (эвристика: ≥X выполненных шагов + возраст аккаунта; параметры Эпизода 1) → `confirmed` (пользователь отметил «переехал»).

**«Дневник переезда».** Отметка «я переехал» превращает план одним действием в черновик структурированной user story (маршрут, сроки, шаги предзаполнены) → обычная модерация историй. Петля: план → переезд → история → данные для следующих. С ростом реальных историй применяется **политика вытеснения синтетики** (решение владельца): synthetic-истории снимаются с публичных поверхностей по мере покрытия соответствующих сценариев/стран реальными (маркировка и dq-контроль уже существуют).

**Сервис и скрипт.** `services/migration_flows.py` (чистый агрегатор: trips active/confirmed + опубликованные посты доски → пары × кварталы; отбрасывание групп < k; веса по тирам) + `scripts/recompute_migration_flows.py` (идемпотентно, dry-run, сводка).

**API.** `GET /migration-flows?origin=&destination=&period=`; `GET /me/trips/{id}/flow-context` — «в вашем направлении в этом квартале: ≥N планов» + ссылка на доску попутчиков.

**Privacy-инварианты (тесты обязательны).** Ни при каких параметрах API не возвращает группу < 20 (значение — параметр `flows.k_anonymity`); только страна→страна × квартал — без городов, дат, ID; пересчёт не логирует состав групп.

**Acceptance.** Агрегаты соответствуют сиду; группа k−1 не видна; тир повышается по правилам; «дневник» создаёт черновик истории из завершённого плана. **Не делаем:** списки участников направлений; realtime-счётчики; хранение документов (навсегда).

**Объём.** 1 миграция, 1 сервис + скрипт, ~25–35 тестов.

### Эпизод 7 — `feat/community-threads-v1`: треды на контактах

**Цель.** Лёгкий обмен сообщениями строго между взаимно согласившимися (принятый contact request) + Telegram deep-link. Не мессенджер.

**Модель данных (миграция 052).**

```sql
CREATE TABLE contact_threads (
    id UUID PK,
    contact_request_id UUID NOT NULL UNIQUE REFERENCES migration_board_contact_requests(id),
    status TEXT NOT NULL DEFAULT 'open',      -- open|closed_by_a|closed_by_b|frozen
    created_at TIMESTAMPTZ
);
CREATE TABLE thread_messages (
    id UUID PK, thread_id UUID NOT NULL REFERENCES contact_threads(id) ON DELETE CASCADE,
    sender_user_id UUID NOT NULL REFERENCES users(id),
    body TEXT NOT NULL,
    created_at TIMESTAMPTZ
);
```

**Правила.** Тред — только на `accepted` контакт; лимит сообщений/день (параметр); любая сторона закрывает; блокировка замораживает; репорт треда → модератор видит переписку **только по репорту** (политика приватности + аудит доступа). Доставка — polling (`GET …/messages?after=`), без WebSocket в v1. Deep-link: при взаимной линковке Telegram — обмен контактами по явному двухшаговому согласию обеих сторон.

**Acceptance.** Переписка возможна только в принятом контакте; закрытие/блокировка/репорт работают; лимиты применяются; модераторский доступ к переписке оставляет след в аудите. **Не делаем:** вложения, групповые чаты, realtime, поиск по переписке.

**Объём.** 1 миграция, расширение пакета migration_board, ~20–30 тестов.

## 3. Сквозные инженерные темы

### 3.1 Приватность как программа

- **Private-first**: всё личное создаётся приватным; публикация — явное действие с предпросмотром публичной проекции.
- **Никогда не храним** (инварианты): проездные документы/брони/сканы; сырые share-токены (только hash+prefix); IP/user-agent в аналитике.
- **k-анонимность** всех агрегатов о людях (k=20, параметр).
- **Права пользователя**: экспорт всех своих данных (JSON) и каскадное удаление аккаунта — реализуются в Эпизоде 2 (с появлением первых «тяжёлых» личных данных), не откладываются.
- Retention interaction-логов и аналитики — параметр методологии.

### 3.2 Производительность и вычислительный стек

Правило: **Postgres до триггера**. Все расчёты — за чистыми функциональными швами (образцы: агрегатор CII, trust, drift). Триггер пересмотра: агрегатный запрос стабильно >1с на реальной нагрузке или batch не влезает в окно → точечная замена одного модуля (NumPy-джоба/Rust-бинарь за CLI), контракт не меняется. PyTorch — вне повестки (ML-задач нет по архитектурной позиции).

### 3.3 Чек-лист нового домена

1. Ветка `feat/<slug>-v1`; миграция со следующим номером + заголовочный комментарий.
2. Сервис-пакет по house style (`helpers.py`, квалифицированный доступ, re-export в `__init__.py`).
3. Feature-флаг (выключен) + `ensure_feature_enabled` на каждой поверхности; права — `require_roles`/`require_capability`.
4. События — только через outbox (`event_key`-идемпотентность).
5. dq-модуль `_append_<domain>_checks` + подключение в отчёт.
6. Тесты: unit, API (mock-repo), migration-тест схемы, privacy/rights-инварианты.
7. Полный quality gate перед merge; push в main — по стандарту.

### 3.4 Наблюдаемость

Каждый новый домен пишет analytics-события по существующему паттерну (hash-сессии, безопасные metadata): created/updated/shared для планов; published/subscribed/forked для метрик; proposal-события для стран; reminder_sent — из метрик relay. Счётчики пересчётов — в сводки recompute-скриптов.

### 3.5 Основные риски

| Риск | Митигация |
|---|---|
| Модерационная нагрузка UGC на одного человека | Пирамида автономии (уровни 0–1 до людей); invite-only авторы/контрибуторы; премодерация до зрелости института модераторов; параметризованные лимиты |
| Холодный старт творческого контура без «автора №1» | Платформенные методологии как витрина; шаблоны из готовых чек-листов; invite-only ранние авторы |
| Размывание доверия авторским контентом | Жёсткая сепарация «платформа vs автор»; обязательная методология; инвариант-тесты ядра |
| Накрутка статистики потоков | Уровни достоверности + k=20 + взвешивание; аномалии в аналитике |
| Юридические риски UGC-советов | Поза «не юридическая консультация» на всех авторских/пользовательских поверхностях |
| Scope creep solo-проекта | Эпизодная дисциплина; отклонённые направления зафиксированы в видении §5; изменения — только явным пересмотром |
| Рост outbox-трафика (напоминания) | Батчинг relay; отдельный event_type; при необходимости отдельный топик — конфигурацией |

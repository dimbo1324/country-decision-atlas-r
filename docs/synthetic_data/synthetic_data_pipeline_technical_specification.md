# Техническое задание: конвейер синтетических данных Country Decision Atlas

## 1. Назначение документа

Этот документ описывает мини-проект по развитию `scripts/synthetic_data` в
управляемый конвейер синтетических данных для локального тестирования и
отладки Country Decision Atlas.

Документ определяет целевую архитектуру, границы безопасности, формат данных,
режимы запуска, порядок внедрения и критерии приёмки. Это проектное ТЗ, а не
описание конкретного коммита: перед реализацией отдельные решения должны быть
сверены с актуальными архитектурными документами и состоянием репозитория.

## 2. Контекст и проблема

Приложение работает с решениями о выборе страны и использует структурированные
сведения: страны, метрики, источники, правовые сигналы, статьи, комментарии,
пользователи, уведомления и документы. Для полноценного ручного тестирования
нужны не отдельные случайные файлы, а связанные между собой тестовые данные.

Существующий `scripts/synthetic_data` уже умеет создавать JSON, Markdown,
XLSX, DOCX и PDF, в том числе документы с копируемым текстом, сканы и
смешанные документы. Сейчас он генерирует в основном нейтральный текст и не
создаёт согласованный предметный мир: один и тот же факт не связывается
надёжно с SQL, статьёй, источником и документом.

Целевое состояние: один запуск создаёт самостоятельный вымышленный набор
данных, в котором все артефакты ссылаются на одни и те же сущности и могут
быть загружены в изолированное локальное окружение приложения.

## 3. Цели

Конвейер должен:

1. Создавать от 4 до 5 полностью вымышленных стран с уникальными названиями,
   кодами, профилями, временными изменениями и связанными материалами.
2. Формировать логически правдоподобные, но не реальные характеристики стран:
   экономика, безопасность, свободы, стоимость жизни, цифровая инфраструктура,
   миграционная открытость, устойчивость институтов и другие нужные приложению
   показатели.
3. Создавать синтетических пользователей, авторов, статьи, комментарии,
   источники, правовые сигналы, сценарии действий и документы.
4. Генерировать несколько представлений одного набора фактов: канонический
   JSON, SQL-fixtures, DOCX, PDF, XLSX, TXT и при необходимости ZIP-архив.
5. Поддерживать воспроизводимость: одинаковая версия входных данных и один
   `seed` должны давать одинаковый логический набор данных.
6. Позволять получать разные, но согласованные миры при смене `seed` или
   профиля сценария.
7. Проверять себя до выдачи результата: ссылки, уникальность, диапазоны,
   отсутствие реальных стран и согласованность фактов между артефактами.
8. Оставаться автономным: без сетевых запросов, внешних API, реальных
   персональных данных и реальных государственных документов.

## 4. Не-цели и границы

В рамках мини-проекта не требуется:

- заменять продуктовые миграции базы данных автоматически;
- менять существующую схему БД без отдельной задачи и миграции;
- подключаться к государственным сайтам, LLM, новостным API или внешним
  сервисам;
- имитировать реальные страны, их граждан, документы, символику или законы;
- использовать данные реальных людей;
- делать генератор инструментом для подготовки поддельных удостоверений,
  справок, виз, банковских документов или иных документов, которые могут быть
  приняты за реальные;
- публиковать созданные артефакты в production-окружение.

Конвейер предназначен только для разработки, тестов, демонстраций и локального
ручного тестирования. Каждый синтетический артефакт должен быть явно помечен
как тестовый.

## 5. Базовый архитектурный принцип

### 5.1. Один канонический мир

Сначала создаётся единственный канонический набор доменных сущностей,
условно `SyntheticWorld`. Только после этого отдельные адаптеры рендерят его
в JSON, SQL, документы и сценарии.

Недопустим подход, при котором PDF, SQL и JSON по отдельности случайно
выбирают факты о стране. Он приводит к противоречиям: например, в базе страна
с низкой безопасностью, а в PDF названа самой безопасной.

```text
Входная конфигурация + seed
             |
             v
    Построение SyntheticWorld
             |
             v
   Семантическая валидация мира
             |
             +--------------------+-------------------+------------------+
             |                    |                   |                  |
             v                    v                   v                  v
      canonical JSON          SQL-fixtures      Рендереры файлов      Сценарии
                                                PDF/DOCX/XLSX/TXT
             |                    |                   |                  |
             +--------------------+-------------------+------------------+
                                  |
                                  v
                      manifest.json и ZIP-пакет
```

### 5.2. Разделение ответственности

Будущая реализация должна сохранить текущий стиль скрипта и разделить код по
назначению:

| Слой | Ответственность |
| --- | --- |
| `input` | Загрузка и проверка описаний мира, блоков и профилей. |
| `domain` | Модели канонического мира и правила предметной области. |
| `generation` | Детерминированное создание сущностей из seed и профиля. |
| `validation` | Проверка связности, ограничений и безопасности. |
| `renderers` | Вывод канонических сущностей в JSON, SQL, PDF, DOCX, XLSX, TXT. |
| `scenarios` | Создание пошаговых пользовательских сценариев и ожидаемых результатов. |
| `packaging` | Manifest, контрольные суммы, ZIP-архив и отчёт о запуске. |
| `cli` | Удобная командная строка, режимы dry-run и понятные ошибки. |

Маршрутизация, бизнес-правила приложения и SQL не должны смешиваться с
рендерингом документов. Скрипт остаётся внешним инструментом подготовки
тестовых данных.

## 6. Каноническая модель SyntheticWorld

### 6.1. Метаданные запуска

Каждый мир должен содержать:

- `dataset_id` — уникальный идентификатор набора;
- `schema_version` — версия структуры синтетического датасета;
- `generator_version` — версия генератора;
- `seed` — исходный seed;
- `profile` — имя профиля мира;
- `supported_locales` — перечень языков, включённых в набор;
- `generated_at` — время создания;
- `fictional_notice` — обязательная маркировка, что данные вымышлены;
- `source_config_checksum` — контрольная сумма входной конфигурации.

Время создания не должно влиять на логические данные. Для строгого сравнения
результатов необходимо иметь режим с фиксированным временем или исключать
операционные поля из проверки воспроизводимости.

### 6.2. Страны

Для каждой страны нужны как минимум:

- стабильный внутренний UUID;
- уникальные вымышленные `code`, `slug` и локализованные названия;
- короткое описание и набор тематических фактов;
- профиль или архетип;
- показатели текущего периода;
- историю показателей по нескольким датам;
- набор связанных источников, статей, правовых сигналов и документов;
- признаки, используемые сценариями и SQL-fixtures.

Рекомендуемые измерения страны:

| Группа | Примеры показателей |
| --- | --- |
| Экономика | доходы, рынок труда, инфляционное давление, стоимость жизни, устойчивость валюты. |
| Общество | безопасность, доступность медицины, образование, качество среды. |
| Свободы и институты | гражданские свободы, независимость институтов, предсказуемость регулирования. |
| Практика переезда | сложность легализации, доступность удалённой работы, качество цифровых услуг. |
| Риски | политическая волатильность, природные риски, риск резких изменений правил. |
| Качество данных | число источников, свежесть, уверенность, наличие противоречий. |

Набор показателей должен соответствовать реальным полям и методологии
приложения. Ядро CII и его математика не меняются генератором: он только
создаёт допустимые исходные значения для существующей модели.

### 6.3. Люди и контент

Синтетический мир включает:

- пользователей с искусственными именами, ролями и настройками;
- авторов материалов с репутацией, специализацией и историей публикаций;
- статьи с заголовком, блоками текста, ссылками на страны и источники;
- комментарии с автором, временем, контекстом статьи и состоянием
  модерации;
- источники с типом, датой, уровнем доверия, ссылкой-заглушкой и извлечёнными
  утверждениями;
- правовые сигналы: событие, дата вступления в силу, затронутые страны,
  влияние и уверенность.

Имена, e-mail, URL и идентификаторы обязаны быть синтетическими. Для e-mail
следует применять зарезервированные домены, например `example.test`, а не
рабочие публичные почтовые домены.

### 6.4. Документы

Документ — это производный артефакт, а не самостоятельный источник истины.
В каноническом мире он содержит:

- `document_id`;
- тип: обзор страны, аналитическая заметка, табличный отчёт, выдержка закона,
  письмо пользователя, пакет источников и т. п.;
- связанные сущности и факты;
- рецепт из блоков;
- список форматов для рендеринга;
- режим текста: `copyable`, `non_copyable` или `mixed`;
- обязательную синтетическую маркировку.

## 7. Правдоподобная случайность

### 7.1. Архетипы вместо независимого рандома

Показатели нельзя выбирать независимо в диапазоне от 0 до 100. Такая
случайность быстро выглядит нелогично. Сначала стране назначается архетип, а
затем параметры варьируются в его допустимых границах.

Минимальный стартовый набор архетипов:

1. Стабильная, технологичная и дорогая страна.
2. Доступная, но институционально нестабильная страна.
3. Свободная и безопасная страна с небольшим рынком труда.
4. Экономически сильная страна с ограниченными свободами.
5. Страна восстановления или ухудшения после заметного события.

Архетипы не должны быть оценочными ярлыками. Они определяют распределения,
корреляции, риски и возможные сюжетные изменения.

### 7.2. Правила связности

Генератор должен иметь набор явных ограничений, например:

- высокая стоимость жизни обычно повышает вероятность высокого дохода, но не
  обязана это гарантировать;
- высокая политическая нестабильность снижает уверенность в долгосрочных
  прогнозах и повышает частоту противоречивых источников;
- свежий правовой сигнал обязан отражаться в истории изменений и связанных
  материалах;
- отрицательный тренд не может быть заявлен без хотя бы одного события или
  источника, объясняющего его;
- у каждой страны должны быть сильные стороны, ограничения и неопределённости;
- среди 4–5 стран не должно быть одной неизменно лучшей во всех измерениях.

Численные связи могут быть вероятностными. Цель — не моделировать реальную
экономику, а создавать полезные и объяснимые тестовые варианты.

### 7.3. Временная динамика

Мир должен содержать несколько снимков во времени: например, базовый период,
недавний период и текущий период. Изменения формируются как события:

- изменение миграционного правила;
- улучшение цифрового сервиса;
- рост стоимости жизни;
- снижение уверенности в данных;
- появление противоречащего источника;
- изменение параметра безопасности.

Это позволяет тестировать функции «что изменилось», свежесть, доверие,
drift, watchlist и уведомления.

## 8. Конструктор текстовых и документных блоков

### 8.1. Блоки вместо жёстких шаблонов

Входная конфигурация хранит библиотеку блоков, а не один статичный документ.
Блок имеет идентификатор, назначение, условия применимости, варианты текста,
плейсхолдеры и метаданные.

Пример логической структуры блока:

```json
{
  "id": "country_risk_summary",
  "kind": "paragraph",
  "applies_to": ["country_overview", "migration_report"],
  "requires": ["country.name", "country.risks", "country.metrics"],
  "variants": [
    "Для {country.name} ключевым ограничением остаётся {risk_label}.",
    "При оценке {country.name} следует учитывать фактор: {risk_label}."
  ]
}
```

Варианты выбираются детерминированно из seed конкретной сущности. При этом
документ сохраняет связность: в нём нельзя упомянуть риск, которого нет в
канонической модели страны.

### 8.2. Рецепты документов

Рецепт описывает порядок блоков, а не готовый текст. Например, рецепт
`country_brief` может включать заголовок, сводку, таблицу метрик, сильные
стороны, риски, правовые сигналы, перечень источников и маркировку.

Рендереры DOCX, PDF, TXT и Markdown используют один и тот же рецепт,
сохраняя содержание, но адаптируя оформление к формату.

### 8.3. Многоязычные данные и проверка кодировок

Конвейер должен поддерживать 15 языков (точнее, 15 локалей) не только для
перевода интерфейсных строк, но и для генерации статей, комментариев,
источников, имён, документов и табличных значений. Цель — проверить полную
цепочку обработки Unicode: хранение в JSON и БД, API, отображение в браузере,
создание DOCX/PDF/XLSX/TXT, поиск, копирование текста и последующий перевод.

Первая утверждённая матрица локалей:

| Локаль | Язык и письменность | Языковая группа | Что дополнительно проверяет |
| --- | --- | --- | --- |
| `en-US` | английский, латиница | индоевропейская, германская | Базовый латинский текст. |
| `es-ES` | испанский, латиница | индоевропейская, романская | Диакритика и инвертированные знаки. |
| `ru-RU` | русский, кириллица | индоевропейская, славянская | Кириллица и иной алфавит. |
| `hi-IN` | хинди, деванагари | индоевропейская, индоарийская | Сложные индийские графемы. |
| `fa-IR` | персидский, арабское письмо | индоевропейская, иранская | Справа налево и соединяемые символы. |
| `ar-SA` | арабский, арабское письмо | афразийская, семитская | Справа налево, цифры и смешанное направление. |
| `zh-Hans-CN` | китайский, упрощённые иероглифы | сино-тибетская | Иероглифы без пробелов между словами. |
| `ja-JP` | японский, кандзи/кана | японская | Смешение нескольких систем письма. |
| `ko-KR` | корейский, хангыль | корейская | Корейские блоки символов. |
| `tr-TR` | турецкий, латиница | тюркская | Специальные латинские буквы и регистр. |
| `id-ID` | индонезийский, латиница | австронезийская | Нелатинская языковая семья на латинице. |
| `sw-KE` | суахили, латиница | нигеро-конголезская, банту | Ещё одна самостоятельная языковая группа. |
| `vi-VN` | вьетнамский, латиница с тонами | австроазиатская | Плотная диакритика и нормализация Unicode. |
| `th-TH` | тайский, тайское письмо | кра-дайская | Сегментация слов и нестандартные переносы. |
| `ta-IN` | тамильский, тамильское письмо | дравидийская | Ещё одна индийская письменность. |

Региональные части кодов локалей служат только стандартными техническими
идентификаторами языка и не означают, что синтетические страны имеют отношение
к существующим государствам. Все страны, люди, учреждения и сюжетные факты
остаются полностью вымышленными.

#### Принципы создания многоязычного контента

1. Факт существует один раз в канонической модели и имеет нейтральный ключ,
   например «страна / показатель / период / значение». Язык меняет только
   представление этого факта, а не сам факт.
2. Для каждого языка библиотека содержит собственные варианты заголовков,
   предложений, имён и комментариев. Нельзя строить весь набор простым
   машинным переводом одной английской заготовки: это не проверит реальные
   особенности письма и грамматики.
3. Для каждого блока указывается, для каких локалей он доступен. Если блока
   нет, генератор останавливается с понятной ошибкой или применяет заранее
   утверждённый нейтральный fallback; молча подменять язык английским нельзя.
4. Должны быть варианты с короткими и длинными строками, диакритикой,
   смешанным текстом, числом, датой, URL-заглушкой и названием синтетической
   страны. Это выявит обрезание и неверный перенос текста.
5. Переводы не должны создавать другую оценку страны. Утверждение, метрика,
   источник и идентификатор сущности должны оставаться одинаковыми во всех
   локалях.

#### Режимы работы с переводом

Для первой версии генератор создаёт исходные локализованные тексты из
подготовленных библиотек блоков. Это надёжно проверяет кодировку и рендеринг.

Позднее можно добавить отдельный тестовый режим проверки нейросетевого
перевода. Он не заменяет исходные тексты и всегда сохраняет:

- исходную локаль и исходный текст;
- целевую локаль;
- название и версию переводческого провайдера либо локальной модели;
- дату, тестовый seed и статус проверки;
- маркировку, что перевод синтетический и не является источником истины.

Этот режим должен быть отделён от генерации мира. Он получает уже созданный
синтетический текст и создаёт производный тест перевода. До отдельного
согласования интеграционной части проекта он работает через fake-by-default
адаптер или локальную модель, без передачи данных во внешние сервисы.

#### Unicode, шрифты и направление письма

1. Все текстовые файлы записываются в UTF-8 без неявной подстановки символов.
2. JSON сохраняется с настоящими Unicode-символами, а не только с
`\\uXXXX`-последовательностями: так легче увидеть повреждения кодировки.
3. PDF должен встраивать или явно использовать шрифты с покрытием нужного
письма; нельзя рассчитывать на случайный шрифт операционной системы.
4. DOCX задаёт шрифт для обычного текста и проверяет открытие в целевой среде;
при отсутствии нужного шрифта результат должен считаться непригодным, а не
молча отображаться квадратиками.
5. Для арабского и персидского текста необходимо явно задавать направление
абзаца справа налево. В одном документе должны быть также смешанные строки:
арабский/персидский текст, число, латинский идентификатор и URL-заглушка.
6. Проверки PDF/DOCX должны включать визуальный рендер страниц и извлечение
текста: один тест не заменяет другой.
7. Поиск и сравнение строк должны учитывать нормализацию Unicode. Для тестов
нужны пары визуально похожих, но по-разному нормализованных символов, прежде
всего для вьетнамского и текстов с диакритикой.

### 8.4. Входные данные для генератора

Входом конвейера будет не один хаотичный огромный файл, а понятный набор
версируемых JSON-файлов. Их можно хранить рядом в папке исходных данных и
проверять по схеме до запуска.

Минимальный состав входа:

- конфигурация мира: профили стран, диапазоны метрик, правила связности и
  перечень допустимых событий;
- словари вымышленных названий стран, городов, людей и организаций;
- библиотека текстовых блоков для каждой из 15 локалей;
- рецепты документов: какие блоки и таблицы собирать для каждого типа файла;
- каталог сценариев: какие действия и ожидаемые результаты должны появиться;
- конфигурация форматов: режимы PDF/DOCX, лимиты файлов, шрифты и локали;
- параметры конкретного запуска: `seed`, профиль, число стран, нужные форматы
  и выходная папка.

Все входные файлы содержат только вымышленные заготовки и технические
настройки. Они не содержат паролей, настоящих адресов, токенов, ключей,
персональных данных или production-параметров.

## 9. Сценарии ручного и автоматизированного тестирования

### 9.1. Формат сценария

Сценарий должен быть машиночитаемым и понятным человеку. Он состоит из:

- `scenario_id`, названия, категории и профиля;
- исходного состояния: пользователь, страны, нужные записи БД;
- последовательности действий пользователя;
- ожидаемых наблюдаемых результатов;
- ссылок на нужные документы и источники;
- меток риска: авторизация, модерация, свежесть, поиск, уведомления и т. п.

Пример сценария: пользователь сравнивает две страны, открывает источник,
оставляет комментарий, добавляет одну страну в watchlist, после чего видит
синтетическое изменение показателя и уведомление.

### 9.2. Виды сценариев

Первый набор должен покрывать:

- просмотр и сравнение стран с различными компромиссами;
- поиск по статьям, странам, источникам и сигналам;
- работу с документами: текстовый PDF, скан и смешанный файл;
- комментарии и состояния модерации;
- источники с высокой, низкой и противоречивой уверенностью;
- изменение данных и экран «что изменилось»;
- watchlist и безопасную фиктивную доставку уведомления;
- ошибки валидации, неполные данные и запрещённые действия по ролям.

Сценарии не должны содержать настоящие учётные данные. Для локального запуска
они используют предусмотренный приложением тестовый режим аутентификации или
специально созданные фиктивные записи.

## 10. Выходные артефакты и структура папок

Рекомендуемая структура одного запуска:

```text
docs/synthetic_data/output_data/<dataset_id>/
  manifest.json
  canonical/
    synthetic_world.json
    scenarios.json
  sql/
    seed_synthetic_world.sql
    cleanup_synthetic_world.sql
  documents/
    <locale>/
      pdf/copyable/
      pdf/non_copyable/
      pdf/mixed/
      docx/copyable/
      docx/non_copyable/
      docx/mixed/
      xlsx/
      txt/
  reports/
    validation_report.json
    generation_summary.md
  package/
    synthetic_dataset_<dataset_id>.zip
```

`manifest.json` обязан перечислять каждый артефакт, его тип, размер,
контрольную сумму, связанные сущности и способ создания. Это делает набор
удобным для ручной проверки и расследования проблем.

### 10.1. Выходные данные простыми словами

После одного запуска получается одна большая тестовая «папка с миром».
Внутри неё есть:

- главный JSON-файл — понятная компьютеру карточка всего вымышленного мира:
  какие страны существуют, какие у них показатели, статьи, люди, комментарии
  и события;
- набор сценариев — простые инструкции, что открыть и что нажать в
  приложении, чтобы проверить определённую ситуацию;
- SQL-файл — аккуратная инструкция для тестовой базы данных, как создать те
  же вымышленные страны и связанные записи;
- документы — одни и те же синтетические факты в PDF, Word, Excel, TXT и
  других выбранных форматах, на одной или нескольких из 15 локалей;
- отчёт проверки — список того, что генератор создал и что успешно проверил;
- manifest — оглавление всей папки: где лежит каждый файл и с какими
  странами, статьями или сценариями он связан;
- ZIP-архив — удобная упакованная копия всего набора для передачи в локальное
  тестовое окружение.

Например, если генератор придумал страну «Варелия», в главном JSON будет её
описание, в SQL — запись для тестовой базы, в PDF — обзор Варелии, в Excel —
таблица показателей, а в сценарии — задание сравнить её с другой вымышленной
страной. Все эти части будут говорить об одной и той же Варелии, а не о разных
случайных версиях страны.

Генерируемые файлы не должны попадать в git по умолчанию. В репозитории
хранятся только исходные конфигурации, тестовые образцы малого размера и код
генератора. Если конкретный набор нужен как регрессионный fixture, он должен
добавляться осознанно отдельной задачей.

## 11. SQL-fixtures и безопасность базы данных

### 11.1. Отдельно от миграций

Генератор не должен переписывать, удалять или изменять существующие миграции.
Особенно нельзя менять уже применённые миграции, содержащие данные реальных
стран: их имена и контрольные суммы могут быть частью истории базы.

Правильный путь — отдельный, идемпотентный SQL-fixture для временной локальной
или тестовой базы, который запускается после применения схемы. Его выполнение
не должно быть частью обычного запуска приложения.

### 11.2. Безопасный SQL

SQL-генератор должен:

- использовать только значения, созданные внутри конвейера;
- экранировать строки и идентификаторы через надёжный генератор/адаптер, а не
  ручной конкатенацией;
- создавать записи с отдельным префиксом набора, например `syn_<dataset_id>`;
- быть идемпотентным в пределах одного `dataset_id`;
- иметь отдельный безопасный скрипт очистки только для записей этого набора;
- отказываться запускаться на production-конфигурации;
- требовать явный флаг подтверждения для загрузки в БД.

На раннем этапе лучше генерировать SQL-файл без автоматического подключения к
базе. Позднее можно добавить отдельную команду загрузки с проверкой среды,
но только после согласования её с архитектурой и правилами проекта.

## 12. Командная строка и удобство использования

Текущий CLI следует расширять без нарушения существующих форматов.

Пример целевого интерфейса:

```powershell
# Проверить конфигурацию, ничего не создавать
python -m scripts.synthetic_data.cli validate --profile balanced

# Показать план создаваемых сущностей и файлов
python -m scripts.synthetic_data.cli plan --profile crisis --seed 42017

# Создать полный пакет в указанной папке
python -m scripts.synthetic_data.cli generate --profile balanced --seed 42017

# Создать только канонические данные и SQL-fixture
python -m scripts.synthetic_data.cli generate --formats canonical,sql --seed 42017

# Перерендерить документы из уже созданного canonical JSON
python -m scripts.synthetic_data.cli render --dataset <dataset_id> --formats pdf,docx,xlsx,txt

# Упаковать существующий набор
python -m scripts.synthetic_data.cli package --dataset <dataset_id>
```

Требования к UX CLI:

- `--dry-run` не создаёт папки и файлы;
- каждая ошибка содержит причину и путь к проблемной конфигурации;
- команда в конце выводит `dataset_id`, seed, путь к manifest и краткую
  статистику;
- `--seed` печатается всегда, включая случайно сгенерированный seed;
- `--output-root` позволяет не загрязнять репозиторий;
- опасные операции требуют явных флагов и понятного подтверждения;
- режим `validate` доступен без дополнительных тяжёлых библиотек рендеринга.

## 13. Детерминизм и генерация случайности

Обычного общего объекта `random.Random(seed)` недостаточно для устойчивой
воспроизводимости: добавление одного документа в середину может изменить
случайные значения всех последующих сущностей.

Рекомендуемый подход — независимые детерминированные потоки случайности. Для
каждой сущности и задачи создаётся seed, производный от главного seed и
стабильного контекста, например:

```text
derived_seed = hash(master_seed, "country", country_id, "metrics")
```

Так изменение одного рецепта документа не меняет метрики всех стран. Алгоритм
вывода должен быть стабильным и не зависеть от случайного порядка словарей,
текущего времени или UUID, созданного после генерации. Идентификаторы при
необходимости также могут выводиться детерминированно из dataset seed.

## 14. Валидация и контроль качества

Перед записью файлов конвейер обязан валидировать:

### 14.1. Структуру

- соответствие входной и канонической модели схемам Pydantic/JSON Schema;
- обязательные поля и типы;
- уникальность кодов, slug, идентификаторов и имён файлов;
- существование всех ссылок на страны, пользователей, статьи и источники.

### 14.2. Семантику

- попадание метрик в допустимые диапазоны;
- выполнение правил архетипа и временной динамики;
- наличие сильных сторон, рисков и неопределённостей у каждой страны;
- соответствие текста фактам, которые он описывает;
- наличие источника или события у существенного изменения;
- отсутствие «идеальной» страны без компромиссов, если это не выделенный
  крайний тестовый профиль.

### 14.3. Безопасность и фиктивность

- отсутствие названий, кодов и доменов реальных стран;
- отсутствие реальных персональных данных и настоящих контактов;
- обязательную маркировку `SYNTHETIC TEST DATA — NOT REAL` в текстовых
  артефактах;
- наличие всех обязательных текстовых блоков для выбранных локалей;
- корректное направление строк и абзацев для RTL-локалей;
- наличие шрифта с нужным покрытием и отсутствие потерянных символов при
  визуальном рендеринге;
- отсутствие полей, похожих на номера документов, платёжные реквизиты,
  реальные адреса или действующие QR-коды;
- отсутствие сетевых вызовов;
- отсутствие SQL-команд, способных удалить чужие данные или изменить схему.

### 14.4. Артефакты

- JSON читается и соответствует схеме;
- XLSX открывается и содержит ожидаемые листы/таблицы;
- DOCX и PDF создаются и открываются;
- copyable-файлы содержат извлекаемый текст;
- non-copyable-файлы не содержат основной текстовый слой;
- mixed-файлы содержат и текстовый, и графический фрагменты;
- тексты всех 15 локалей сохраняют Unicode-символы после записи, чтения и
  извлечения из поддерживаемых форматов;
- manifest содержит все выданные файлы и корректные контрольные суммы.

## 15. Маркировка синтетических материалов

На каждом документе, в JSON, SQL-комментарии и manifest должна быть заметная
маркировка, например:

```text
SYNTHETIC TEST DATA — FICTIONAL COUNTRIES AND PERSONS — NOT FOR PRODUCTION USE
```

Для PDF/DOCX маркировка размещается в заголовке или колонтитуле каждой
страницы. Для XLSX — на первом листе и в свойствах книги. Для TXT/JSON — в
первой строке или в поле метаданных. Маркировка должна быть достаточно ясной,
чтобы материал нельзя было случайно принять за настоящий.

Сама маркировка должна быть доступна во всех поддерживаемых локалях, но её
английский вариант сохраняется дополнительно во всех артефактах как единый
технический сигнал для тестов и ручной проверки.

## 16. Профили мира

Минимальная первая версия должна поддерживать профили:

| Профиль | Назначение |
| --- | --- |
| `balanced` | Разнообразный набор компромиссов, базовый режим ручного тестирования. |
| `crisis` | Больше изменений, рисков, противоречивых источников и устареваний. |
| `optimistic` | В целом благоприятные условия, но без страны, лучшей во всём. |
| `data_quality` | Неполные, конфликтующие и разной свежести данные. |
| `moderation` | Акцент на статьях, комментариях, ролях и состояниях модерации. |

Позже могут появиться профили для миграционных сценариев, маршрутов,
персонализации и нагрузочных объёмов. Новый профиль не должен незаметно менять
смысл существующего профиля при фиксированном seed.

## 17. Предлагаемый порядок реализации

### Этап 1. Основание

1. Зафиксировать JSON Schema/Pydantic-модели канонического мира.
2. Описать библиотеку вымышленных названий, архетипов и текстовых блоков.
3. Ввести стабильную производную генерацию случайности.
4. Реализовать `validate` и `plan`.

Результат: валидный `synthetic_world.json` без SQL и документов.

### Этап 2. Страны и временная динамика

1. Генерировать 4–5 стран разных архетипов.
2. Добавить метрики, тренды, события, источники и правила связности.
3. Реализовать семантическую валидацию и отчёт о ней.

Результат: один мир содержит достаточно данных для сравнения стран и экранов
доверия/изменений.

### Этап 3. Контент и сценарии

1. Добавить пользователей, авторов, статьи, комментарии и статусы модерации.
2. Добавить каталог сценариев ручного тестирования.
3. Привязать сценарии к конкретным сущностям канонического мира.

Результат: воспроизводимые пользовательские истории с ожидаемым результатом.

### Этап 4. Рендеринг документов

1. Перевести текущие генераторы PDF/DOCX/XLSX/Markdown на канонические
   сущности и рецепты.
2. Добавить TXT.
3. Сохранить и развить режимы copyable/non-copyable/mixed.
4. Добавить визуальную и структурную проверку артефактов.

Результат: комплект согласованных документов разных типов.

### Этап 5. SQL-fixtures и упаковка

1. Сопоставить синтетические сущности с актуальной схемой БД.
2. Генерировать только отдельные идемпотентные seed-fixtures.
3. Добавить безопасную очистку по `dataset_id`.
4. Добавить manifest, контрольные суммы и ZIP-пакет.

Результат: переносимый пакет для локальной тестовой базы и ручного теста.

### Этап 6. Интеграция и регрессии

1. Добавить тесты генератора и контрактные проверки SQL-fixture против чистой
   локальной БД.
2. Добавить небольшой фиксированный smoke-датасет в тестовый контур.
3. Зарегистрировать необходимые проверки в developer tooling проекта.
4. Проверить запуск в Docker/local development режиме без внешних сервисов.

Результат: воспроизводимый и безопасный инструмент, не требующий ручной
подготовки данных.

## 18. Тестирование реализации

Нужны как минимум следующие тесты:

- одинаковый seed даёт равный канонический мир после исключения операционных
  метаданных;
- смена seed меняет мир, сохраняя валидность;
- добавление нового рецепта документа не меняет существующие метрики при том
  же seed;
- каждый профиль генерирует 4–5 вымышленных стран;
- все ссылки между сущностями разрешаются;
- запрещённые реальные названия, домены и данные отклоняются;
- SQL-fixture безопасно выполняется на чистой тестовой БД и повторный запуск
  не дублирует записи;
- cleanup удаляет только объекты нужного `dataset_id`;
- PDF/DOCX/XLSX/TXT проходят структурную проверку;
- `--dry-run` не создаёт файлов;
- невалидная конфигурация завершает работу с ненулевым кодом и понятным
  сообщением;
- manifest соответствует фактически созданным файлам;
- каждая из 15 локалей проходит запись и чтение UTF-8, а целевые PDF/DOCX
  открываются без квадратов, вопросительных знаков вместо символов и
  повреждённого направления текста;
- поиск находит подготовленные слова с диакритикой, кириллицей, иероглифами и
  RTL-текстом; проверка сравнения строк учитывает Unicode-нормализацию;
- тестовый переводческий адаптер сохраняет пару «исходный текст — перевод» и
  не меняет канонический факт.

Тесты `scripts/synthetic_data` следует явно включить в целевой запуск
качества, поскольку стандартный `pytest` проекта может не собирать их по
умолчанию.

## 19. Риски и меры защиты

| Риск | Мера |
| --- | --- |
| Несогласованные факты в разных файлах | Один канонический `SyntheticWorld`, рендеринг только из него. |
| Случайное попадание реальных стран или людей | Закрытые словари, deny-list, обязательная валидация фиктивности. |
| Повреждение данных разработки | Отдельные SQL-fixtures, явный `dataset_id`, никакого изменения миграций. |
| Запуск против production | Отказ по production-конфигурации, явное подтверждение опасной операции. |
| Невоспроизводимый баг | Seed, версия генератора, checksum входа, manifest. |
| Слишком большой и медленный набор | Профили объёма, лимиты количества сущностей и потоковый рендеринг при необходимости. |
| Случайное коммитирование больших файлов | Игнорирование output-папок и хранение в git только исходников/малых fixtures. |
| Ложное ощущение реалистичности | Явная маркировка, запрет на имитацию удостоверяющих документов и реальных атрибутов. |
| Повреждение многоязычного текста | UTF-8, проверенные шрифты, RTL-правила, визуальный рендер и извлечение текста. |
| Ошибка перевода меняет смысл факта | Канонические факты не переводятся; перевод хранится как отдельный производный артефакт. |

## 20. Критерии готовности первой полезной версии

Первая версия считается готовой, если:

1. Одна команда создаёт `balanced`-мир из 4–5 вымышленных стран.
2. Канонический JSON содержит страны, метрики, историю изменений, источники,
   статьи, комментарии, документы и не менее трёх сценариев.
3. Один и тот же seed повторяемо создаёт тот же логический набор.
4. Создаются минимум JSON, TXT, XLSX, DOCX и PDF; PDF/DOCX поддерживают
   copyable, non-copyable и mixed-варианты там, где это требуется.
5. Все документы основаны на данных канонического мира и содержат маркировку.
6. Создаётся отдельный SQL-fixture, не меняющий миграции и безопасный для
   чистой локальной тестовой БД.
7. Есть manifest, отчёт валидации и ZIP-пакет.
8. Автоматические тесты покрывают детерминизм, связность, фиктивность,
   безопасность SQL и базовую пригодность файлов.
9. Скрипт не использует сеть, реальные данные и production-интеграции.
10. Инструкция запуска понятна разработчику без знания внутреннего кода.
11. Для каждой из 15 локалей создаётся хотя бы один текстовый артефакт; набор
    проверяет UTF-8, шрифты, переносы и направление письма там, где это нужно.

## 21. Решения, которые нужно принять до начала кодирования

Перед реализацией следует отдельно утвердить:

1. Какие именно доменные таблицы и сущности текущей БД должны заполняться в
   первой версии SQL-fixture.
2. Полный перечень метрик страны и допустимые диапазоны, согласованные с
   существующей методологией и инвариантами продукта.
3. Нужны ли синтетические файлы только для ручного тестирования или также для
   CI/E2E; от этого зависят размер и время генерации.
4. Должны ли сгенерированные страны полностью заменить текущие демонстрационные
   записи локальной базы либо существовать рядом с ними. Рекомендуемое решение
   для начала: существовать отдельно, с изолированным `dataset_id`.
5. Полный словарь текстовых блоков для 15 утверждённых локалей и набор
   шрифтов, которые разрешено использовать/встраивать для PDF и DOCX.
6. Какие документы должны создаваться на всех 15 языках, а какие достаточно
   создать на выборочной матрице локалей и форматов для экономии времени.
7. Когда и через какой отдельный адаптер будет подключаться проверка
   нейросетевого перевода. Она должна использовать текущий слой локализации
   приложения и не может незаметно менять канонические данные.
8. Какие типы документов действительно необходимы первой версии и какие поля
   приложение извлекает из каждого из них.

## 22. Итоговая концепция

Конвейер синтетических данных должен быть не фабрикой случайных файлов, а
локальным симулятором правдоподобных ситуаций вокруг выбора страны. Его
ценность создают не количество PDF или строк SQL, а связность: источник
подтверждает факт, факт влияет на метрику, метрика отражена в статье и
документе, а пользовательский сценарий проверяет, как приложение показывает
это изменение.

Такой подход позволит безопасно моделировать разнообразные пользовательские
истории, тестировать сложные ветви интерфейса и серверной логики, проверять
Unicode и 15 языковых локалей, а также воспроизводить найденные ошибки по
одному `seed` без использования реальных
стран, людей и документов.

## 23. Пошаговый план реализации

### 23.1. Правила выполнения плана

Каждый этап ниже является отдельной, законченной задачей разработки. Его
начинают только после успешного завершения предыдущего этапа. Перед началом
каждого этапа нужно сверить актуальное состояние кода, архитектурные
инварианты, `task-checklist.md` и схему базы данных.

Для каждого этапа обязательно:

- [ ] Создать отдельную ветку и отдельный `task-checklist.md` по рабочему
  стандарту проекта.
- [ ] Не изменять применённые миграции и не подключать реальные внешние
  сервисы.
- [ ] Добавить тесты ровно для нового поведения этапа.
- [ ] Выполнить целевые проверки после реализации и полный quality gate перед
  merge.
- [ ] Отметить выполненные и невыполненные пункты в checklist и подготовить
  честный итоговый отчёт.

Статусы этапов в этом документе начинают как **запланировано** и меняются на
«реализовано» только после завершения кода, тестов и проверок. Нельзя считать
этап готовым только потому, что появился файл или команда CLI.

- [ ] Реализовать отдельный идемпотентный `seed_synthetic_world.sql` с
  `dataset_id` и безопасным экранированием значений.
- [ ] Реализовать отдельный `cleanup_synthetic_world.sql`, который способен
  удалить только записи своего набора и никогда не трогает схему или чужие
  данные.
- [ ] Добавить отдельную явную команду локальной загрузки с проверкой
  окружения, блокировкой production и требованием подтверждения.
- [ ] Не изменять, не удалять и не перегенерировать существующие миграции с
  демонстрационными странами.

**Проверка и критерий перехода.**

- [ ] Fixture применяется к чистой локальной тестовой базе после миграций.
- [ ] Повторное применение не создаёт дубликатов.
- [ ] Cleanup удаляет только записи нужного `dataset_id`.
- [ ] Попытка запуска в production-подобной конфигурации завершается до
  подключения к базе.
- [ ] Сценарии из Этапа 3 находят созданные SQL-данные по ожидаемым
  идентификаторам.

### Этап 7 — интеграция в рабочий процесс и выпуск первой версии

**Статус.** Запланировано.

**Цель.** Сделать конвейер удобным и надёжным инструментом, который можно
запускать регулярно разработчику, тестировщику и CI без ручной сборки мира.

**Зависимости.** Этапы 1–6.

**Состав работ.**

- [ ] Завершить CLI-команды `validate`, `plan`, `generate`, `render` и
  `package`, сохранить обратную совместимость текущих режимов там, где это
  возможно.
- [ ] Подготовить небольшой фиксированный smoke-набор для быстрых тестов и
  расширенные профили для ручного тестирования.
- [ ] Явно включить тесты `scripts/synthetic_data` в подходящий quality gate
  или отдельный зарегистрированный инструмент, чтобы они не выпадали из
  регулярной проверки.
- [ ] Добавить проверки времени выполнения, размера пакета и понятности
  ошибок CLI.
- [ ] Провести тестовый запуск в чистом локальном/Docker development-окружении
  без реальных интеграций.
- [ ] Сверить результат с этим ТЗ, архитектурными инвариантами и правилами
  проекта; обновить архитектурную документацию, если фактическая архитектура
  изменилась.

**Проверка и критерий завершения мини-проекта.**

  15 локалями, сценариями, документами, manifest и SQL-fixture.
- [ ] Все целевые тесты генератора, статические проверки и полный quality gate

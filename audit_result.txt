ОБЩИЙ ВЕРДИКТ (кратко)
--------------------------------------------------------------------------------
Проект в целом в очень хорошей форме для solo-разработки: слои разделены
дисциплинированно (router → service → repository), SQL всюду параметризован,
контракт OpenAPI синхронизируется тестом байт-в-байт, есть outbox, feature-флаги,
audit-события, data-quality гейты, deny-by-default RBAC, PII-фильтры, Go-сервис
написан аккуратно (graceful shutdown, DLQ, метрики, таймауты). Фронтенд строго
типизирован (strict, ноль `any`), эффекты чистятся.

Главные системные риски, найденные аудитом:
  1. Массовый анти-паттерн `uuid_колонка::text = %s` (187 мест) — под ростом
     данных превращает почти каждый точечный запрос в seq scan (P0).
  2. Токен сессии в localStorage + отсутствие CSP + TTL 7 дней — XSS даёт
     полный захват аккаунта (P0).
  3. Брутфорс логина фактически не ограничен (P0).
  4. Публикация в Kafka внутри открытой транзакции с блокировками строк (P1).
  5. Класс TOCTOU-гонок на «лимитах в день» и на «check-then-insert» (P1).
  6. Полное отсутствие политики retention для append-only таблиц (P1).
  7. Кеш-слой создаёт новый Redis-клиент на каждый запрос (P1, спящая мина —
     проявится при включении CACHE_MODE=redis).

[P0-1] Несаргируемые предикаты `uuid_колонка::text = %s` — 187 вхождений
--------------------------------------------------------------------------------
Разделы: 4 (PostgreSQL), 10 (Масштабируемость), 2 (Ресурсы)
Файлы (главные, полный список — grep '::text = %s' по apps/api/app/repositories):
  apps/api/app/repositories/auth.py            (~20 мест, самый горячий путь)
  apps/api/app/repositories/capabilities.py    (has_active_grant — каждый
                                                защищённый запрос)
  apps/api/app/repositories/author_metrics/*.py
  apps/api/app/repositories/country_contribution/*.py
  apps/api/app/repositories/admin_content.py
  apps/api/app/repositories/countries.py:111,150
  apps/api/app/repositories/country_pairs.py:93,127
  apps/api/app/repositories/decision_passports.py:134
  apps/api/app/repositories/common.py:75 (require_country_id)
  ... и далее (всего 187 вхождений в 20+ файлах)

Почему это ошибка.
  Условие вида `WHERE id::text = %s` применяет функцию (каст) к ИНДЕКСИРОВАННОЙ
  колонке. B-tree индекс построен по типу uuid; выражение `id::text` в него не
  попадает, поэтому планировщик не может использовать ни PK-индекс, ни FK-индексы
  и выбирает Seq Scan + каст каждой строки. То же для `user_id::text = %s`,
  `metric_id::text = %s` и т.д.

Последствия.
  Сейчас (сотни строк в таблицах) — незаметно, все тесты зелёные.
  С ростом: auth_sessions читается/обновляется НА КАЖДЫЙ аутентифицированный
  запрос (validate_session_token → get_active_session..., touch_session,
  count_active_sessions), user_capabilities читается на каждый запрос под
  require_capability. При 100k сессий каждый API-вызов = полное сканирование
  таблицы сессий. Это тихая деградация всей платформы, которую невозможно
  починить «одним индексом» постфактум — паттерн размазан по 187 местам.

Как устранить.
  Инвертировать каст: приводить ПАРАМЕТР к uuid, а не колонку к тексту:
    WHERE id = %s::uuid          (psycopg сам приведёт str → uuid и без каста,
                                  если передавать uuid.UUID или строку в
                                  параметре с `= %s` при типизированном столбце)
  Практический рецепт для команды:
    1) Единый хелпер в core (например `as_uuid_param`) либо просто договориться
       писать `column = %s::uuid` и передавать строку.
    2) Механическая замена по регулярке `(\w+)::text = %s` → `\1 = %s::uuid`
       (кроме мест, где колонка реально text).
    3) Добавить тест-инвариант (grep-тест в духе существующих), запрещающий
       появление `::text = %s` в repositories.
  Замена безопасна: все передаваемые значения — валидные UUID-строки (приходят
  из БД или из path-параметров, где невалидный uuid и так даёт пустой результат;
  для path-параметров добавить 422-валидацию UUID на уровне Pydantic/route).

[P0-2] Bearer-токен в localStorage + отсутствие Content-Security-Policy
--------------------------------------------------------------------------------
Разделы: 5 (Безопасность), 6 (Frontend)
Файлы:
  apps/web/src/shared/auth/session.ts        (localStorage.setItem cda_auth_token)
  apps/api/app/bootstrap/app_factory.py      (_register_middleware — заголовки)
  apps/api/app/core/config.py                (auth_session_ttl_hours: 168)

Почему это ошибка.
  Токен сессии доступен любому JS на странице. Единственная защита от кражи
  токена через XSS — не иметь XSS, но CSP не настроен ни на API, ни на Next.js
  (next.config.mjs не задаёт headers), а платформа целенаправленно строится на
  пользовательском контенте (Q&A, доска, авторские методологии) — то есть
  поверхность для stored-XSS будет только расти. TTL токена — 7 суток, ротации
  нет: украденный токен живёт неделю.

Последствия.
  Один пропущенный XSS в любом месте фронтенда = полный захват аккаунта
  (включая owner/editor-аккаунты — у них те же механизмы сессий) на 7 дней.
  Для платформы, чей продукт — доверие, это репутационно фатальный сценарий.

Как устранить (по нарастающей стоимости):
  1) Минимум: добавить CSP (default-src 'self'; script-src 'self'; object-src
     'none'; base-uri 'self') через Next headers() + убрать устаревший
     X-XSS-Protection на API, добавить HSTS (см. P2-5).
  2) Правильно: перевести сессию на httpOnly + Secure + SameSite=Lax cookie,
     выдаваемый бэкендом; Authorization-заголовок оставить для API-клиентов.
     Потребуется CSRF-защита на мутирующих эндпоинтах (double-submit token) —
     сейчас её нет, потому что cookie не используются.
  3) Сократить TTL до 24-48ч + скользящее продление (touch уже есть) либо
     refresh-токены.

[P0-3] Брутфорс /auth/login фактически не ограничен
--------------------------------------------------------------------------------
Разделы: 5 (Безопасность), 3 (Конкурентность), 7 (Отказоустойчивость)
Файлы:
  apps/api/app/services/auth.py (login_user — нет учёта неудачных попыток)
  apps/api/app/bootstrap/app_factory.py:141-162 (rate limiter)
  apps/api/app/main.py (_rate_limit_client)
  docker-compose.yml:40 (APP_ENV: local по умолчанию)

Почему это ошибка.
  Единственная защита — глобальный in-memory лимитер, который:
    а) работает ТОЛЬКО при APP_ENV=production (в compose по умолчанию local —
       т.е. в том виде, в котором стек сейчас поднимается, лимитера нет вообще);
    б) считает per-IP: 120 (в compose 600) запросов/мин с одного IP — этого
       достаточно для перебора словаря против одного аккаунта: ~170k-860k
       попыток/сутки;
    в) per-process: несколько воркеров = умножение лимита;
    г) не различает эндпоинты: login не строже, чем чтение стран.
  Отдельно: PBKDF2 с 260k итераций — это намеренно дорого (~50-150мс CPU);
  без лимитера залп логинов — ещё и дешёвый CPU-DoS.

Последствия.
  Перебор паролей и credential stuffing против будущих пользователей;
  исчерпание CPU одним скриптом.

Как устранить.
  1) Пер-аккаунт счётчик неудачных попыток (таблица или Redis):
     после N неудач — экспоненциальная задержка/временная блокировка +
     audit-событие. 2) Отдельный жёсткий лимит на /auth/login и /auth/register
     (например 5/мин/IP + 20/час/email) независимо от APP_ENV.
  3) Долгосрочно — вынести лимитер в Redis (он уже в стеке), см. P1-6.

[P1-1] Публикация в Kafka внутри открытой DB-транзакции с row-locks
--------------------------------------------------------------------------------
Разделы: 3 (Конкурентность), 7 (Отказоустойчивость), 4 (PostgreSQL)
Файлы:
  scripts/outbox_relay_runner.py:100-157 (run_relay)
  apps/api/app/repositories/domain_events.py (lock_pending_notifiable_...
  FOR UPDATE SKIP LOCKED)

Почему это ошибка.
  run_relay открывает транзакцию, берёт FOR UPDATE SKIP LOCKED на батч (до 50
  событий) и затем ВНУТРИ транзакции для каждого события делает
  producer.produce(...) + producer.flush(). flush() блокируется до подтверждения
  доставки; у confluent-kafka дефолтный message.timeout.ms = 300 секунд.
  Если Kafka недоступна/деградировала, транзакция и блокировки строк держатся
  до 50 × 300с = часами: это длинная транзакция (мешает autovacuum, держит xmin),
  заблокированные строки и невозможность параллельного релея.

Последствия.
  Один инцидент с Kafka каскадно бьёт по Postgres (bloat, блокировки).
  Также flush() на каждое сообщение — это синхронная сериализация: пропускная
  способность релея ограничена RTT до брокера на каждое событие.

Как устранить.
  Вариант А (минимальный): задать produce-конфиг с коротким
  message.timeout.ms (например 10s) и delivery.timeout; flush один раз на батч,
  ПОСЛЕ выхода из транзакции; статусы relayed/failed отмечать во второй короткой
  транзакции по результатам delivery-callbacks.
  Вариант Б (каноничный outbox): в транзакции только SELECT batch + пометка
  «in-flight», публикация вне транзакции, финальная пометка отдельной
  транзакцией. Семантика at-least-once сохраняется (notifier уже дедуплицирует
  по event_key через Mongo dedup).

[P1-2] Класс TOCTOU-гонок: «посчитал — проверил лимит — вставил»
--------------------------------------------------------------------------------
Разделы: 3 (Конкурентность и целостность данных)
Файлы (репрезентативные, паттерн повторяется):
  apps/api/app/services/migration_board/threads.py:66-84
      (count_messages_created_since >= limit → create_message)
  apps/api/app/services/migration_board/helpers.py / contacts.py
      (лимиты постов/контакт-запросов/репортов в день — тот же паттерн)

Почему это ошибка.
  Между COUNT(...) и INSERT нет ни блокировки, ни уникального ограничения.
  Два параллельных запроса оба видят count = limit-1 и оба вставляют — лимит
  превышен. READ COMMITTED не защищает от этого по определению.

Последствия.
  Все «пирамида автономии, уровень 0»-лимиты (антиспам-механика платформы!)
  обходимы конкурентными запросами. Сейчас — теоретически; при появлении
  недоброжелателей — практический обход антиспама скриптом на 20 параллельных
  соединений.

Как устранить (на выбор, по месту):
  1) `SELECT pg_advisory_xact_lock(hashtext('thread_msg:' || user_id))` перед
     count — сериализует только конкурентов одного пользователя, дёшево.
  2) INSERT ... SELECT ... WHERE (SELECT COUNT(*) ...) < %s RETURNING id —
     атомарная проверка в одном statement.
  Одного общего хелпера в migration_board/helpers.py достаточно, чтобы закрыть
  весь класс.

[P1-3] Гонки check-then-insert возвращают 500 вместо 409 (нет обработки
       UniqueViolation)
--------------------------------------------------------------------------------
Разделы: 3 (Конкурентность), 1 (Архитектура — единообразие обработки ошибок)
Файлы:
  apps/api/app/services/auth.py:104-113 (register_user: get_user_by_email →
      create_user; гонка на UNIQUE(email))
  apps/api/app/services/country_contribution/proposals.py (create_proposal:
      country_slug_exists/country_iso_exists → INSERT; гонка на UNIQUE slug/iso)
  apps/api/app/services/author_metrics/definitions.py:45-53
      (get_definition_by_author_slug → create_definition)
Образец правильного решения уже есть в кодовой базе:
  apps/api/app/services/weight_profiles.py:57,124 (except UniqueViolation → 409)

Почему это ошибка.
  Проверка существования и вставка — разные транзакционные моменты. При гонке
  вставка падает UniqueViolation, глобальный обработчик превращает это в
  500 database_error. Клиент получает «ошибка сервера» на легитимный конфликт,
  ретраи выглядят как баги.

Последствия.
  Двойной сабмит формы регистрации (двойной клик) с вероятностью гонки даёт 500;
  мониторинг ошибок засоряется ложными «database_error».

Как устранить.
  Обернуть INSERT в try/except psycopg.errors.UniqueViolation → поднять тот же
  409-api_error, что и синхронная проверка (по образцу weight_profiles.py).
  Проверку заранее можно оставить как fast-path.

[P1-4] Retention отсутствует: append-only таблицы растут бесконечно
--------------------------------------------------------------------------------
Разделы: 4 (PostgreSQL), 2 (Ресурсы), 10 (Масштабируемость)
Файлы/объекты:
  analytics_events        (migration 030; INSERT на каждое действие; POST
                           /api/v1/analytics — анонимный)
  audit_events            (migration 007; каждый admin/mod/грант)
  domain_events           (migration 0xx; relayed-строки никогда не удаляются)
  ai_interaction_logs     (Эпизод 12-13)
  auth_sessions           (expired/revoked строки живут вечно; touch_session
                           дописывает dead tuples)
  Ни одного cleanup-скрипта в scripts/ нет (проверено grep retention/DELETE).

Почему это ошибка.
  Документация (02_Текущее_состояние_системы.md §3.1 приватность; план §3.1
  «Retention interaction-логов и аналитики — параметр методологии») заявляет
  retention как параметр — в коде ни параметра, ни джобы. Это прямой
  drift доков и кода. Анонимный POST /analytics без аутентификации (лимитер —
  см. P0-3 — в local выключен) позволяет раздувать таблицу извне.

Последствия.
  Медленная деградация: индексы растут, COUNT(*) OVER() в админ-листах (см.
  P2-2) дорожает, бэкапы толстеют. Плюс privacy-обязательство «мы храним
  ограниченно» не выполняется технически.

Как устранить.
  1) Скрипт scripts/cleanup_retention.py (по образцу recompute-скриптов:
     идемпотентный, dry-run, сводка): DELETE analytics_events старше N дней,
     domain_events со status='relayed' старше N, auth_sessions expired/revoked
     старше N; N — параметры методологии (methodology_parameters), как и
     обещано доками. Зарегистрировать в dev_tools_scripts_runner.py.
  2) При реальном росте — перевести analytics_events на партиционирование по
     месяцу (сейчас дешевле всего: таблица молодая).

[P1-5] get_cache_backend() создаёт новый Redis-клиент на каждый вызов
--------------------------------------------------------------------------------
Разделы: 2 (Память и ресурсы), 10 (Масштабируемость)
Файлы:
  apps/api/app/services/cache.py:111-124 (Redis.from_url на каждый вызов)
  Горячие вызовы: services/country_read_model.py:47, services/cii_matrix.py:60,
  services/routes.py:63, services/legal_signal_timeline.py:42,
  services/home_overview.py:31, services/cache_invalidation.py (5 мест)

Почему это ошибка.
  Пока CACHE_MODE=null (дефолт) — невидимо. В момент включения redis каждый
  запрос к странице страны/матрице/маршрутам будет создавать НОВЫЙ Redis-клиент
  (и TCP-соединение) — классическое «соединение на запрос» из чек-листа §2.
  Дополнительная ловушка: socket_timeout=0.2s задан, но сам коннект на каждый
  запрос под нагрузкой обвалит и Redis, и латентность.

Последствия.
  Включение кеша (задумано как оптимизация) ухудшит производительность и
  устроит утечку сокетов. Обнаружится только в проде, потому что тесты гоняются
  с NullCache.

Как устранить.
  Сделать backend синглтоном процесса: module-level переменная либо
  @lru_cache(maxsize=1) поверх фабрики (по образцу get_settings), сброс — только
  в тестах. redis-py сам держит connection pool внутри клиента.

--------------------------------------------------------------------------------
[P1-6] Rate limiter: in-memory, per-process, только APP_ENV=production
--------------------------------------------------------------------------------
Разделы: 5 (Безопасность), 10 (Масштабируемость), 7 (Отказоустойчивость)
Файлы:
  apps/api/app/bootstrap/app_factory.py:141-162
  apps/api/app/main.py:18-59 (_rate_windows, cleanup)

Почему это ошибка.
  Сам код честно логирует ограничение (per-process). Но комбинация свойств
  делает защиту иллюзорной: (а) выключен всюду, кроме production; (б) обнуляется
  рестартом; (в) масштабирование воркеров умножает лимит; (г) при
  trusted_proxy_headers=true берётся первый элемент X-Forwarded-For без списка
  доверенных прокси — клиент может подставить заголовок и получить безлимит
  (per-IP ключ подделан).

Последствия.
  См. P0-3; плюс ложное чувство защищённости.

Как устранить.
  Перенести лимитер в Redis (INCR + EXPIRE или скользящее окно в sorted set) —
  инфраструктура уже в docker-compose; включить независимо от APP_ENV
  (конфигурируемые лимиты); для X-Forwarded-For — валидировать, что
  request.client.host входит в список доверенных прокси, прежде чем верить
  заголовку.

--------------------------------------------------------------------------------
[P1-7] Поиск обновляется только ручным скриптом — публикации не попадают
       в индекс автоматически
--------------------------------------------------------------------------------
Разделы: 1 (Архитектура/границы), 8 (Наблюдаемость данных), 10 (Масштабируемость)
Файлы:
  scripts/rebuild_search_index.py (единственный писатель search_documents)
  apps/api/app/services/admin_content.py (publish-переходы: инвалидируют
      Redis-кеш, событие в outbox — но search_documents не трогают)

Почему это ошибка.
  Контент публикуется редакцией/куратором, но в /api/v1/search появится только
  после ручного `rebuild_search_index.py --all`. Никакого напоминания/джобы нет.
  CI это маскирует (там rebuild вызывается явно). Search-DQ-проверки
  (list_active_countries_missing_from_index и т.п.) поймают рассинхрон лишь при
  ручном прогоне data-quality отчёта.

Последствия.
  На живой системе поиск будет систематически отставать от контента; для
  продукта «доказательная платформа» это выглядит как потеря данных.

Как устранить.
  Либо (а) инкрементальный upsert search_documents в publish-транзакции (у
  rebuild-скрипта уже есть вся логика маппинга — вынести в сервис и
  переиспользовать), либо (б) подписать rebuild на domain_events
  (*.published уже эмитятся) в существующем relay-контуре, либо (в) как минимум
  задокументированная периодическая джоба в оркестраторе. Вариант (а)
  предпочтителен: одна транзакция = строгая консистентность.

[P1-8] touch_session: UPDATE на каждый аутентифицированный запрос
--------------------------------------------------------------------------------
Разделы: 4 (PostgreSQL), 2 (Ресурсы), 10 (Масштабируемость)
Файлы:
  apps/api/app/services/auth.py:219
  apps/api/app/repositories/auth.py:343-353 (touch_session; + ::text cast,
      + fetch_all для UPDATE)

Почему это ошибка.
  Каждый GET под авторизацией порождает запись в auth_sessions (dead tuple +
  WAL). Вместе с P0-1 (seq scan) это самый горячий write-путь системы, который
  не даёт пользы точнее «минуты последней активности».

Последствия.
  Bloat auth_sessions, лишний WAL/репликационный трафик, autovacuum-давление —
  всё на ровном месте.

Как устранить.
  Обновлять last_seen_at не чаще раза в N минут: в validate_session_token
  сравнить session["last_seen_at"] с NOW()-interval и делать UPDATE только при
  превышении (данные сессии уже прочитаны — сравнение бесплатно). Заодно
  заменить fetch_all на execute и убрать ::text.

[P2-2] COUNT(*) OVER() в списковых запросах
--------------------------------------------------------------------------------
Разделы: 4 (PostgreSQL), 10 (Масштабируемость)
Файлы (9 мест):
  apps/api/app/repositories/audit.py:80
  apps/api/app/repositories/capabilities.py
  apps/api/app/repositories/author_metrics/definitions.py (модерационные списки)
  apps/api/app/repositories/country_contribution/proposals.py:165,192
  apps/api/app/repositories/migration_board/* (списки постов/тредов)

Почему это ошибка.
  Оконный COUNT(*) OVER() заставляет Postgres материализовать ПОЛНЫЙ результат
  фильтра для каждой страницы, сводя пагинацию LIMIT/OFFSET к O(N) на страницу.
  Для audit_events/analytics-масштабов (см. P1-4) это станет самым тяжёлым
  запросом админки.

Последствия.
  Деградация админ-листов и модерационных очередей по мере роста данных.

Как устранить.
  Для админ-листов достаточно: отдельный быстрый COUNT по индексу выполняемый
  реже (или приблизительный из pg_class.reltuples), либо keyset-пагинация
  (created_at, id) без total. Не трогать до появления реальных объёмов, но
  заложить в стандарт новых списков keyset-подход.

[P2-3] Админ-рекомпьюты идут синхронно в HTTP-запросе по ВСЕМ странам
--------------------------------------------------------------------------------
Разделы: 7 (Отказоустойчивость), 10 (Масштабируемость), 1 (Архитектура)
Файлы:
  apps/api/app/api/v1/trust.py:86-101 (admin_recompute_all_trust)
  apps/api/app/api/v1/platform_metrics.py (admin_recompute_all)

Почему это ошибка.
  Цикл по всем активным странам с расчётами и upsert'ами — в одном HTTP-запросе

  страной либо переделать в enqueue-модель (запись задания + фоновый прогон).
  Коммитить по-странно, а не одним куском.

[P2-4] Пул БД: фиксированный max_size=10 без таймаута и телеметрии,

Разделы: 2 (Ресурсы), 7 (Отказоустойчивость), 10 (Масштабируемость)

[P2-9] Транзакционная дисциплина: ручной connection.commit() в 40+ роутерах
--------------------------------------------------------------------------------
Разделы: 3 (Конкурентность/целостность), 1 (Архитектура)
Файлы:
  apps/api/app/api/v1/*.py (auth.py, author_metrics.py, country_contribution.py,
      admin_*.py, migration_board.py, ... — все мутирующие хендлеры)
  apps/api/app/core/database.py:41-43 (get_connection — отдаёт соединение;
      psycopg_pool коммитит при чистом выходе, откатывает при исключении)

Почему это ошибка.
  Атомарность запроса сейчас держится на комбинации «commit в роутере» + «pool
  докоммитит/откатит на выходе». Это работает, но правило неявное: забытый
  commit в новом роутере НЕ приводит к потере данных (пул докоммитит), а
  ЛИШНИЙ commit до окончания всех записей фиксирует половину работы (частичная
  запись при последующем исключении). Ревьюеру приходится проверять порядок
  вручную; сервисы иногда открывают connection.transaction() (savepoint),
  иногда нет — смешение стилей.

Последствия.
  Класс трудно-воспроизводимых багов «половина эффекта сохранилась» при будущих
  правках, где кто-то вставит commit посреди сценария или доработает код после
  существующего commit.

Как устранить.
  Выбрать одну модель: (а) убрать все commit из роутеров и положиться на
  commit-on-success пула (поведение уже есть — нужно только зафиксировать
  договорённость и тест-инвариант «в api/v1 нет .commit(»), либо (б) middleware
  «транзакция на запрос». Вариант (а) — почти бесплатный.

  Прод-оверлей docker-compose.prod.yml: restart: unless-stopped, паролизация
  Redis (requirepass) и Mongo, отказ от публикации их портов (только внутренняя
  сеть), mem_limit хотя бы для api/redpanda. Плюс регламент бэкапов Postgres
  (pg_dump в cron + периодическая проверка восстановления) — сейчас бэкапов нет
  вообще (чек-лист §4: «непротестированный бэкап — иллюзия»).

--------------------------------------------------------------------------------
[P2-11] Frontend: нет таймаутов у fetch — зависший API вешает SSR-страницы
--------------------------------------------------------------------------------
Разделы: 6 (Frontend), 7 (Отказоустойчивость)
Файлы:
  apps/web/src/shared/api/http.ts (apiGet/apiPost/apiPatch/apiDelete — fetch
      без signal)

Почему это ошибка.
  Все страницы — force-dynamic и делают серверный fetch к API. Без таймаута
  зависший бэкенд-запрос держит рендер страницы до платформенного таймаута;
  клиентские блоки — вечный «Загрузка...». Чек-лист §7: «у всех внешних вызовов
  явные таймауты» — не выполнен на фронте.

Как устранить.
  В http.ts добавить AbortSignal.timeout(N) (например 10s) во все методы +
  единообразный маппинг AbortError в ErrorState. Одно место — закрывает всё.

--------------------------------------------------------------------------------
[P2-12] Frontend: force-dynamic + cache:no-store на всех 22 страницах,
        страница страны делает ~10 клиентских запросов
--------------------------------------------------------------------------------
Разделы: 6 (Frontend), 10 (Масштабируемость)
Файлы:
  apps/web/src/app/**/page.tsx (22 × `export const dynamic = "force-dynamic"`)
  apps/web/src/shared/api/http.ts (cache: "no-store" во всех методах)
  apps/web/src/app/countries/[slug]/page.tsx + features/* (CountryDriftBlock,
      TrustSurfaceBlock, PlatformIntelligenceBlock, CountryRoutesBlock,
      CountryDataJournalBlock, CommunityCountryBlock, CountryWhatChanged,
      CountryMigrationBoardBlock... — каждый блок = отдельный клиентский fetch)

Почему это ошибка.
  Рендер-стратегия выбрана «по умолчанию для всего» (чек-лист §6): методология,
  glossary, списки стран меняются редко, но каждый визит бьёт в API и БД без
  какого-либо кеша (Next data cache отключён, Redis-кеш выключен по умолчанию).
  Страница страны — HTTP-«N+1»: десяток последовательно монтирующихся блоков,
  каждый со своим запросом; те же данные бэкенд умеет отдавать агрегатом
  (country_read_model).

Последствия.
  Кратный запас нагрузки на API/БД сжигается на ровном месте; TTFB и LCP
  страницы страны определяются самым медленным из ~10 запросов.

Как устранить (при старте фронтенд-транша, это его прямой скоуп):
  1) Публичные редко меняющиеся страницы → `export const revalidate = N` (ISR)
     или хотя бы fetch с next: { revalidate }.
  2) Странице страны — один агрегатный серверный fetch (или Promise.all по 2-3
     эндпоинтам) с передачей данных в блоки пропсами; клиентские блоки оставить
     только для реально интерактивного (watchlist, community-действия).
  3) Инвалидация ISR — по publish-событиям (у бэкенда уже есть
     cache_invalidation-хуки, добавить revalidateTag-вызов или короткий TTL).

--------------------------------------------------------------------------------
[P2-13] Notifier: конфиг NOTIFIER_ALLOWED_COUNTRIES — жёсткий список демо-стран
--------------------------------------------------------------------------------
Разделы: 1 (Архитектура/границы), 7 (Отказоустойчивость)
Файлы:
  docker-compose.yml:129
  apps/notifier/internal/subscriptions (валидация подписок по AllowedCountries)

Почему это ошибка.
  Список разрешённых стран для подписок задан в инфраструктурном env и содержит
  скрытые демо-страны. Эпизод 5 открыл добавление стран через пайплайн — каждая
  опубликованная страна потребует ручной правки env и рестарта notifier, иначе
  подписки на неё будут отклоняться. Знание «какие страны существуют»
  продублировано вне БД (анти-паттерн shared knowledge, чек-лист §1).

Как устранить.
  Убрать список из env; валидировать страну через существующий gRPC-канал к API
  либо (проще) убрать проверку на стороне notifier вовсе — подписка создаётся
  через API, который уже валидирует страну по БД.

--------------------------------------------------------------------------------
[P2-14] validate_session_token: 3-4 запроса к БД на каждый auth-запрос
--------------------------------------------------------------------------------
Разделы: 4 (PostgreSQL), 10 (Масштабируемость)
Файлы:
  apps/api/app/services/auth.py:190-220
  apps/api/app/repositories/auth.py (get_active_session_by_token_hash,
      get_session_by_token_hash, get_user_by_id)

Почему это ошибка.
  Последовательно: активная сессия → (при промахе ещё одна выборка «а не
  истекла ли») → пользователь → touch. Это 3 round-trip'а в счастливом пути
  (4 c touch), тогда как достаточно одного JOIN'а auth_sessions × users по
  token_hash с вычислением expired-статуса в SQL.

Как устранить.
  Один запрос: SELECT s.*, u.* FROM auth_sessions s JOIN users u ... WHERE
  s.token_hash=%s; ветвление expired/revoked — по полям результата. touch —
  по правилу из P1-8. Вместе с P0-1 это сократит стоимость каждого
  авторизованного запроса в разы.

================================================================================
БЛОК P3 — НИЗКИЙ (гигиена и наблюдения)
================================================================================

[P3-1] `require_country_id` / `get_country` принимают «id ИЛИ slug» одним
       параметром (`WHERE id::text = %s OR slug = %s`) — двойной OR-предикат
       гарантирует seq scan и смешивает типы идентификаторов в API.
       Файлы: apps/api/app/repositories/common.py:69-81, countries.py:101-121.
       Устранение: развести на два явных метода (by_id / by_slug); публичные
       эндпоинты работают только со slug.

[P3-2] fetch_all() используется для UPDATE/DELETE (touch_session и др.) —
       семантически это execute; мелочь, но мешает читать код.
       Файл: apps/api/app/repositories/auth.py:343.

[P3-3] Фильтр статуса admin-списков (`status: str | None = Query(None)`)
       не валидируется по enum — произвольная строка тихо даёт пустой список
       вместо 422. Файлы: api/v1/admin_country_contribution.py:27,
       admin_author_metrics.py. Устранение: Query с Literal/Enum.

[P3-4] POST-эндпоинты создания контента (sources, evidence, timeline-events)
       не идемпотентны: двойной клик = дубликат черновика. Для черновиков
       терпимо; при фронтенд-транше добавить disable-on-submit и/или
       Idempotency-Key на создающие ручки.

[P3-5] `login`/`register` не различают «email не существует» и «пароль неверен»
       — правильно (перечисление аккаунтов затруднено), НО register возвращает
       409 email_already_registered — перечисление email'ов всё же возможно
       через форму регистрации. Компромисс стандартный; при появлении реальных
       пользователей рассмотреть нейтральный ответ + письмо (после email-транша).

[P3-6] Readiness `/ready` берёт соединение из общего пула — при исчерпании пула
       readiness падает первым и может спровоцировать рестарт здорового
       инстанса оркестратором. Устранение: отдельное короткоживущее соединение
       или отдельный мини-пул для health. Файл: apps/api/app/main.py:69-84.

[P3-7] Go: главный цикл app.Run не дожидается остановки gRPC-сервера при
       завершении (http-серверу дан Shutdown(5s), gRPC полагается на ctx внутри
       Serve). Стоит симметрично дождаться GracefulStop.
       Файл: apps/notifier/internal/app/app.go:89-158.

[P3-8] Пагинация LIMIT/OFFSET на публичных списках (sources, evidence,
       timeline) — на глубоких страницах OFFSET линейно дорожает. До реальных
       объёмов не критично; для новых эндпоинтов закладывать keyset.

[P3-9] Модель «id или slug» и `id::text` в путях: path-параметры proposal_id/
       definition_id и т.п. не валидируются как UUID на границе (строка любого
       вида доходит до SQL). Вреда нет (параметризовано), но 422 на границе
       чище, чем пустой 404 из глубины. Устранение: тип UUID в сигнатурах
       роутеров (FastAPI сам даст 422).

[P3-10] В миграциях 002-025 контент-сиды (включая RU-переводы и demo-данные)
       зашиты в файлы миграций. Это осознанное решение, но оно навсегда: любые
       структурные изменения обязаны оставаться совместимыми со старыми
       INSERT'ами. Зафиксировано как ограничение; новые страны идут через
       пайплайн Эпизода 5 — не через миграции (уже соблюдается).

[P3-11] Дублирование хелперов `_total(rows)` / `_audit(...)` / PROPOSAL-view
       маппингов между доменными пакетами (author_metrics, country_contribution,
       migration_board) — микро-дублирование, приемлемое по house style
       («квалифицированный доступ важнее DRY»), но _total и audit-обёртку можно
       вынести в общий модуль без потери монкипатчабельности.

[P3-12] X-Forwarded-For: при trusted_proxy_headers=true берётся первый элемент
       списка (клиентский, спуфится). Правильно — последний недоверенный хоп
       при известном числе прокси. Связано с P1-6.
       Файл: apps/api/app/main.py:23-32.

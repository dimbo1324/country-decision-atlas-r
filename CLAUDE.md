# Country Decision Atlas — working notes for Claude Code

Country Decision Atlas is a full-stack decision-support platform that helps a
person choose a country for a specific goal: relocation, residency or
citizenship, remote work, low-budget living, business, or safety. It produces
structured, sourced, confidence-rated output. It is not a country-ranking blog.

Read `docs/_arch_/` before making domain assumptions. Those Russian documents
hold the product vision and domain model. Keep this file aligned with
`AGENTS.md` so Claude Code and Codex operate from the same project contract.

## Repository Map

- `apps/api` — FastAPI backend. Keep `app/api/v1/*` routers thin; put business
  logic in `app/services/*` and SQL in `app/repositories/*`.
- `apps/web` — Next.js frontend.
- `apps/notifier` — Go gRPC notifier service.
- `contracts/openapi.yaml` — source contract for generated frontend types.
- `packages/contracts/generated` — generated by `pnpm contracts:generate`.
  Never hand-edit generated files.
- `database/migrations` — numbered SQL migrations. Do not rename or edit an
  already-applied migration unless the user explicitly accepts the local
  database reset cost.
- `scripts/dev_tools` — developer automation. Prefer these scripts over ad hoc
  command sequences.
- `docs/_arch_` — product and domain architecture in Russian.
- `.codex` — project-scoped Codex configuration, custom agents, and reusable
  skills. Keep shared assistant rules aligned with this file and `AGENTS.md`.

## Claude Code Workspace

- `.claude/launch.json` contains local run targets for Claude Code.
- `.claude/skills/` contains project-specific reusable workflows.
- Prefer project skills and `scripts/dev_tools` before inventing new shell
  sequences.
- Keep long-running or risky commands visible in the conversation.
- When changing shared assistant behavior, keep `AGENTS.md` and `.codex`
  guidance aligned where their scopes overlap.

## First Steps For Agents

1. Run `git status --short --branch` before changing files.
2. Read the smallest relevant code/doc area before editing.
3. Preserve user changes. Do not revert unrelated work.
4. Use `rg`/`rg --files` for search.
5. Use structured parsers or existing helpers instead of brittle string hacks.
6. Keep changes scoped to the request.

## House Style

- No comments unless the reason is genuinely non-obvious.
- No docstrings that merely repeat a function name.
- Services or repositories over roughly 800 lines should become packages, not
  larger flat files.
- When splitting modules, keep monkeypatch-sensitive helpers in one sibling
  helper module and call them by qualified module access.
- Do not add countries or languages unless the user explicitly asks.
- Do not use AI-driven translation unless the user explicitly asks.
- Keep legacy DB columns in place after superseding migrations.
- Localization reads go through `app/services/localization.py` and
  `overlay_localized_fields`.
- Test files use existing collision-safe abbreviations such as `_repo`, `_dq`,
  and `_mig`.

## Git Workflow

- Commit per logical task with a message explaining why.
- Prefer fast-forward syncs: `git pull --ff-only origin main`.
- If the user explicitly asks in the current turn to commit and push, Claude may
  commit and push to `main` after the quick quality gate passes.
- If the user only asks for code changes, do not push to `main` without a fresh,
  explicit push request.
- For a guided one-command release path, use:

```powershell
python dev_tools_scripts_runner.py ship-main --message "type: explain the change"
```

## Product Guardrails

- Treat Country Decision Atlas as a decision-support system with sources,
  confidence, freshness, and transparent tradeoffs.
- Do not turn it into a generic listicle/ranking product.
- Prefer read-model overlays and existing domain services to per-field branching.
- When unsure about domain intent, read `docs/_arch_/` before coding.
